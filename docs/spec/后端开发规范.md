## 一、概述

### 1.1 目的

本规范旨在：
- 确保后端代码的一致性、可维护性和可读性
- 降低新成员的学习成本
- 提高代码质量和系统稳定性
- 建立统一的开发标准

### 1.2 适用对象

- 所有后端开发人员
- 代码审查人员
- 技术负责人

### 1.3 强制性

本规范中：
- **"必须"、"强制"、"禁止"** 表示不可违反的硬性要求
- **"应该"、"推荐"** 表示最佳实践，特殊情况下可例外
- **"可以"、"建议"** 表示可选的优化建议

## 二、代码规范

### 2.1 通用代码规范

#### 2.1.1 类型提示（强制）

**规则：** 所有函数和方法**必须**添加完整的类型提示。

```python
# ✅ 正确
def create(self, data: Dict[str, Any], operator: str) -> Dict[str, Any]:
    """创建实体"""
    pass

def get_by_id(self, entity_id: str) -> Dict[str, Any]:
    """获取实体详情"""
    pass

# ❌ 错误：缺少类型提示
def create(self, data, operator):
    pass
```

**强制要求：**
- 所有函数参数必须有类型提示
- 所有函数返回值必须有类型提示
- 使用 `Optional[T]` 表示可选参数
- 使用 `Union[T1, T2]` 表示多类型
- 复杂类型使用 `typing` 模块的类型

---

#### 2.1.2 代码风格（强制）

**规则：** 遵循 PEP 8 规范，使用自动格式化工具。

**命名约定：**
- 变量和函数：`snake_case`
- 类名：`PascalCase`
- 常量：`UPPER_CASE`
- 私有成员：`_leading_underscore`

```python
# ✅ 正确
class CustomerService:
    MAX_PAGE_SIZE = 100

    def __init__(self, db: Database) -> None:
        self.db = db

    def get_by_id(self, customer_id: str) -> Dict[str, Any]:
```

```bash

```

#### 2.1.3 日志记录（强制）

**规则：** **禁止**使用 `print`，**必须**使用 `logging` 模块。

```python
import logging

logger = logging.getLogger(__name__)

# ✅ 正确
class MyService:
    def create(self, data: dict) -> dict:
        logger.info(f"创建实体: {data.get('name')}")
        try:
            result = self.collection.insert_one(data)
            return result
        except Exception as e:
            logger.error(f"创建失败: {e}", exc_info=True)
            raise

# ❌ 错误：使用 print
def create(self, data: dict):
    print(f"创建实体: {data}")  # 禁止！
```

**日志级别使用规范：**

| 级别 | 使用场景 | 示例 |
|------|---------|------|
| `DEBUG` | 详细的调试信息 | 查询条件、处理前的数据 |
| `INFO` | 关键业务流程 | 用户登录、实体创建 |
| `WARNING` | 警告信息 | 索引创建失败、API 调用缓慢 |
| `ERROR` | 错误信息 | 功能失败但程序继续运行 |
| `CRITICAL` | 严重错误 | 系统级故障 |

**强制要求：**
- 记录 ERROR 级别日志时**必须**使用 `exc_info=True` 或 `logger.exception()`
- 生产环境禁止使用 DEBUG 级别日志
- 日志信息必须包含足够的上下文信息

#### 2.1.4 时区统一（强制）

**规则：** 项目统一使用**本地时间**（不带时区信息的 naive datetime）。

```python
from datetime import datetime

# ✅ 正确：使用本地时间
created_at: datetime = Field(default_factory=datetime.now)
updated_at: datetime = Field(default_factory=datetime.now)

# ❌ 错误：不要使用 UTC 时间
created_at: datetime = Field(default_factory=datetime.utcnow)  # 禁止！
```

**说明：**
1. 项目所有时间字段使用 `datetime.now()` 获取当前系统本地时间
2. 数据库存储时不带时区信息（naive datetime）
3. 前后端统一使用本地时间，简化时区转换逻辑

## 三、数据模型设计规范

### 3.1 模型分层模式（强制）

**规则：** 所有实体**必须**定义以下五种模型。

#### 3.1.1 模型分层结构

```python
from pydantic import BaseModel, Field
from typing import Optional, List
from datetime import datetime

# 第 1 层：创建模型（Create Model）
class CustomerCreate(BaseModel):
    """客户创建模型 - 仅包含创建时必需/允许的字段"""
    user_name: str = Field(..., min_length=1, description="客户全称")
    short_name: str = Field(..., min_length=1, description="客户简称")
    # 不包含 id, created_at, updated_at 等自动生成字段

# 第 2 层：更新模型（Update Model）
class CustomerUpdate(BaseModel):
    """客户更新模型 - 所有字段都是可选的"""
    user_name: Optional[str] = Field(None, min_length=1)
    short_name: Optional[str] = Field(None, min_length=1)
    # 允许部分更新

# 第 3 层：完整模型（Full Model - 数据库实体）
class Customer(BaseMongoModel, CustomerCreate):
    """客户完整模型 - 包含所有数据库字段"""
    id: PyObjectId = Field(default_factory=PyObjectId, alias="_id")
    status: Literal["prospect", "pending", "active", "suspended", "terminated"] = "prospect"
    created_at: datetime = Field(default_factory=datetime.now)
    updated_at: datetime = Field(default_factory=datetime.now)
    created_by: Optional[str] = None
    updated_by: Optional[str] = None

# 第 4 层：列表项模型（List Item Model - 轻量化）
class CustomerListItem(BaseMongoModel):
    """客户列表项模型 - 仅包含列表展示所需字段"""
    id: str
    user_name: str
    short_name: str
    status: str
    created_at: datetime
    # 不包含嵌套的复杂数据（如 utility_accounts）

# 第 5 层：响应模型（Response Model - 分页）
class CustomerListResponse(BaseModel):
    """客户列表响应"""
    total: int
    page: int
    page_size: int
    items: List[CustomerListItem]
```

**强制要求：**
- `Create` 模型：不包含自动生成字段（id, 时间戳等）
- `Update` 模型：所有字段都是 `Optional`
- `Full` 模型：包含所有数据库字段
- `ListItem` 模型：精简，避免嵌套复杂对象
- `ListResponse` 模型：包含分页信息

---

### 3.2 字段命名约定（强制）

| 字段类型 | 命名规范 | 示例 | 禁止使用 |
|---------|---------|------|---------|
| 主键 | `id` (别名 `_id`) | `id: PyObjectId` | `_id`, `ID`, `userId` |
| 时间戳 | `{action}_at` | `created_at`, `updated_at` | `createTime`, `create_time` |
| 操作人 | `{action}_by` | `created_by`, `updated_by` | `creator`, `operator` |
| 状态字段 | `status` | `status: Literal["draft", "active"]` | `state`, `statusCode` |
| 布尔字段 | `is_{condition}` 或 `has_{item}` | `is_green_power`, `has_contract` | `greenPower`, `contracted` |
| 外键引用 | `{entity}_id` | `package_id`, `customer_id` | `packageID`, `package` |
| 列表字段 | 复数形式 | `utility_accounts`, `metering_points` | `utilityAccount`, `account_list` |

---

### 3.3 枚举类型定义规范（推荐）

**规则：** 简单枚举使用 `Literal`，复杂枚举使用 `Enum`。

```python
from typing import Literal
from enum import Enum

# ✅ 方式1：使用 Literal（推荐用于简单枚举）
status: Literal["draft", "active", "archived"]

# ✅ 方式2：使用 Enum（推荐用于复杂枚举，需要额外元数据）
class CustomerStatus(str, Enum):
    """客户状态枚举"""
    PROSPECT = "prospect"      # 意向客户
    PENDING = "pending"        # 待生效
    ACTIVE = "active"          # 执行中
    SUSPENDED = "suspended"    # 已暂停
    TERMINATED = "terminated"  # 已终止

    @property
    def display_name(self) -> str:
        """返回中文显示名称"""
        return {
            self.PROSPECT: "意向客户",
            self.PENDING: "待生效",
            self.ACTIVE: "执行中",
            self.SUSPENDED: "已暂停",
            self.TERMINATED: "已终止"
        }[self]

# 使用
status: CustomerStatus = Field(default=CustomerStatus.PROSPECT)
```

## 四、Service 层开发规范

### 4.1 Service 类结构（强制）

**规则：** 所有 Service 类**必须**遵循以下结构模板。

```python
from pymongo.database import Database
from typing import Dict, List, Optional, Any
from datetime import datetime
import logging

logger = logging.getLogger(__name__)


class EntityService:
    """
    实体服务层

    职责：
    - 封装业务逻辑
    - 数据库操作
    - 数据校验
    - 索引管理

    依赖：
    - db: MongoDB 数据库实例（必须是 webapp.tools.mongo.DATABASE）
    """

    def __init__(self, db: Database) -> None:
        """初始化服务"""
        self.db = db
        self.collection = self.db.entity_collection_name
        self._ensure_indexes()

    def _ensure_indexes(self) -> None:
        """确保数据库索引存在（私有方法）"""
        try:
            # 定义所有索引
            indexes = [
                ([('status', 1)], {'name': 'idx_status'}),
                ([('created_at', -1)], {'name': 'idx_created_at'}),
            ]

            # 获取已存在的索引
            existing_indexes = {idx.get('name') for idx in self.collection.list_indexes()}

            # 创建缺失的索引
            for keys, options in indexes:
                if options['name'] not in existing_indexes:
                    self.collection.create_index(keys, **options)
                    logger.info(f"创建索引: {options['name']}")

        except Exception as e:
            # 索引创建失败不应该阻止服务启动
            logger.warning(f"创建索引时出错: {str(e)}")

    # ========== CRUD 标准方法（强制） ==========

    def create(self, data: Dict[str, Any], operator: str) -> Dict[str, Any]:
        """
        创建实体

        Args:
            data: 实体数据
            operator: 操作人用户名

        Returns:
            创建后的实体字典（包含 id）

        Raises:
            ValueError: 业务逻辑错误（如名称重复）
        """
        pass

    def get_by_id(self, entity_id: str) -> Dict[str, Any]:
        """
        根据ID获取实体

        Args:
            entity_id: 实体ID

        Returns:
            实体详情字典

        Raises:
            ValueError: 实体不存在或ID无效
        """
        pass

    def list(self, filters: Dict[str, Optional[str]], page: int = 1, page_size: int = 20) -> Dict[str, Any]:
        """
        获取实体列表（分页）

        Args:
            filters: 筛选条件字典
            page: 页码（从1开始）
            page_size: 每页大小

        Returns:
            {
                "total": int,
                "page": int,
                "page_size": int,
                "items": List[dict]
            }
        """
        pass

    def update(self, entity_id: str, data: Dict[str, Any], operator: str) -> Dict[str, Any]:
        """
        更新实体

        Args:
            entity_id: 实体ID
            data: 更新数据（仅包含需要更新的字段）
            operator: 操作人用户名

        Returns:
            更新后的实体字典

        Raises:
            ValueError: 实体不存在、状态不允许更新等业务错误
        """
        pass

    def delete(self, entity_id: str) -> None:
        """
        删除实体（物理删除或软删除）

        Args:
            entity_id: 实体ID

        Raises:
            ValueError: 实体不存在或状态不允许删除
        """
        pass

    # ========== 业务方法 ==========

    def change_status(self, entity_id: str, new_status: str, operator: str) -> Dict[str, Any]:
        """状态变更方法（如果实体有状态机）"""
        pass

    # ========== 私有辅助方法 ==========

    def _convert_to_dict(self, doc: Dict[str, Any]) -> dict:
        """将 MongoDB 文档转换为 API 返回格式（私有方法）"""
        doc["id"] = str(doc.pop("_id"))
        return doc
```

**强制要求：**
- 所有 Service 类必须有 `__init__(self, db: Database) -> None:` 构造函数
- CRUD 方法必须使用标准命名：`create`, `get_by_id`, `list`, `update`, `delete`
- 所有修改操作必须接受 `operator: str` 参数记录操作人
- 私有方法使用下划线前缀（`_method_name`）
- 所有方法必须有完整的类型提示和文档字符串

### 4.2 索引管理规范（强制）

**规则：** 所有 Service 类**必须**实现 `_ensure_indexes()` 方法。

**索引命名规范：**
- 单字段索引：`idx_{field_name}`
- 组合索引：`idx_{field1}_{field2}`
- 唯一索引：`idx_unique_{field_name}`
- 全文索引：`idx_text_{field_name}`

```python
def _ensure_indexes(self) -> None:
    """确保数据库索引存在"""
    try:
        # 定义所有索引
        indexes = [
            # 格式：([字段列表], {索引选项})
            ([('status', 1)], {'name': 'idx_status'}),
            ([('created_at', -1)], {'name': 'idx_created_at'}),
            ([('status', 1), ('created_at', -1)], {'name': 'idx_status_created_at'}),
            ([('name', 1)], {'name': 'idx_unique_name', 'unique': True}),
        ]

        # 获取已存在的索引
        existing_indexes = {idx.get('name') for idx in self.collection.list_indexes()}

        # 创建缺失的索引
        for keys, options in indexes:
            if options['name'] not in existing_indexes:
                self.collection.create_index(keys, **options)
                logger.info(f"创建索引: {options['name']}")

    except Exception as e:
        # 索引创建失败不应该阻止服务启动，记录错误即可
        logger.warning(f"创建索引时出错: {str(e)}")
```

## 五、时序数据存储与处理规范

### 5.1 长/窄表格式（Long Format）原则（强制）

**规则：** 所有时序数据（负荷曲线、价格曲线等）**必须**采用"长/窄"格式存储。

**定义：**
- **长格式（Long Format）：** 每个时间点作为一条独立记录
- **宽格式（Wide Format）：** 一天的所有点存储在一个数组中（**禁止使用**）

#### 5.1.1 正确示例

```python
# ✅ 正确：长表格式（每个时间点一条记录）
{
    "_id": ObjectId("..."),
    "timestamp": datetime(2025, 11, 10, 0, 15, 0),  # 本地时间
    "customer_id": "673b2e3c8f4e5a1b2c3d4e5f",
    "metering_point_id": "673b2e3c8f4e5a1b2c3d4e60",
    "value": 23.1,
    "data_source": "prediction"  # 预测曲线 or 结算曲线
}
{
    "_id": ObjectId("..."),
    "timestamp": datetime(2025, 11, 10, 0, 30, 0),
    "customer_id": "673b2e3c8f4e5a1b2c3d4e5f",
    "metering_point_id": "673b2e3c8f4e5a1b2c3d4e60",
    "value": 24.5,
    "data_source": "prediction"
}
```

#### 5.1.2 错误示例

```python
# ❌ 错误：宽表格式（禁止使用）
{
    "date": "2025-11-10",
    "customer_id": "673b2e3c8f4e5a1b2c3d4e5f",
    "points": [23.1, 24.5, 25.3, ..., 22.8]  # 96个值 - 禁止！
}
```

**长格式优势：**
- ✅ 简化聚合查询（使用 MongoDB 聚合管道）
- ✅ 便于创建时间索引
- ✅ 灵活处理缺失数据
- ✅ 易于扩展字段

---

### 5.2 24:00 时刻处理规范（强制）

**规则：** 业务日的第96个点（24:00）在数据库中**必须**存储为次日的 00:00:00。

#### 5.2.1 业务日定义

电力业务日从 `00:15` 开始，到次日 `00:00`（即 `24:00`）结束，共96个点（15分钟间隔）。

#### 5.2.2 存储规范

```python
# 2025-11-10 的第96个点（24:00）
{
    "timestamp": datetime(2025, 11, 11, 0, 0, 0),  # ✅ 存储为次日00:00
    "customer_id": "...",
    "value": 22.8
}
```

**原因：**
1. `24:00` 不是标准的时间表示法（Python datetime 不支持）
2. 存储为次日 `00:00` 是 ISO 8601 标准做法
3. 简化数据库查询逻辑

---

### 5.3 时间查询区间规范（强制）

**规则：** 查询一整天（96个点）的数据时，时间区间**必须**使用**左开右闭** `(start_of_day, end_of_day]`。

```python
from datetime import datetime, timedelta

def get_day_load_data(collection, customer_id: str, date: datetime) -> list:
    """
    获取指定日期的完整96点负荷数据

    Args:
        collection: MongoDB 集合
        customer_id: 客户ID
        date: 查询日期（本地时间）

    Returns:
        包含96个数据点的列表
    """
    # 构造时间范围
    start_of_day = date.replace(hour=0, minute=0, second=0, microsecond=0)
    end_of_day = start_of_day + timedelta(days=1)

    # 构造查询条件
    query = {
        "customer_id": customer_id,
        "timestamp": {
            "$gt": start_of_day,   # 不包含 00:00（属于前一天的第96个点）
            "$lte": end_of_day      # 包含次日 00:00（即当天的第96个点）
        }
    }

    # 执行查询并按时间排序
    result = list(collection.find(query).sort("timestamp", 1))
    return result
```

**关键要点：**
- ✅ 使用 `$gt`（大于）而不是 `$gte`（大于等于）
- ✅ 使用 `$lte`（小于等于）包含当天的第96个点
- ✅ 结果应恰好包含96条记录（如果数据完整）

### 5.4 数据格式化职责（强制）

**规则：** 后端API在返回给前端时，**必须**将时间戳格式化为业务层的时间标签。

```python
from datetime import datetime, timedelta

def format_load_point_for_frontend(point: dict, query_date: datetime) -> dict:
    """
    将数据库时间戳格式化为前端业务时间标签

    Args:
        point: 数据库查询结果 {"timestamp": datetime(...), "value": float}
        query_date: 查询的业务日期

    Returns:
        前端格式 {"time": str, "value": float, "timestamp": str}
    """
    timestamp: datetime = point['timestamp']
    query_next_day = query_date.replace(hour=0, minute=0, second=0, microsecond=0) + timedelta(days=1)

    # 判断是否为某一天的第96个点（次日00:00）
    if timestamp.hour == 0 and timestamp.minute == 0 and timestamp.date() == query_next_day.date():
        time_label = "24:00"
    else:
        time_label = timestamp.strftime("%H:%M")

    return {
        "time": time_label,           # 业务时间标签："00:15", "23:45", "24:00"
        "value": point['value'],
        "timestamp": timestamp.isoformat()  # ISO格式，供前端调试使用
    }

# API 端点示例
@router.get("/load-data")
async def get_load_data(customer_id: str, date: str):
    """获取指定日期的负荷数据"""
    query_date = datetime.strptime(date, "%Y-%m-%d")
    raw_data = get_day_load_data(collection, customer_id, query_date)

    # 格式化为前端所需格式
    formatted_data = [
        format_load_point_for_frontend(point, query_date)
        for point in raw_data
    ]

    return {"date": date, "data": formatted_data}
```

**关键要点：**
- ✅ 后端负责将 `00:00` 格式化为 `"24:00"`
- ✅ 前端无需理解"24:00实际存储为次日00:00"的细节
- ✅ 保持前端代码简洁

## 七、错误处理规范

### 7.1 异常处理模式（强制）

**规则：** Service 层抛出 `ValueError`，API 层转换为 `HTTPException`。

```python
# Service 层
class CustomerService:
    def get_by_id(self, customer_id: str) -> Dict[str, Any]:
        if not ObjectId.is_valid(customer_id):
            raise ValueError(f"无效的客户ID: {customer_id}")  # ✅ 抛出 ValueError

        customer = self.collection.find_one({"_id": ObjectId(customer_id)})
        if not customer:
            raise ValueError(f"客户不存在: {customer_id}")  # ✅ 抛出 ValueError

        return self._convert_to_dict(customer)

# API 层
@router.get("/{customer_id}")
async def get_customer(customer_id: str):
    service = CustomerService(DATABASE)
    try:
        result = service.get_by_id(customer_id)
        return result
    except ValueError as e:
        # ✅ 业务逻辑错误 -> 404
        raise HTTPException(
            status_code=status.HTTP_404_NOT_FOUND,
            detail=str(e)
        )
    except Exception as e:
        # ✅ 意外错误 -> 500，并记录日志
        logger.exception(f"获取客户时发生未预期错误: {customer_id}")
        raise HTTPException(
            status_code=status.HTTP_500_INTERNAL_SERVER_ERROR,
            detail="服务器内部错误"
        )
```

**强制要求：**
- Service 层只抛出 `ValueError`
- API 层捕获 `ValueError` 并转换为合适的 HTTPException
- 禁止裸 `except:` 或 `except Exception: pass`
- 记录 ERROR 级别日志时使用 `exc_info=True` 或 `logger.exception()`

## 八、数据库操作最佳实践

### 8.1 查询优化（推荐）

**规则：** 复杂查询**应该**使用聚合管道优化性能。

```python
# ✅ 推荐：使用聚合管道
def get_users_with_order_counts(user_ids: List[str]):
    pipeline = [
        {"$match": {"_id": {"$in": [ObjectId(uid) for uid in user_ids]}}},
        {"$lookup": {
            "from": "orders",
            "localField": "_id",
            "foreignField": "user_id",
            "as": "orders"
        }},
        {"$addFields": {
            "order_count": {"$size": "$orders"}
        }},
        {"$project": {"orders": 0}}
    ]
    return list(db.users.aggregate(pipeline))

# ❌ 低效：N+1 查询
def get_users_with_order_counts_bad(user_ids: List[str]):
    users = db.users.find({"_id": {"$in": [ObjectId(uid) for uid in user_ids]}})
    for user in users:
        orders = list(db.orders.find({"user_id": str(user["_id"])}))  # N+1！
        user["order_count"] = len(orders)
    return users
```

### 8.2 批量操作规范（推荐）

**规则：** 批量更新**应该**使用 `bulk_write`。

```python
from pymongo import UpdateOne

# ✅ 推荐：使用 bulk_write
def batch_update_status(updates: List[dict]):
    operations = [
        UpdateOne(
            {"_id": ObjectId(item["id"])},
            {"$set": {"status": item["new_status"], "updated_at": datetime.now()}}
        )
        for item in updates
    ]

    result = collection.bulk_write(operations, ordered=False)
    logger.info(f"批量更新完成: 成功 {result.modified_count}")
    return result

# ❌ 低效：循环调用单次更新
def batch_update_status_bad(updates: List[dict]):
    for item in updates:
        collection.update_one(...)  # 每次都是一次网络往返！
```
