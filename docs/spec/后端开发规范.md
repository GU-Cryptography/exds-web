#### 1.1 模型分层模式（必须遵循）

```python
# ✅ 推荐的三层模型结构

# 第 1 层：创建模型（Create Model）
class CustomerCreate(BaseModel):
    """客户创建模型 - 仅包含创建时必需/允许的字段"""
    user_name: str = Field(..., min_length=1, description="客户全称")
    short_name: str = Field(..., min_length=1, description="客户简称")
    # 不包含 id, created_at, updated_at 等自动生成字段

# 第 2 层：更新模型（Update Model）
class CustomerUpdate(BaseModel):
    """客户更新模型 - 所有字段都是可选的"""
    user_name: Optional[str] = Field(None, min_length=1)
    short_name: Optional[str] = Field(None, min_length=1)
    # 允许部分更新

# 第 3 层：完整模型（Full Model - 数据库实体）
class Customer(BaseMongoModel, CustomerCreate):
    """客户完整模型 - 包含所有数据库字段"""
    id: PyObjectId = Field(default_factory=PyObjectId, alias="_id")
    status: Literal["prospect", "pending", "active", "suspended", "terminated"] = "prospect"
    created_at: datetime = Field(default_factory=datetime.utcnow)
    updated_at: datetime = Field(default_factory=datetime.utcnow)
    created_by: Optional[str] = None
    updated_by: Optional[str] = None

# 第 4 层：列表项模型（List Item Model - 轻量化）
class CustomerListItem(BaseMongoModel):
    """客户列表项模型 - 仅包含列表展示所需字段"""
    user_name: str
    short_name: str
    status: str
    created_at: datetime
    # 不包含嵌套的复杂数据（如 utility_accounts）

# 第 5 层：响应模型（Response Model - 分页）
class CustomerListResponse(BaseModel):
    """客户列表响应"""
    total: int
    page: int
    page_size: int
    items: List[CustomerListItem]
```

**强制要求：**
- 所有实体必须定义 `Create`, `Update`, `Full`, `ListItem`, `ListResponse` 五种模型
- `Create` 模型不包含自动生成字段（id, 时间戳等）
- `Update` 模型所有字段都是 `Optional`
- `ListItem` 模型应精简，避免嵌套复杂对象

#### 1.2 字段命名约定（强制）

| 字段类型 | 命名规范 | 示例 | 禁止使用 |
|---------|---------|------|---------|
| 主键 | `id` (别名 `_id`) | `id: PyObjectId` | `_id`, `ID`, `userId` |
| 时间戳 | `{action}_at` | `created_at`, `updated_at` | `createTime`, `create_time` |
| 操作人 | `{action}_by` | `created_by`, `updated_by` | `creator`, `operator` |
| 状态字段 | `status` | `status: Literal["draft", "active"]` | `state`, `statusCode` |
| 布尔字段 | `is_{condition}` 或 `has_{item}` | `is_green_power`, `has_contract` | `greenPower`, `contracted` |
| 外键引用 | `{entity}_id` | `package_id`, `customer_id` | `packageID`, `package` |
| 列表字段 | 复数形式 | `utility_accounts`, `metering_points` | `utilityAccount`, `account_list` |

**统一时区要求：**

#### 2.2 索引管理规范

```python
def _ensure_indexes(self) -> None:
    """确保数据库索引存在"""
    try:
        # 定义所有索引
        indexes = [
            # 格式：([字段列表], {索引选项})
            ([('status', 1)], {'name': 'idx_status'}),
            ([('created_at', -1)], {'name': 'idx_created_at'}),
            ([('status', 1), ('created_at', -1)], {'name': 'idx_status_created_at'}),
        ]

        # 获取已存在的索引
        existing_indexes = {idx.get('name') for idx in self.collection.list_indexes()}

        # 创建缺失的索引
        for keys, options in indexes:
            if options['name'] not in existing_indexes:
                self.collection.create_index(keys, **options)
                logger.info(f"创建索引: {options['name']}")

    except Exception as e:
        # 索引创建失败不应阻止服务启动
        logger.warning(f"创建索引时出错: {str(e)}")
```

**索引命名规范：**
- 单字段索引：`idx_{field_name}`
- 组合索引：`idx_{field1}_{field2}`
- 唯一索引：`idx_unique_{field_name}`
- 全文索引：`idx_text_{field_name}`

#### 3.1 长/窄表格式（Long Format）原则（强制）

**强制要求**：所有时序数据（负荷曲线、价格曲线等）**必须**采用"长/窄"格式存储。

❌ **错误示范（宽表格式）**：
```python
# 错误：将一天的96个点存储在一个数组中
{
    "date": "2025-11-10",
    "customer_id": "673b2e3c8f4e5a1b2c3d4e5f",
    "points": [23.1, 24.5, 25.3, ..., 22.8]  # 96个值
}
```

**问题**：
- 无法对单个时间点建立索引
- 聚合查询复杂（需要展开数组）
- 难以处理缺失数据
- 扩展性差

✅ **正确示范（长表格式）**：
```python
# 正确：每个时间点作为一条独立记录
{
    "timestamp": datetime(2025, 11, 10, 0, 15, 0),  # 本地时间
    "customer_id": "673b2e3c8f4e5a1b2c3d4e5f",
    "metering_point_id": "673b2e3c8f4e5a1b2c3d4e60",
    "value": 23.1,
    "data_source": "prediction"  # 预测曲线 or 结算曲线
}
{
    "timestamp": datetime(2025, 11, 10, 0, 30, 0),
    "customer_id": "673b2e3c8f4e5a1b2c3d4e5f",
    "metering_point_id": "673b2e3c8f4e5a1b2c3d4e60",
    "value": 24.5,
    "data_source": "prediction"
}
```

**优势**：
- ✅ 简化聚合查询（使用 MongoDB 聚合管道）
- ✅ 便于创建时间索引：`db.collection.create_index([("timestamp", 1), ("customer_id", 1)])`
- ✅ 灵活处理缺失数据（某些时间点不存在记录）
- ✅ 易于扩展字段（如添加 `quality_flag` 质量标识）

#### 3.2 24:00 时刻的标准化处理（强制）

**业务日定义**：
- 电力业务日从 `00:15` 开始，到次日 `00:00`（即 `24:00`）结束，共96个点（15分钟间隔）

**存储规范**：
- 业务日的第96个点（`24:00`）在数据库中存储为**次日的 `00:00:00`**
- 例如：`2025-11-10` 的第96个点（24:00）存储为 `datetime(2025, 11, 11, 0, 0, 0)`

**原因**：
- `24:00` 不是标准的时间表示法（Python datetime 不支持）
- 存储为次日 `00:00` 是 ISO 8601 标准做法
- 简化数据库查询逻辑

**查询规范**：
查询一整天（96个点）的数据时，时间区间**必须**使用**左开右闭 `(start_of_day, end_of_day]`**

```python
from datetime import datetime, timedelta

def get_day_load_data(collection, customer_id: str, date: datetime) -> list:
    """
    获取指定日期的完整96点负荷数据

    Args:
        collection: MongoDB 集合
        customer_id: 客户ID
        date: 查询日期（本地时间）

    Returns:
        包含96个数据点的列表
    """
    # 构造时间范围
    start_of_day = date.replace(hour=0, minute=0, second=0, microsecond=0)
    end_of_day = start_of_day + timedelta(days=1)

    # 构造查询条件
    query = {
        "customer_id": customer_id,
        "timestamp": {
            "$gt": start_of_day,   # 不包含 00:00（属于前一天的第96个点）
            "$lte": end_of_day      # 包含次日 00:00（即当天的第96个点）
        }
    }

    # 执行查询并按时间排序
    result = list(collection.find(query).sort("timestamp", 1))

    return result
```

**关键要点**：
- ✅ 使用 `$gt`（大于）而不是 `$gte`（大于等于）来排除前一天的第96个点
- ✅ 使用 `$lte`（小于等于）来包含当天的第96个点（次日00:00）
- ✅ 结果应该恰好包含96条记录（如果数据完整）

#### 3.3 数据格式化职责（强制）

**强制要求**：后端API在返回给前端时，**必须**将时间戳格式化为业务层的时间标签。

```python
from datetime import datetime

def format_load_point_for_frontend(point: dict, query_date: datetime) -> dict:
    """
    将数据库时间戳格式化为前端业务时间标签

    Args:
        point: 数据库查询结果 {"timestamp": datetime(...), "value": float}
        query_date: 查询的业务日期（用于判断是否为第96个点）

    Returns:
        前端格式 {"time": str, "value": float, "timestamp": str}
    """
    timestamp: datetime = point['timestamp']
    query_next_day = query_date.replace(hour=0, minute=0, second=0, microsecond=0) + timedelta(days=1)

    # 判断是否为某一天的第96个点（次日00:00）
    if timestamp.hour == 0 and timestamp.minute == 0 and timestamp.date() == query_next_day.date():
        time_label = "24:00"
    else:
        time_label = timestamp.strftime("%H:%M")

    return {
        "time": time_label,           # 业务时间标签："00:15", "23:45", "24:00"
        "value": point['value'],      # 数值
        "timestamp": timestamp.isoformat()  # ISO格式，供前端调试使用
    }

# 使用示例
@router.get("/load-data")
async def get_load_data(customer_id: str, date: str):
    """获取指定日期的负荷数据"""
    query_date = datetime.strptime(date, "%Y-%m-%d")

    # 从数据库查询原始数据
    raw_data = get_day_load_data(collection, customer_id, query_date)

    # 格式化为前端所需格式
    formatted_data = [
        format_load_point_for_frontend(point, query_date)
        for point in raw_data
    ]

    return {"date": date, "data": formatted_data}
```

**返回示例**：

**关键要点**：
- ✅ 后端负责将 `00:00` 格式化为 `"24:00"`
- ✅ 前端无需理解"24:00实际存储为次日00:00"的细节
- ✅ 保持前端代码简洁，降低维护成本

#### 3.4 数据存储层级设计

**三层存储架构**（推荐）：

**1. 原始数据层（Raw Data Layer）**
- **用途**：保留用户上传的原始96点数据，作为审计和追溯依据
- **集合示例**：`raw_user_load_imports`
- **特点**：只增不改，永久保留
- **数据结构**：
```python
{
    "_id": ObjectId("..."),
    "customer_id": "...",
    "import_date": datetime.now(),
    "operator": "admin",
    "file_name": "load_data_20251110.xlsx",
    "raw_points": [
        {"point_number": 1, "value": 23.1},
        {"point_number": 2, "value": 24.5},
        # ... 96个点
    ]
}
```

**2. 工作数据层（Working Data Layer）**
- **用途**：存储清洗后的预测/结算曲线（长格式）
- **粒度**：计量点级别（Metering Point）
- **集合示例**：
  - `prediction_curves` - 预测曲线
  - `settlement_curves` - 结算曲线（RPA爬取）
- **数据结构**（每条记录为一个时间点）：
```python
{
    "_id": ObjectId("..."),
    "timestamp": datetime(2025, 11, 10, 0, 15, 0),
    "customer_id": "...",
    "metering_point_id": "...",
    "value": 23.1,
    "data_source": "prediction",  # or "settlement"
    "created_at": datetime.now()
}
```

**3. 聚合数据层（Aggregated Data Layer）**
- **用途**：预聚合的客户级别曲线，优化前端查询性能
- **集合示例**：`customer_load_aggregates`
- **生成方式**：后台任务或事件驱动（数据导入后自动触发）
- **数据结构**：
```python
{
    "_id": ObjectId("..."),
    "timestamp": datetime(2025, 11, 10, 0, 15, 0),
    "customer_id": "...",
    "total_value": 123.5,  # 该客户所有计量点的总和
    "metering_point_count": 5,
    "data_source": "prediction",
    "last_updated": datetime.now()
}
```

**架构原则**：
- ✅ **数据纯洁性**：维护独立的预测/结算曲线，禁止数据库层面的混合
- ✅ **内存借用**：数据融合只在应用层内存中进行（如预测优先，结算补充）
- ✅ **预聚合**：通过后台任务生成客户级聚合数据，避免前端实时计算
- ✅ **索引优化**：为时间戳、客户ID、计量点ID创建组合索引

#### 3.5 聚合管道示例

**示例1：客户级负荷聚合**
```python
def aggregate_customer_load(customer_id: str, date: datetime) -> list:
    """
    聚合指定客户在指定日期的所有计量点负荷

    Returns:
        按时间排序的聚合数据列表
    """
    start_of_day = date.replace(hour=0, minute=0, second=0, microsecond=0)
    end_of_day = start_of_day + timedelta(days=1)

    pipeline = [
        # 1. 筛选条件
        {
            "$match": {
                "customer_id": customer_id,
                "timestamp": {"$gt": start_of_day, "$lte": end_of_day}
            }
        },
        # 2. 按时间分组并求和
        {
            "$group": {
                "_id": "$timestamp",
                "total_value": {"$sum": "$value"},
                "metering_point_count": {"$sum": 1}
            }
        },
        # 3. 重命名字段
        {
            "$project": {
                "_id": 0,
                "timestamp": "$_id",
                "total_value": 1,
                "metering_point_count": 1
            }
        },
        # 4. 排序
        {"$sort": {"timestamp": 1}}
    ]

    return list(collection.aggregate(pipeline))
```

```python
            "expected_points": 96,
            "actual_points": 95,
            "completeness": 98.96,
            "missing_timestamps": ["2025-11-10T12:30:00"]
        }
    """
    start_of_day = date.replace(hour=0, minute=0, second=0, microsecond=0)
    end_of_day = start_of_day + timedelta(days=1)

    # 查询实际数据点
    actual_data = list(collection.find({
        "customer_id": customer_id,
        "timestamp": {"$gt": start_of_day, "$lte": end_of_day}
    }).sort("timestamp", 1))

    # 生成期望的时间戳列表（96个点）
    expected_timestamps = [
        start_of_day + timedelta(minutes=15 * i)
        for i in range(1, 97)  # 从第1个点（00:15）到第96个点（次日00:00）
    ]

    # 找出缺失的时间戳
    actual_timestamps = {d["timestamp"] for d in actual_data}
    missing_timestamps = [
        ts.isoformat() for ts in expected_timestamps
        if ts not in actual_timestamps
    ]

    return {
        "expected_points": 96,
        "actual_points": len(actual_data),
        "completeness": round(len(actual_data) / 96 * 100, 2),
        "missing_timestamps": missing_timestamps
    }
```

```python
def get_user_with_orders(user_id: str):
    pipeline = [
        {"$match": {"_id": ObjectId(user_id)}},
        {"$lookup": {
            "from": "orders",
            "localField": "_id",

def get_users_with_order_counts(user_ids: List[str]):

            "localField": "_id",
            "foreignField": "user_id",
```

#### 5.2 事务处理规范（MongoDB 4.0+）

```python
from pymongo.errors import ConnectionFailure

def transfer_funds(from_account: str, to_account: str, amount: float):
    """资金转账示例（需要事务）"""
    # 开启事务
    with self.db.client.start_session() as session:
        with session.start_transaction():
            try:
                # 扣款
                result1 = self.collection.update_one(
                    {"account_id": from_account, "balance": {"$gte": amount}},
                    {"$inc": {"balance": -amount}},
                    session=session
                )
                if result1.matched_count == 0:
                    raise ValueError("余额不足")

                # 入账
                self.collection.update_one(
                    {"account_id": to_account},
                    {"$inc": {"balance": amount}},
                    session=session
                )

                # 提交事务
                session.commit_transaction()
                logger.info(f"转账成功: {from_account} -> {to_account}, 金额: {amount}")

            except Exception as e:
                # 回滚事务
                session.abort_transaction()
                logger.error(f"转账失败: {str(e)}")
                raise
```

**事务使用原则：**
- 仅在**必须保证原子性**的操作中使用事务（如资金转账、库存扣减）
- 事务会影响性能，避免过度使用
- 确保 MongoDB 版本 >= 4.0 且使用副本集
