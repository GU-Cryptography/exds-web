# 用户与权限页面 UI 设计方案（v1.2）

## 1. 设计目标

1. 管理员在一个页面完成：角色管理、用户管理、权限分配、审计查询。
2. 保持高效率：少跳转、少弹窗、核心操作可批量。
3. 保持高安全：高风险变更可预览差异并二次确认。

## 2. 页面布局（桌面端）

采用“三栏工作台 + 右侧任务抽屉”结构。

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ 顶部工具条：全局搜索 | 角色筛选 | 用户状态筛选 | 刷新 | 最近变更提示         │
├─────────────────────┬────────────────────────────────────┬───────────────────┤
│ 左栏：角色列表       │ 中栏：用户列表（主工作区）          │ 右栏：权限详情面板   │
│ - 角色卡片           │ - 表格/分页/批量勾选               │ - 模块 view/edit 矩阵│
│ - 新建/复制角色      │ - 新建用户/分配角色/启停/重置密码   │ - 例外权限区         │
│ - 系统角色只读标识   │ - 行内快捷操作                     │ - 变更影响预览       │
├─────────────────────┴────────────────────────────────────┴───────────────────┤
│ 底部状态条：当前选中对象、待提交变更数、最后一次保存时间                     │
└──────────────────────────────────────────────────────────────────────────────┘
```

## 3. 关键交互设计

## 3.1 角色管理

1. 左栏点击角色后，中栏用户自动过滤到该角色。
2. 新建角色支持“复制现有角色”快速生成。
3. 系统内置角色（如 `super_admin`）展示只读锁标识。

## 3.2 用户管理

1. 表格列：用户名、显示名、角色、状态、最近活跃、操作。
2. 行内操作：分配角色、启停、重置密码。
3. 批量操作：批量分配角色、批量启停（高风险需确认）。

## 3.3 权限编辑

1. 权限矩阵行=模块，列=`view`/`edit`。
2. 联动规则：勾选 `edit` 自动勾选 `view`；取消 `view` 自动取消 `edit`。
3. 高风险例外权限（如 `data:critical:delete`）单独分组并高亮警示。

## 3.4 提交确认

1. 点击保存后弹出“变更差异对比”：
- 新增权限
- 移除权限
- 受影响用户数
2. 若包含高风险例外权限，要求输入确认文本后才可提交。

## 4. 组件清单（基于 MUI）

1. 页面框架：`Box` + `Grid` + `Paper`。
2. 左栏角色区：`List`、`ListItemButton`、`Chip`、`IconButton`。
3. 中栏用户区：`DataGrid`（或 `Table`）+ `Pagination` + `Toolbar`。
4. 右栏权限区：`Accordion`、`Switch`、`Checkbox`、`Alert`。
5. 抽屉与弹窗：`Drawer`、`Dialog`、`DialogContent`、`DialogActions`。
6. 状态反馈：`Snackbar`、`LinearProgress`、`Tooltip`。

## 5. API 对接映射

1. 当前用户上下文：`GET /api/v1/auth/me`
2. 权限字典：`GET /api/v1/auth/permissions`
3. 角色管理：
- `GET /api/v1/auth/roles`
- `POST /api/v1/auth/roles`
- `PUT /api/v1/auth/roles/{role_code}`
- `PUT /api/v1/auth/roles/{role_code}/permissions`
4. 用户管理：
- `GET /api/v1/auth/users`
- `POST /api/v1/auth/users`
- `PUT /api/v1/auth/users/{username}/roles`
- `PUT /api/v1/auth/users/{username}/status`
- `PUT /api/v1/auth/users/{username}/password/reset`
5. 审计查询：`GET /api/v1/auth/audit-logs`

## 6. 权限与可见性规则

1. 页面访问：需 `module:system_user_auth:view`。
2. 页面写操作：需 `system:auth:manage`。
3. 无 `system:auth:manage` 时：
- 页面只读可见（如允许）或路由拒绝（按当前路由策略）。
- 所有写按钮禁用并展示权限提示。

## 7. 响应式策略（移动端）

1. 三栏降级为三段页签：`角色` / `用户` / `权限`。
2. 编辑动作统一走底部抽屉，不使用多层弹窗。
3. 表格降级为卡片列表，保留关键字段与主操作。

## 8. 验收标准（UI）

1. 管理员可在 3 步内完成“新建角色并分配给用户”。
2. 高风险权限变更必须经过差异确认。
3. 所有操作有成功/失败反馈，失败可定位具体原因。
4. 页面在桌面与移动端均可完整完成核心流程。

## 9. 实施建议顺序（前端）

1. 先完成页面骨架与数据读取（只读态）。
2. 再完成角色/用户 CRUD。
3. 最后接权限矩阵编辑、差异确认与审计查询。
