# 用户权限管理实施计划 1.1（基于两档权限模型 + 例外规则）

## 一、版本说明

- 版本：1.1
- 日期：2026-03-20
- 依据：`docs/pages/用户权限管理/前端写请求-操作路径-权限码清单.md` 第7章
- 相对1.0的核心变化：
  - 权限模型采用“模块两档（view/edit）+ 高风险例外权限”
  - 鉴于 1.0 动作级权限未实际运行（仅测试数据），1.1 采用直接切换方案，不走双轨兼容
  - 实施目标由“先拦截写请求”升级为“入口、页面、按钮、请求、后端一致化前置体验”

---

## 二、目标与原则

### 2.1 目标

1. 用户在无权限场景下，尽量在最早阶段得到明确反馈，不再出现“填完表单才提示无权限”。
2. 全项目写操作实现统一前置判定：`R + B + Q + S`。
3. 角色配置简化为“模块两档权限”，降低维护成本。

### 2.2 原则

1. 最小改动优先：先做权限框架与前置能力，不做无关重构。
2. 后端兜底必须保留：前端前置改善体验，后端权限保障安全。
3. 一致性优先：桌面端与移动端的无权限行为保持一致。

---

## 三、权限模型（1.1基线）

### 3.1 模块两档权限

每个菜单模块定义两档权限：

- `module:{module_code}:view`：可进入模块并查看数据（菜单可见、路由可进入）
- `module:{module_code}:view`：可查看模块数据（列表/详情/查询）
- `module:{module_code}:edit`：可修改模块数据（新增/编辑/删除/导入/同步/上传/重算等）

继承关系：

- `edit` 包含 `view` 和 `open`
- `view` 包含 `open`

### 3.2 例外规则（高风险独立权限）

以下不并入 `module:*:edit`：

1. `system:logs:resolve`（告警处理）
2. `system:data_access:manage`（RPA任务重试/控制）
3. `settlement:recalc:execute`（大规模结算重算）
4. `data:critical:delete`（关键业务数据删除）

## 四、无权限打开页面的最优展示设计

> 结论：采用“默认置灰+锁（无 open）+ 深链友好拒绝页 + 可执行下一步”是最优方案。

### 4.1 交互策略（推荐）

1. 菜单层（无 `open`）：
- 默认采用“置灰 + 锁图标 + 悬浮提示无权限”。
- 菜单项保持可见但不可点击，减少“菜单消失”的认知负担。
- 悬浮提示文案统一：`无权限（需要 module:{module_code}:view）`。

2. 路由层（手工输入URL或书签进入）：
- 不重定向登录页，不闪退。
- 展示统一“访问受限”页：
  - 标题：无权访问该页面
  - 说明：缺少 `module:*:view` 或例外权限
  - CTA：返回上一页 / 回到首页 / 申请权限（复制权限码）

3. 页面层（有 `open` 无 `view`）：
- 页面框架可进入，但内容区显示“无查看权限”空态。
- 不触发数据请求，避免无效网络流量。

4. 按钮层（有 `view` 无 `edit`）：
- 所有写按钮禁用或隐藏（建议业务页禁用并说明原因，避免用户困惑）。
- 提示文案统一：`当前角色无修改权限（需要 module:{x}:edit）`。

5. 请求层：
- 请求发起前做 `Q` 拦截（兜住漏网按钮）。
- 后端继续 `S` 兜底。

### 4.2 统一视觉与文案规范

1. 使用统一组件：`AccessDeniedState`。
2. 统一文案分级：
- 页面拒绝：`无权访问该页面`
- 数据拒绝：`无查看权限`
- 操作拒绝：`无修改权限`
3. 展示字段建议：
- 当前模块名
- 缺失权限码（可复制）
- 联系管理员按钮

### 4.3 性能与体验细节

1. `isPermissionLoaded` 前不渲染受保护内容，避免“先显示再消失”闪烁。
2. 首屏权限加载失败时展示可重试状态，不直接白屏。
3. 移动端与桌面端使用同一权限判定函数，避免行为不一致。

## 五、实施范围与分期

### 阶段A：权限字典与路由元数据（0.5天）

1. 为每个菜单模块定义 `module_code`。
2. 在 `routes/menu` 增加 `requiredOpenPermission`、`requiredViewPermission`、`requiredEditPermission`。
3. 输出《模块权限字典》作为唯一来源。

**验收**：权限字典可覆盖全部菜单路由。

### 阶段B：后端权限补齐（1.5天）

1. 将“未挂 `require_permission` 的写接口”全部补齐。
2. 对例外规则接口挂独立权限码。
3. 鉴于动作级权限未落地生产，后端直接切换为三档+例外判定，不保留双轨逻辑。

**验收**：任意写接口都可被后端权限控制。

### 阶段C：前端统一前置框架（1天）

1. 菜单过滤接入 `open`。
2. 路由守卫接入 `open/view`。
3. 全局写请求拦截接入 `edit + 例外权限`。

**验收**：无权限用户无法进入无权页面，且不会发出无权写请求。

### 阶段D：页面按钮治理（2天）

1. 按模块清理所有写按钮（新增/编辑/删除/导入/重算/上传）。
2. 统一按钮禁用态和提示文案。
3. 对“高风险操作”单独校验例外权限。

**验收**：不再出现“填完才报无权限”。

### 阶段E：角色配置与数据初始化（0.5天）

1. 更新 `auth_permissions`、`auth_roles` 初始化脚本。
2. 角色模板：`viewer/analyst/business_admin/system_admin`。
3. 输出回滚与迁移指引（回滚到“仅登录可访问”模式）。

**验收**：初始化后权限行为与 1.1 字典一致。

### 阶段F：联调与回归（1天）

1. 按角色全链路测试（重点：客户档案、合同、套餐、结算、系统管理）。
2. 回归移动端路径。

**验收**：关键场景通过率100%，无P1/P2权限缺陷。

## 七、验收标准（必须同时满足）

1. 无 `open`：菜单置灰+锁且不可点击，鼠标悬浮提示“无权限”；深链进入显示“访问受限”页。
2. 有 `open` 无 `view`：可进入页面壳，但不发数据请求，显示“无查看权限”。
3. 有 `view` 无 `edit`：所有写按钮禁用/隐藏，点击不给业务弹窗入口。
4. 任一漏网写操作在请求层被拦截（Q），后端仍能兜底（S）。
5. viewer 在客户档案管理等模块不再出现“填完后才报错”的体验。

---

## 八、风险与应对

1. 风险：三档模型落地时模块边界不清。  
- 应对：先冻结 `module_code`，变更走评审。

2. 风险：例外权限漏挂。  
- 应对：例外接口建立白名单与自动检查脚本。

3. 风险：前端判定与后端判定不一致。  
- 应对：以后端权限字典为源，前端映射自动生成。

---

## 九、建议优先级

1. 第一优先：客户档案管理、零售合同管理、零售套餐管理（高频业务写）。
2. 第二优先：结算重算、系统告警、RPA任务（高风险例外）。
3. 第三优先：其余导入与分析触发类接口。

1. create/edit/view 详情子路由默认继承其主路由 `module_code`。例如：
- `/customer/profiles/create`

- 先判 `module:*` 两档权限。

## 十一、数据模型与集合设计（1.1补充）

### 11.1 设计结论

1. 运行时权限数据以数据库为主（可热更新）。  
2. 代码仅保存初始化基线（seed），用于首装与补齐。  
3. 鉴于 1.0 动作级权限未落地生产，1.1 不引入旧模型兼容表。  

### 11.2 集合设计

#### A. `auth_modules`（新增，模块字典）

用途：管理菜单模块与两档权限元数据。

字段建议：

- `module_code: str`（唯一，如 `customer_profiles`）
- `module_name: str`（显示名，如“客户档案管理”）
- `menu_group: str`（上级菜单，如“客户管理”）
- `route_paths: string[]`（模块路由集合）
- `is_active: bool`
- `sort_order: int`
- `created_at: datetime`
- `updated_at: datetime`

索引：

- `unique(module_code)`
- `index(menu_group, sort_order)`

#### B. `auth_permissions`（增强）

用途：权限点定义，覆盖两档权限和例外权限。

字段建议：

- `code: str`（唯一，如 `module:customer_profiles:view`）
- `name: str`（权限名称）
- `permission_type: str`（`module_open`/`module_view`/`module_edit`/`exception`）
- `module_code: str | null`（例外权限可为空）
- `description: str`
- `is_exception: bool`
- `is_active: bool`
- `created_at: datetime`
- `updated_at: datetime`

索引：

- `unique(code)`
- `index(module_code, permission_type)`
- `index(is_exception, is_active)`

#### C. `auth_roles`（增强）

用途：角色定义及权限集合。

字段建议：

- `code: str`（唯一，如 `viewer`）
- `name: str`
- `description: str`
- `permissions: string[]`（权限码集合）
- `is_system: bool`
- `is_active: bool`
- `created_at: datetime`
- `updated_at: datetime`

索引：

- `unique(code)`
- `index(is_active)`

#### D. `users`（沿用）

用途：用户与角色绑定。

关键字段：

- `username`
- `roles: string[]`
- `is_active`
- `last_active_at`

索引建议：

- `unique(username)`
- `index(roles)`

### 11.3 示例文档（简化）

`auth_modules`：

```json
{
  "module_code": "customer_profiles",
  "module_name": "客户档案管理",
  "menu_group": "客户管理",
  "route_paths": ["/customer/profiles", "/customer/profiles/create", "/customer/profiles/edit/:customerId"],
  "is_active": true,
  "sort_order": 10
}
```

`auth_permissions`：

```json

  "code": "module:customer_profiles:edit",
  "name": "客户档案-可修改",
  "permission_type": "module_edit",
  "module_code": "customer_profiles",
  "is_exception": false,
  "is_active": true
}
```

## 十二、初始化与迁移策略（1.1补充）

### 12.1 前提

1.0 动作级权限未实际运行，集合中仅测试数据，不作为兼容基线。

### 12.2 迁移策略

1. 清理或归档测试权限数据（保留备份快照）。
2. 以 1.1 字典为唯一来源重建：
- `auth_modules`
- `auth_permissions`
- `auth_roles`
3. 给现有用户分配新角色（默认 `viewer`，管理员 `system_admin`）。

### 12.3 初始化脚本要求

`webapp/scripts/init_auth_data.py` 需满足：

1. 幂等：可重复执行，不重复插入。
2. 补齐：缺失记录自动补齐。
3. 保护：非系统角色不被脚本覆盖（仅提示差异）。
4. 输出：执行日志包含新增/更新/跳过统计。

### 12.4 回滚策略

1. 备份集合：`auth_modules/auth_permissions/auth_roles/users`。
2. 回滚方式：
- 恢复集合快照，或
- 临时关闭授权开关（仅应急），随后尽快恢复。

## 十三、角色模板初稿（1.1补充）

### 13.1 `viewer`

- 原则：全模块 `open+view`，无 `edit`，无例外权限。

### 13.2 `analyst`

- 原则：分析预测相关模块 `open+view+edit`，业务管理模块通常仅 `open+view`。
- 默认不授予例外权限。

### 13.3 `business_admin`

- 原则：业务模块 `open+view+edit`，可按需授予 `settlement:recalc:execute`。

### 13.4 `system_admin`

- 原则：全模块 `open+view+edit` + 全部例外权限。

### 14.1 角色-场景矩阵（最低要求）
