# 前端写请求与权限映射清单（全项目）

更新时间：2026-03-20  
范围：`frontend/src` 中所有 `apiClient.post/put/patch/delete` 写请求，结合 `webapp/api` 中 `require_permission` 现状梳理。

## 1. 前置方式约定

- `R`：路由/入口前置（无权限不允许进入 create/edit 子路由）
- `B`：按钮前置（按钮隐藏或禁用，并给出“无权限”提示）
- `Q`：请求前置（`apiClient` request interceptor 拦截）
- `S`：服务端兜底（后端 `require_permission` 返回 403）

推荐标准：
- 页面级写操作：`R + B + Q + S`
- 弹窗/行内写操作：`B + Q + S`
- 系统级写操作：`B + Q + S`

## 7. 简化权限模型（每个菜单模块两档）+ 例外规则

### 7.1 两档权限模型（主干）

对每个菜单模块统一定义两档权限：

- `module:{module_code}:view`：可打开模块（菜单可见、路由可进入）
- `module:{module_code}:view`：可查看（查询、列表、详情）
- `module:{module_code}:edit`：可修改（该模块全部写操作：新增、编辑、删除、导入、重算、上传、同步等）

继承关系：

- `edit` 自动包含 `view` 和 `open`
- `view` 自动包含 `open`

前置实施标准：

- `open`：菜单和路由前置（`R`）
- `view`：页面数据加载前置（`R/B`，无查看权限直接提示并终止加载）
- `edit`：按钮和请求前置（`B+Q`）+ 后端兜底（`S`）

### 7.2 例外规则（高风险操作单独权限）

以下操作不并入通用 `module:*:edit`，要求额外权限码：

1. 系统告警处理  
- 接口：`POST /api/v1/system/alerts/{id}/resolve`  
- 例外权限：`system:logs:resolve`

2. 数据下载任务重试/控制  
- 接口：`POST /api/v1/rpa/tasks/{pipeline}/{task}/retry`  
- 例外权限：`system:data_access:manage`

3. 大规模结算重算（建议例外）  
- 接口：`POST /api/v1/settlement/calculate`、`POST /api/v1/retail-settlement/calculate`、`POST /api/v1/retail-settlement/monthly-calc`  
- 例外权限：`settlement:recalc:execute`

4. 关键删除（建议例外）  
- 接口：客户/合同/套餐/意向客户等删除类接口  
- 例外权限：`data:critical:delete`

### 7.3 角色配置建议（基于三档 + 例外）

- `viewer`：仅配置各模块 `open/view`，不配置任何 `edit` 与例外权限
- `analyst`：相关分析模块配置 `view/edit`；例外权限默认不授予
- `business_admin`：业务模块配置 `view/edit`；按需授予 `settlement:recalc:execute`
- `system_admin`：全模块 `view/edit` + 全部例外权限

### 7.4 与当前清单映射原则

- 本文第 2/3 章中的“建议权限码”可逐步收敛为 `module:*:(view/edit)`。
- 第 7.2 节列出的接口保持独立权限，不被 `module:*:edit` 覆盖。
- 迁移期允许“双轨并存”：旧动作级权限与新两档权限同时生效；稳定后移除旧权限。
