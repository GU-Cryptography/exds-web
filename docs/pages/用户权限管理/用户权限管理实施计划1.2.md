# 用户权限管理实施计划 1.2

## 1. 版本目标

本版本在 1.1 基础上，按以下原则推进：

1. 权限模型保持“模块两档 + 例外权限”：`view/edit`。
2. `view` 无权限保持当前策略：菜单不可点、路由拒绝访问（不实现页面壳空态）。
3. 实施顺序调整为：先前端管理闭环，再逐步落地安全增强。

## 2. 范围边界

### 2.1 本期纳入

1. 前端“用户与权限”管理台完整落地。
2. 按钮级 `edit` 权限治理查漏补缺并清单化。
3. 角色场景回归报告（viewer/analyst/business_admin/system_admin）。
4. 安全增强分步实施（会话互斥、首登安全设置、新设备邮件验证、登录审计查询）。

### 2.2 本期不纳入

1. `view` 无权限页面壳保留 + 内容空态（明确不做）。
2. 数据域隔离（组织/客户行级权限）。
3. 密码过期策略（90/180 天）。

## 3. 实施阶段

## 阶段 A（前端优先）：权限管理闭环

### A1. 用户与权限管理台

目标：替换 `/system-settings/user-permissions` 占位页，打通管理闭环。

交付：
1. 角色管理：列表、创建、编辑名称描述、分配权限。
2. 用户管理：列表、创建、分配角色、启停、重置密码。
3. 审计查看：按操作人/事件/时间范围筛选。
4. 页面内所有写操作按钮接 `system:auth:manage`。
5. UI 与交互按《用户与权限页面 UI 设计方案（v1.2）》执行。

接口复用：`/api/v1/auth/*` 现有接口。
UI 方案文档：`docs/pages/用户权限管理/用户与权限页面UI设计方案.md`

### A2. 按钮级 `edit` 权限全量核查

目标：清零“按钮可点但最终保存报无权限”的体验问题。

交付：
1. 全页面写按钮巡检清单（新增/编辑/删除/导入/上传/重算/触发）。
2. 每个按钮绑定 `edit` 或例外权限，缺权统一禁用/隐藏与提示文案。
3. 更新前端写请求映射清单与代码一致性。

### A3. 角色场景回归报告

交付：
1. 四类角色回归矩阵（页面访问、按钮可用、写请求结果）。
2. 关键页面截图与异常案例闭环。

## 阶段 B（安全增强第 1 批）：会话可控 + 首登安全

### B1. 会话互斥登录（单账号单会话）

交付：
1. 新增 `auth_sessions` 集合与索引。
2. 登录冲突检测、强制登录确认、踢旧会话。
3. token 增加 `sid`，受保护接口校验会话状态。

### B2. 首次登录强制改密 + 邮箱绑定验证

交付：
1. 登录返回 `required_actions`（如 `CHANGE_PASSWORD/BIND_EMAIL/VERIFY_EMAIL`）。
2. 首登安全设置流程页与后端对应接口。
3. 完成动作前不签发业务会话。

## 阶段 C（安全增强第 2 批）：新设备邮件验证 + 审计查询增强

### C1. 新设备邮件验证码

交付：
1. `auth_trusted_devices`、`auth_login_challenges` 集合。
2. `email_mfa_enabled` 开关能力。
3. 新设备 challenge 流程（登录 202 -> 验证 -> 发 token）。

### C2. 登录审计与在线会话查询

交付：
1. 登录成功/失败、登出、被踢、超时事件入审计。
2. 会话历史与在线用户查询接口（管理端可用）。

## 4. 开发与验证门禁

每个阶段完成后必须通过：

```bash
.venv/Scripts/python scripts/check_auth_all.py
npm run build --prefix frontend
```

并补充：
1. 阶段回归记录（角色矩阵）。
2. 接口变更与权限码映射说明。
3. 风险与回滚说明。

## 5. 里程碑建议

1. M1（阶段 A 完成）：前端权限管理闭环上线。
2. M2（阶段 B 完成）：单账号互斥 + 首登安全上线。
3. M3（阶段 C 完成）：新设备邮件验证 + 登录审计查询上线。

## 6. 验收标准（1.2）

1. `view` 缺失用户无法从菜单进入页面，深链访问被拒绝。
2. 所有写按钮都具备前置权限控制，不再出现“填写后保存才报无权限”的主流程问题。
3. `/system-settings/user-permissions` 可完成角色与用户管理全流程。
4. 会话安全增强能力按阶段通过用例验证。
5. 自动检查与构建检查持续通过。
