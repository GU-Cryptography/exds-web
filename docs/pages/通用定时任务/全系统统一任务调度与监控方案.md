# 全系统统一任务调度与监控方案

> **版本**: 1.0  
> **状态**: 拟定中  
> **适用范围**: EXDS-Web (Web端), EXDS-Forecast (预测端), EXDS-RPA (数据采集端)

## 1. 概述

### 1.1 背景
当前电力交易辅助决策系统包含三个核心子系统：
1.  **Web 子系统 (exds-web)**: 提供用户界面、业务逻辑、报表展示。
2.  **预测子系统 (exds-stef)**: 执行价格预测、负荷预测、模型训练。
3.  **RPA 子系统 (exds-rpa)**: 负责从交易中心等外部平台自动抓取数据。

三个子系统均有独立的定时任务需求，且需要相互配合（如：RPA下载数据 -> 触发Web缓存刷新 -> 触发预测计算）。目前缺乏统一的状态监控、日志记录和告警机制，导致运维困难，故障排查链路断裂。

### 1.2 设计目标
1.  **分布式执行，集中式监控**: 各子系统保留各自的任务调度/执行能力（Web用APScheduler, 预测/RPA用Native Service），但统一将状态上报至共享数据库。
2.  **标准化数据模型**: 制定统一的日志、告警、命令数据结构，消除"数据孤岛"。
3.  **全链路可观测**: 在 Web 端提供统一的监控仪表盘，能够追踪跨系统的业务流程。
4.  **互操作性**: 支持通过 Web 端下发指令触发后端子系统的任务（如手动触发补测）。

## 3. 核心数据模型 (MongoDB Schema) 

为实现统一监控，需严格定义以下三个集合的 Schema。 

### 3.1 任务执行日志集合 (`task_execution_logs`)

记录所有任务的单次执行流水。

#### 3.1.1 字段定义

| 字段名 | 类型 | 必填 | 说明 | 示例 | 
| :--- | :--- | :--- | :--- | :--- |
| `_id` | ObjectId | Y | 自动生成 | |
| `task_id` | String | Y | **全局唯一ID**，格式 `{service}_{type}_{timestamp}_{uuid}` | `forecast_d1_price_20251215_x9d8s` |
| `service_type` | String | Y | **一级分类**：服务来源 | `forecast` |
| `task_type` | String | Y | **二级分类**：具体的任务类型的代码 | `d1_price_pred` |
| `task_name` | String | Y | 任务显示名称（用于前端展示） | `D-1 日前价格预测` |
| `status` | String | Y | 执行状态 | `SUCCESS` |
| `start_time` | Date | Y | 开始时间 | `2025-12-15T09:00:00Z` |
| `end_time` | Date | N | 结束时间（RUNNING时为空） | `2025-12-15T09:05:00Z` |
| `duration` | Float | N | 耗时（秒） | `300.5` |
| `summary` | String | N | 执行结果摘要 | `预测成功，生成96个点数据` |
| `details` | Object | N | **扩展详情**，不同任务类型结构不同 | `{"point_count": 96, "model_ver": "v1.0"}` |
| `error` | Object | N | 错误信息 | `{"code": "DB_ERR", "msg": "..."}` |
| `trigger_type` | String | Y | 触发方式 | `schedule` (定时), `manual` (手动), `event` (事件) |

#### 3.1.2 分类枚举字典

**Service Type (一级分类)**

*   `web`: Web 后端服务
*   `forecast`: 预测服务
*   `rpa`: RPA 数据采集服务
*   `system`: 基础设施/运维

**Task Type (二级分类) - 推荐值**

| Service | Task Type | 说明 | details 扩展字段建议 |
| :--- | :--- | :--- | :--- |
| **web** | `cache_refresh` | 缓存刷新 | `{"keys_refreshed": ["k1", "k2"]}` |
| | `data_archive` | 数据归档 | `{"archived_count": 1000}` |
| | `report_gen` | 报表生成 | `{"report_id": "..."}` |
| **forecast** | `d1_price` | D-1 价格预测 | `{"target_date": "...", "model_version": "..."}` |
| | `load_pred` | 负荷预测 | `{"customer_count": 50}` |
| | `model_train` | 模型训练 | `{"accuracy": 0.95}` |
| **rpa** | `rpa_login` | 自动登录操作 | `{"result": "success", "user": "xxx"}` |
| | `rpa_data_batch_download` | 批次数据下载 | `{"total_items": 22, "successful_items": 20, "failed_items": 2, "download_items_summary": [{"name": "日前价格", "status": "SUCCESS"}, {"name": "日内负荷", "status": "FAILED", "error_msg": "数据缺失"}]}` |
| | `rpa_single_item_download` | 单项数据下载 | `{"item_name": "xxxx", "collection": "xxxx"}` |

### 3.2 系统告警集合 (`system_alerts`)

记录系统运行过程中的异常情况，支持确认和解决流程。

#### 3.2.1 字段定义

| 字段名 | 类型 | 必填 | 说明 | 示例 |
| :--- | :--- | :--- | :--- | :--- |
| `_id` | ObjectId | Y | | |
| `alert_id` | String | Y | 唯一告警ID | `alert_20251215_xxxxx` |
| `level` | String | Y | 告警级别 | `P1` |
| `category` | String | Y | **告警分类** | `DATA_MISSING` |
| `title` | String | Y | 告警标题 | `交易中心数据下载失败` |
| `content` | String | Y | 详细内容 | `登录验证码识别超时...` |
| `service_type` | String | Y | 来源服务 | `rpa` |
| `task_type` | String | N | 关联任务类型 | `download_market` |
| `related_task_id`| String | N | 关联的任务执行ID | |
| `status` | String | Y | 处理状态 | `ACTIVE` |
| `created_at` | Date | Y | 发生时间 | |
| `resolved_at` | Date | N | 解决时间 | |
| `context` | Object | N | 上下文数据 | `{"retry_count": 3}` |

#### 3.2.2 枚举字典

**Alert Level (告警级别)**

*   `P0`: **阻断性故障**。系统完全不可用，或关键业务逻辑无法执行（如：交易日无法获取明日电价）。需要立即介入。
*   `P1`: **严重故障**。某个具体任务失败，但系统部分功能仍可用（如：某单个客户负荷预测失败）。
*   `P2`: **警告**。非预期行为，但自动恢复或不影响核心业务（如：RPA 第一次登录失败，重试后成功）。

**Alert Category (告警分类)**

*   `TASK_FAILED`: 任务执行抛出异常。
*   `TASK_TIMEOUT`: 任务执行时间超过阈值。
*   `DATA_MISSING`: 依赖的前置数据缺失（如：预测时发现没有天气数据）。
*   `LOGIN_FAILED`: 外部系统登录失败（RPA专用）。
*   `VALIDATION_ERR`: 数据校验不通过（如：下载的数据字段缺失）。
*   `SYSTEM_ERR`: 基础设施错误（数据库连接失败、磁盘满）。

**Alert Status (处理状态)**

*   `ACTIVE`: 发生中，未处理。
*   `ACKNOWLEDGED`: 已确认（运维人员已知晓，正在处理）。
*   `RESOLVED`: 已解决。
*   `IGNORED`: 已忽略（误报或无需处理）。

### 3.4 告警状态自动流转机制

为了减少无效告警骚扰，系统应实现基于任务执行结果的告警自动状态管理。

#### 3.4.1 自动解决 (Auto-Resolve)
*   **触发条件**: 当某个任务执行记录写入且状态为 `SUCCESS` 时。
*   **执行逻辑**: 系统自动查找 `system_alerts` 集合中同时满足以下条件的告警：
    1.  `status` 为 `ACTIVE` 或 `ACKNOWLEDGED`。
    2.  `service_type` 和 `task_type` 与当前成功任务一致。
    3.  `category` 属于可自动恢复类型（如 `TASK_FAILED`, `TASK_TIMEOUT`, `LOGIN_FAILED`, `DATA_MISSING`）。
    4.  *(高级匹配)* 如果告警 `context` 和任务 `details` 中都包含关键业务键（如 `target_date`, `customer_id`），则需精确匹配才消除。
*   **动作**: 将匹配告警的 `status` 更新为 `RESOLVED`，设置 `resolved_at` 为当前时间，并在 `resolution_note` 中自动备注 *"系统检测到后续任务 (task_id: xxx) 执行成功，自动消除"*。

#### 3.4.2 示例场景
1.  **09:00** RPA 执行 "交易中心数据下载" 失败，生成一条 `P1` 级 `TASK_FAILED` 告警（状态 `ACTIVE`）。
2.  **09:15** 运维人员收到通知，但未操作。
3.  **09:30** RPA 自动重试机制再次执行该任务，本次执行 **成功**。
4.  **系统行为**: 监测到成功日志，自动找到 09:00 的那条告警，将其状态改为 `RESOLVED`。运维看板上的"未解决告警"数字自动减一。

## 4. Web 子系统内部调度设计

EXDS-Web 自身也需要一个轻量级的调度器。

### 4.1 技术选型
使用 **APScheduler** (`AsyncIOScheduler`)，完美集成到 FastAPI 的异步上下文中。

### 4.2 目录结构
```
webapp/
├── scheduler/
│   ├── __init__.py           # 导出 Scheduler 实例
│   ├── core.py               # Scheduler 生命周期管理 (start/shutdown)
│   ├── jobs/                 # 具体任务逻辑
│   │   ├── __init__.py
│   │   ├── system_jobs.py    # 系统级任务 (清理日志等)
│   │   └── business_jobs.py  # 业务级任务 (缓存刷新等)
│   └── logger.py             # 封装向 MongoDB 写入 task_execution_logs 的逻辑
```

### 4.3 核心逻辑代码示例

```python
# webapp/scheduler/core.py

from apscheduler.schedulers.asyncio import AsyncIOScheduler
from webapp.scheduler.jobs.system_jobs import clean_expired_tokens

scheduler = AsyncIOScheduler()

def setup_scheduler(app):
    # 注册任务
    scheduler.add_job(
        clean_expired_tokens, 
        'interval', 
        hours=24, 
        id='web_clean_tokens',
        replace_existing=True
    )
    
    @app.on_event("startup")
    async def start_scheduler():
        scheduler.start()
        
    @app.on_event("shutdown")
    async def stop_scheduler():
        scheduler.shutdown()
```

## 5. 前端功能规划

在 Web 端新增 **"系统监控"** 模块。

### 5.1 任务监控看板 (Task Dashboard)
*   **状态概览**: 三个卡片展示 Web、Forecast、RPA 三个子系统当天的健康度（成功率/告警数）。
*   **执行时间轴**: 使用 Gantt 图或垂直时间轴，展示今天所有任务的执行序列。
    *   *场景*: 可以清晰看到 RPA 9:00 开始下载 -> 9:05 下载结束 -> 9:06 触发预测 -> 9:10 预测完成。
*   **日志列表**: 支持按 Service Type, Task Type, Status, Date 筛选。

### 5.2 告警中心 (Alert Center)
*   **未读告警列表**: 默认展示 `status=ACTIVE` 的告警。
*   **操作**: 提供 "确认(Acknowledge)" 和 "解决(Resolve)" 按钮。
*   **通知配置**: (未来规划) 配置邮件或钉钉机器人推送。

### 5.3 任务控制台 (Task Control)
*   **手动触发**: 提供表单，选择 Service -> Task Type -> 输入参数（如日期） -> 发送指令。
*   **指令追踪**: 查看命令的执行状态。
