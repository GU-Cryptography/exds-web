# 任务调度系统开发指南 (v1.1)

本项目使用 `APScheduler` 作为任务调度引擎，结合 MongoDB `task_execution_logs` 集合实现任务执行过程的持久化记录与监控。

## 1. 核心架构

- **调度引擎**: `APScheduler` (AsyncIOScheduler)
- **存储与执行**: `MemoryJobStore` (任务存储), `AsyncIOExecutor` (异步执行器)
- **执行日志**: `task_execution_logs` 集合 (持久化每一步执行结果)
- **调度配置**: `webapp/scheduler/core.py` (中央配置与注册)

## 2. 核心组件说明

### 2.1 调度器初始化 (`webapp/scheduler/core.py`)
负责调度器的实例化。默认配置：
- `coalesce: True`：合并错过触发的任务。
- `max_instances: 1`：确保同一任务同一时刻只有一个实例在运行。
- `misfire_grace_time: 60`：错过触发 60 秒内仍可启动。

### 2.2 任务日志记录器 (`webapp/scheduler/logger.py`)
提供 `TaskLogger` 静态类，用于标准化的生命周期记录：
- `log_task_start`: 生成全系统唯一的 `task_id` 并初始化 running 状态。
- `log_task_end`: 记录结束时间、耗时、摘要信息及错误堆栈。

## 3. 开发新任务流程

### 第一步：定义任务逻辑与执行策略
在 `webapp/scheduler/jobs/` 下创建任务文件。必须考虑**幂等性**，即一天内只应成功执行一次。

```python
from webapp.scheduler.logger import TaskLogger
from webapp.tools.mongo import DATABASE, get_config

# 推荐实现：内存缓存 + 数据库校验
_daily_cache = {} 

async def my_event_driven_job():
    # 1. 窗口检查：不到 6 点不空转
    start_hour = int(get_config("SCHEDULE", "run_times", "06:00").split(':')[0])
    if datetime.now().hour < start_hour: return

    # 2. 幂等性检查
    if _is_executed_today("my_task", today): return
    
    # 3. 前置事件检查 (例如等待 RPA 成功标记)
    if not DATABASE["task_execution_records"].find_one({"pipeline_name": "...", "status": "SUCCESS"}): return

    # 4. 开始记录
    task_id = await TaskLogger.log_task_start(...)
    try:
        # 执行业务...
        await TaskLogger.log_task_end(task_id, status="SUCCESS", summary="完成")
        _mark_executed_in_cache("my_task", today, "SUCCESS")
    except Exception as e:
        await TaskLogger.log_task_end(task_id, status="FAILED", error={"msg": str(e)})
        _create_alert(level="P1", ...) # 创建系统告警
```

### 第二步：在 `core.py` 中注册任务
```python
def setup_scheduler(app):
    # 注册为 Interval 任务定期扫描前置事件
    scheduler.add_job(
        my_event_driven_job,
        'interval',
        minutes=10,		# 扫描频率
        id='web_my_task_id',
        replace_existing=True
    )
```

## 4. 最佳实践建议

### 4.1 任务隔离
- 每个任务应尽量自治，独立负责前置条件检查和异常处理。
- 耗时过长的操作应通过异步调用 Service 层完成，避免阻塞调度器主线程。

### 4.2 告警机制
- 凡是业务失败（如数据缺失、逻辑错误），必须通过 `system_alerts` 集合产生告警，以便在前端看板显示。

### 4.3 缓存清理
- 内存缓存应在日期变化时自动清空。

## 5. 标准任务模板

为确保系统稳定性，所有新开发的定时任务请参考以下标准架构：

```python
# webapp/scheduler/jobs/standard_template.py
import logging
from datetime import datetime
from webapp.tools.mongo import DATABASE, get_config
from webapp.scheduler.logger import TaskLogger

logger = logging.getLogger(__name__)

# 模块级缓存，确保当日不重复执行
_status_cache = {} 

async def generic_event_task():
    today = datetime.now().strftime("%Y-%m-%d")
    
    # 策略 1: 窗口限制 (非运行时间不扫描)
    if datetime.now().hour < 6: return 

    # 策略 2: 内存/DB 幂等检查
    if _is_done(today): return
    
    # 策略 3: 前置依赖检查 (等待上游 RPA 数据)
    if not check_dependency(today): return

    # 策略 4: 标准日志记录
    task_id = await TaskLogger.log_task_start(
        service_type="web", 
        task_type="my_business", 
        task_name="描述性名称", 
        trigger_type="event"
    )
    
    try:
        # --- 核心业务开始 ---
        processed_count = do_actual_work()
        # --- 核心业务结束 ---
        
        await TaskLogger.log_task_end(
            task_id, 
            status="SUCCESS", 
            summary=f"处理完成，共计 {processed_count} 项",
            details={"count": processed_count}
        )
        _status_cache[today] = "SUCCESS"
    except Exception as e:
        await TaskLogger.log_task_end(task_id, status="FAILED", error={"error": str(e)})
        # 重要：创建系统告警
        create_system_alert(title="任务失败", content=str(e))
        _status_cache[today] = "FAILED"

```

## 6. 日志与状态查询
- 日志：`db.task_execution_logs.find({ task_type: "..." }).sort({start_time: -1})`
- 告警：`db.system_alerts.find({ status: "ACTIVE" }).sort({created_at: -1})`
