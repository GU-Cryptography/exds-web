# 客户负荷分析页面 (CustomerLoadAnalysisPage) 代码流程分析

## 1. 总体交互流程

### 1.1 页面初始化
1.  **入口**:
    *   **独立访问**: `/customer-analysis` -> 显示客户选择器 (Autocomplete)。
    *   **Tab嵌入**: `/customer-analysis/:id` -> 隐藏选择器，通过 Props 传入 `customerId`。
2.  **客户加载**:
    *   调用 `GET /api/v1/customers` (customerApi.getCustomers) 获取客户列表（按签约量降序）。
    *   默认选中第一个客户，或根据 URL 参数/Props 选中指定客户。
3.  **数据触发**:
    *   `useEffect` 监听 `selectedCustomer` 或 `selectedDate` 变化。
    *   触发 `fetchData()` 并行请求：
        *   `fetchDailyView`: 获取日内曲线及统计指标。
        *   `fetchHistory`: 获取历史趋势 (默认30天)。
        *   `getCustomer`: 获取客户最新信息 (主要用于 Tags)。

### 1.2 用户交互
*   **切换日期**: 点击左右箭头 -> 更新 `selectedDate` -> 触发 `fetchData`。
*   **全屏切换**: 
    *   使用 `useChartFullscreen` Hook。
    *   全屏模式下，组件重新渲染到 `fixed` 定位容器中，并显示更详细的导航按钮。
*   **AI 诊断**:
    *   点击 "AI 模式识别" 按钮 -> 调用 `POST /ai-diagnose`。
    *   后端分析最近7天数据，返回建议标签。
    *   前端自动刷新 Tag 列表。

## 2. 后端数据流向 (Backend Data Flow)

后端请求由 `webapp/api/v1_customer_analysis.py` 处理，底层依赖 `LoadQueryService`。

### 2.1 获取日内视图 (`GET /{customer_id}/daily-view`)

**处理逻辑**:
1.  **参数接收**: `customer_id`, `date` (YYYY-MM-DD)。
2.  **曲线查询**:
    *   **当日曲线**: 调用 `LoadQueryService.get_daily_curve(id, date)`。
    *   **昨日曲线**: 调用 `LoadQueryService.get_daily_curve(id, date-1)`。
    *   **数据源**:MongoDB `unified_load_curve` 集合。
    *   **融合策略**: 默认为 `MP_PRIORITY` (优先使用 MP 数据，缺失则使用电表数据)。
3.  **计算统计指标 (AnalysisStats)**:
    *   **本年签约量**: 聚合 `retail_contracts`，计算当前年份该客户所有有效合同总电量。
    *   **当年累计用电**: 调用 `LoadQueryService.get_daily_totals` (从年初到今日)。
    *   **本月用电**: 调用 `LoadQueryService.get_daily_totals` (从月初到今日)。
    *   **去年同期**:
        *   计算去年同日、去年同月、去年同期累计。
        *   调用多次 `LoadQueryService` 获取历史数据。
    *   **同比计算**: `(当前 - 去年) / 去年`。
4.  **分时电价规则**:
    *   调用 `get_tou_rule_by_date(date)` 获取分时时段 (尖峰平谷深)。
    *   映射到 48/96 点曲线中。
5.  **响应**: 返回 `DailyViewResponse` (包含曲线点、各类统计值、同比数据)。

### 2.2 获取历史趋势 (`GET /{customer_id}/history`)

**处理逻辑**:
1.  **模式判断**: `type=daily` 或 `monthly`。
2.  **Daily 模式**:
    *   计算前30天日期范围。
    *   调用 `LoadQueryService.get_daily_totals`。
    *   返回每日总电量序列。
3.  **Monthly 模式**:
    *   计算前13个月日期范围。
    *   调用 `LoadQueryService.get_monthly_totals`。
    *   返回每月总电量序列。

### 2.3 AI 智能诊断 (`POST /{customer_id}/ai-diagnose`)

**处理逻辑**:
1.  **获取数据**: 循环调用 `LoadQueryService.get_daily_curve` 获取过去 7 天的完整曲线。
2.  **特征分析**:
    *   **日夜比**: 判断连续生产 vs 日间单班。
    *   **周末比**: 判断双休 vs 单休。
    *   **峰谷差率**: 判断负荷平稳性。
3.  **标签生成**: 根据规则生成 `AutoTag`。
4.  **持久化**: 将自动标签写入 `customer_archives` (Source="AUTO")。

## 3. 关键代码路径

### 前端
*   `CustomerLoadAnalysisPage.tsx`: 主页面，负责状态管理和 UI 布局。
*   `api/customerAnalysis.ts`: API 客户端封装。
*   `hooks/useChartFullscreen.tsx`: 图表全屏逻辑 (复用组件)。
*   `components/customer/TagSelector.tsx`: 标签选择组件。

### 后端
*   `api/v1_customer_analysis.py`: **目前承担了过多的业务逻辑**，包括统计计算、同比逻辑、AI 规则判断。
*   `services/load_query_service.py`: 
    *   `get_daily_curve`: 单日曲线 (带融合)。
    *   `get_daily_totals`: 日电量范围查询。
    *   `get_monthly_totals`: 月电量聚合查询。
    *   `_get_fusion_expression`: 核心数据融合逻辑 (MP vs Meter)。

## 4. 潜在问题与优化点

1.  **N+1 查询问题**: 在计算同比、环比、累计数据时，API 层多次同步调用 Service 方法，每次方法内部可能产生一次 DB 查询。
    *   *优化*: 应在 Service 层封装聚合查询，如 `get_analysis_dashboard_stats`，在一个聚合管道中完成多维度统计。
2.  **逻辑耦合**: `v1_customer_analysis.py` 包含大量业务逻辑。
    *   *优化*: 迁移至 `CustomerAnalysisService`。
3.  **AI 诊断性能**: 循环获取 7 天曲线 (`LoadQueryService.get_daily_curve` 每次查库) 效率较低。
    *   *优化*: 使用 `LoadQueryService.get_curve_series` 批量获取 7 天数据。
