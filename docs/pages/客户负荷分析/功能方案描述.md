# 客户负荷分析功能方案描述

## 1. 模块概述

本模块旨在为电力交易提供客户维度的负荷数据分析，分为两个层级：
1.  **宏观总览 (CustomerOverviewPage)**：面向所有客户的资产组合分析，关注整体签约规模、电量贡献、涨跌幅排行及用电效率（峰谷比）。
2.  **微观洞察 (CustomerLoadAnalysisPage)**：面向单户的深度诊断，关注日内负荷曲线、分时结构、历史趋势及AI模式识别。

## 2. 现有架构分析

### 2.1 客户负荷总览 (CustomerOverviewPage)

#### 前端实现
*   **路径**: `frontend/src/pages/CustomerOverviewPage.tsx`
*   **核心功能**:
    *   **KPI卡片**: 签约客户数、签约规模（MWh）、当前总电量、综合峰谷比。包含同比分析。
    *   **资产透视**:
        *   **电量贡献构成**: 环形图展示 Top 5 客户及其他客户的电量占比。
        *   **涨跌龙虎榜**: 按照电量增减绝对值排序的 Top 5 榜单。
        *   **峰谷比极值榜**: 展示高成本型（高峰谷比）和优质平稳型（低峰谷比）客户。
    *   **客户列表**: 支持分页、搜索、排序（按签约量、实测、同比等）。
*   **交互**: 点击客户可跳转至客户负荷分析详情页（Tab 模式）。
*   **缓存策略**: 使用 `sessionStorage` 缓存筛选条件和列表数据，减少重复加载。

#### 后端实现
*   **路由**: `webapp/api/v1_customer_overview.py`
*   **服务**: `webapp/services/customer_overview_service.py` (`CustomerOverviewService`)
*   **数据流**:
    *   **签约数据**: 聚合 `retail_contracts` 集合，计算指定年月的签约总电量。
    *   **实测数据**: 调用 `LoadQueryService` 查询 `unified_load_curve`，按照 MP 优先策略融合数据。
    *   **同比计算**:
        *   **Monthly模式**: 对比去年同月（MTD逻辑，若为当前月则截断至昨日）。
        *   **YTD模式**: 对比去年全年累计。

### 2.2 客户负荷分析 (CustomerLoadAnalysisPage)

#### 前端实现
*   **路径**: `frontend/src/pages/CustomerLoadAnalysisPage.tsx`
*   **核心功能**:
    *   **日内曲线**: 48点负荷曲线，支持当日、昨日、基准曲线对比。全屏模式支持 (`useChartFullscreen`)。
    *   **统计指标**: 当日总量/分时结构、本月累计/分时结构、本年签约进度。
    *   **AI 诊断**: 基于规则（夜间/日间比、周末/工作日比、波动率）自动生成标签（如"连续生产"、"周末双休"）。
    *   **标签管理**: 手工标签管理 + 自动标签展示。
    *   **历史趋势**: 切换 30天日级视图 / 12个月月级视图。
*   **复用性**: 支持独立页面访问（带选择器）和嵌入式 Tab 访问（通过 `customerId` props）。

#### 后端实现
*   **路由**: `webapp/api/v1_customer_analysis.py`
*   **服务**: 
    *   **现状**: 业务逻辑直接耦合在 Router 层（`get_daily_view`, `trigger_ai_diagnose`），未封装独立的 `CustomerAnalysisService`。
    *   **依赖**: 强依赖 `LoadQueryService` 获取基础负荷数据，依赖 `CustomerService` 管理标签。
*   **关键逻辑**:
    *   **Daily View**: 并行查询当日、昨日曲线，计算大量统计指标（本月、本年、去年同期）。
    *   **AI Diagnose**: 简单的启发式算法遍历最近7天数据。

## 3. 数据模型

| 业务实体 | 对应集合 (MongoDB) | 说明 |
| :--- | :--- | :--- |
| **客户档案** | `customer_archives` | 存储基础信息、标签 (`tags`) |
| **零售合同** | `retail_contracts` | 存储`purchasing_electricity_quantity` (kWh)，用于签约规模统计 |
| **负荷数据** | `unified_load_curve` | 存储日度96点/48点负荷数据 (MWh)，由 `LoadQueryService` 统一访问 |
| **分时规则** | `tou_rules` | 用于将负荷点映射为 尖/峰/平/谷/深 |

## 4. 优化建议与方案

### 4.1 架构重构 (Backend Refactoring)

**问题**: `webapp/api/v1_customer_analysis.py` 代码臃肿，包含大量业务计算逻辑，难以维护且复用性差。

**方案**:
1.  **创建 `webapp/services/customer_analysis_service.py`**:
    *   将 `get_daily_view` 中的统计计算逻辑（本年签约、累计同比、本月同比）迁移至 Service。
    *   将 `trigger_ai_diagnose` 的规则判断逻辑封装为 `diagnose_load_pattern(customer_id, date)` 方法。
    *   将 `get_history_trend` 逻辑封装为 `get_history_trend(customer_id, type)` 方法。
2.  **统一服务调用**:
    *   Router 层只负责参数解析和响应模型转换。

### 4.2 前端组件优化

**问题**: `CustomerLoadAnalysisPage.tsx` 组件过大 (>800行)，包含图表渲染、数据处理、标签逻辑。

**方案**:
1.  **拆分组件**:
    *   `LoadCurveChart`: 封装 Recharts 曲线图及全屏逻辑。
    *   `LoadStatsCard`: 封装右侧/下侧的统计指标卡片。
    *   `TagManager`: 封装 AI 诊断按钮及标签增删逻辑。
2.  **统一 Hooks**:
    *   确保两个页面在"全屏图表"、"分时背景色"等功能上复用 `src/hooks` 中的代码。

### 4.3 性能优化

*   **并行查询**: 两个页面均已使用 `Promise.all` 并行请求，保持该实践。
*   **数据缓存**: 总览页面的 `sessionStorage` 缓存机制有效，建议在数据更新操作（如编辑客户）后增加清除缓存的机制（目前是基于时间过期）。
*   **数据库查询**:
    *   `CustomerOverviewService` 中使用了 `aggregate` 聚合查询，性能尚可。
    *   `get_growth_ranking` 等方法在 Python 层进行了大量循环查询 (`_get_customer_actual_usage` 内部循环调用 `LoadQueryService`)，对于客户数较多（>100）的场景可能存在 N+1 查询性能问题。
    *   **建议**: 在 `LoadQueryService` 中增加批量查询接口 `get_batch_daily_totals(customer_ids, start, end)`，在总览 Service 中一次性获取所有关注客户的负荷数据，在内存中匹配，大幅减少 DB IO 次数。
