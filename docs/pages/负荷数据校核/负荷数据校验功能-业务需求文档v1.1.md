## 1. 项目背景

我们需要为电力市场交易员和客户经理提供一个强大的工具，用于评估和分析用户负荷数据的完整性和准确性。当前，系统存在两种核心的负荷数据来源：

### 1.1 数据源说明

1.  **手工导入的电表示数数据（负荷曲线）**:

    - **来源**: 电力交易中心，通过文件手工导入
    - **用途**: 用于负荷预测、曲线展示、用电特征分析
    - **粒度**: 96点/日（15分钟）
    - **特点**:
      - 数据周期长（可达1年以上）
      - 更新频率较低（每日一次或按需导入）
      - **是获取近期（T-1日）数据的最快途径**
      - 依赖人工操作，数据中间偶尔会有断点，导入时会做插值处理
      - 数据为电能表示数（累计值）

2.  **RPA自动采集的结算数据**:

    - **来源**: 电力交易中心平台，通过RPA程序自动爬取
    - **用途**: 用于交易结算（权威数据）
    - **粒度**: 48点/日（30分钟）
    - **特点**:
      - 权威性高（官方处理，用于最终结算）
      - 自动化获取，包含官方的缺点补录、换表处理等
      - 数据周期短（仅客户签约后）
      - **延迟高（T-2，即今天只能获取前天的数据）**
      - 数据为计量点级别的时段用电量（差值）

### 1.2 核心问题识别

**数据同源性**：
- 手工导入数据和RPA采集数据**本质上来自同一数据源**（电力交易中心）
- 但处理方式不同：
  - RPA数据：官方处理后的权威数据
  - 手工数据：我方自行处理

**核心问题**：
- 手工导入数据中的**电表-计量点分配系数（allocation_percentage）是自行估算的**
- 这些估算的系数可能存在较大误差
- 需要通过与权威的RPA结算数据对比来**验证和优化这些分配系数**

**误差容忍度**：
- ✅ 可忽略：两者因处理方法差异（如缺点补录方法不同）产生的小幅度误差（通常<1%）
- ⚠️ 需修正：分配系数不准确导致的系统性偏差（可能>1%）

### 1.3 数据关系

**说明**：当前的数据关系保存在 `customers` 数据集的 `utility_accounts` 字段中。

**用户-电力户号-电能表-计量点关系**：
```
用户 (Customer)
  └─ 户号 (Utility Account) [1对多]
       └─ 电表 (Meter) [1对多]
            └─ 计量点 (Metering Point) [多对1]
                 └─ 分配系数 (allocation_percentage)
```

**关键约束**：
- 一个电表可以被多个计量点共享
- 每个电表下所有计量点的分配系数总和 ≤ 100%
- 每个计量点的分配系数：0% ≤ allocation_percentage ≤ 100%

**示例**：
```json
{
  "user_name": "江西省xx物资公司",
  "utility_accounts": [
    {
      "account_id": "3600015996813",
      "meters": [
        {
          "meter_id": "3630001492249049029898",
          "metering_points": [
            {
              "metering_point_id": "211474252",
              "allocation_percentage": 65.0
            },
            {
              "metering_point_id": "227324023",
              "allocation_percentage": 25.0
            }
          ]
        }
      ]
    }
  ]
}
// 电表 xxx898 的系数总和：65.0 + 25.0 = 90.0% ✅（≤100%）
```

## 2. 业务目标

1.  **提供手工导入数据接口**：
    - 提供专门的API接口及前端界面，支持管理员批量导入 Excel 格式的电表示数数据。
    - 实现严格的准入控制，自动识别文件中的表号并匹配客户档案，对未建档电表直接拒绝。
    - 在导入过程中自动执行高级数据清洗（如插值修复），确保原始数据的质量。

2.  **用户级负荷数据准确性校验**：
    - 建立自动审计机制，将手工计算出的用户级总负荷与权威的 RPA 结算数据进行实时比对。
    - 验证电表-计量点分配系数的准确性，对误差超过 1% 的情况自动熔断并告警，防止错误数据进入核心库。
    - 确保手工数据在补充到统一曲线时，能够精准还原用户的实际用电情况。

3.  **用户负荷数据完整性分析**：
    - 对 `unified_load_curve` 进行持续监控，自动识别数据断点和缺失日期。
    - 提供可视化的完整性分析报告，区分“权威数据(RPA)”和“手工估算数据”的覆盖比例。
    - 辅助业务人员快速发现需要补录数据的日期，提升基础数据的完备性。

## 3. 核心决策

### 3.1 存储策略

| 决策点 | 方案 | 理由 |
|--------|------|------|
| **存储粒度** | 原始数据保留最细粒度（按日聚合）；标准化数据为用户级 | 平衡存储成本、查询性能与数据追溯能力 |
| **原始数据** | **两套独立存储**：<br>1. `raw_meter_data` (**手工源，宽表结构**)<br>2. `raw_mp_data` (**RPA源，宽表结构**) | 采用一致的按日聚合策略，大幅降低存储开销，保留原始数据完整性，并统一元数据结构 |
| **标准化曲线** | **唯一标准化曲线** (`unified_load_curve`) | 消除数据孤岛，下游业务（预测、结算）统一调用接口，降低复杂度 |

**数据链路**：
```
[原始层 - 物理隔离]
手工导入 → raw_meter_data (按日宽表)
    结构: { 
        meter_id, 
        date, 
        readings: [96点示数],
        meta: { customer_name, account_id }
    }
    特点: 纯原始示数，无业务计算

RPA采集 → raw_mp_data (按日宽表)
    结构: { 
        mp_id, 
        date, 
        load_values: [48点电量], 
        meta: { customer_name, account_id } 
    }
    特点: 权威负荷电量 (MWh)

      ↓ (融合生成)

[标准化层 - 逻辑统一]
unified_load_curve (长表结构)
(关键字段: customer_name, datetime, load_value(MWh), source)
```

### 3.2 统一曲线生成机制

系统采用 **"权威覆盖，时效互补"** 的策略，并引入 **"自动审计 + 熔断机制"** 确保数据质量。

#### A. 核心原则
1.  **RPA权威性**：RPA数据（结算数据）一经入库，无条件覆盖同时间点的现有记录。
2.  **准入控制**：手工数据导入时，若电表未在档案中，直接拒绝导入。
3.  **审计熔断**：已签约客户的手工数据需与RPA数据比对，误差 > 1% 则自动标记为“未校核”并停止生成曲线。

#### B. 详细生成逻辑

**场景一：RPA数据同步与处理（由Web端执行）**

*   **前提**：`exds-rpa` 项目已完成数据采集，并将原始数据存入共享数据库的 `raw_mp_data` 集合。
*   **触发机制**：
    1.  **定时任务**：本项目 (`exds-web`) 的定时任务（如每日 02:00）自动扫描待处理的 RPA 数据。
    2.  **人工触发**：管理员在前端点击“同步/重算”按钮，调用 API 接口触发指定日期的处理。

**执行流程**：
1.  **读取与扫描**：
    *   从 `raw_mp_data` 获取指定日期（Date D）的计量点数据。
    *   如果数据不存在，则流程结束。

2.  **客户检索与自动建档**：
    *   根据 `raw_mp_data` 中的 `meta.customer_name` 在 `customers` 集合中检索 `customer_id`。
    *   **若未找到匹配客户**：
        - **自动建档**：在 `customers` 集合中新建一条记录，填写 `user_name`、`account_id`、`metering_point_id` 等基础信息，其余字段留空。（这个需求以后再完善）
        - **触发告警**：在 `system_alert` 集合中增加一条记录（`level: "warning"`），提醒管理员完善新客户档案。（这个需求以后再完善）
    *   **若找到匹配客户**：获取对应 `customer_id`。

3.  **动态时间点识别**：
    *   根据 `raw_mp_data.load_values` 数组长度映射时间点。
    *   **48点模式（默认）**：第一个点为当日 00:30，最后一个点为次日 00:00。
    *   **96点模式**：第一个点为当日 00:15，最后一个点为次日 00:00。

4.  **聚合（直接求和）**：
    *   `User_Load[t] = Σ (MP_Load[t])`
    *   *注：RPA 数据已是计量点级权威电量，直接累加用户下所有 MP 即可，无需分配系数。*

5.  **写入**：将聚合后的记录写入 `unified_load_curve`（长表）。
    *   **标记**：`source: "rpa"`
    *   **动作**：`upsert=True`（强制覆盖）。即使该日期已有手工生成的曲线，也被权威数据替代。


**场景二：手工数据入库触发（重构 - 两阶段策略）**

为确保数据的高保真性与系统的容错能力，手工数据的处理采取 **“原始落库与业务计算解耦”** 的两阶段流水线模式。

#### 阶段一：原始数据落库 (Storage Phase - 准入与去重)
*   **目标**：无条件保留所有格式合法的原始示数凭证，不因业务档案缺失（未建档）而拒收数据。
*   **逻辑流**：
    1.  **身份识别与交叉验证**：
        - 从 Excel 文件名提取起始电表号（Meter ID）。
        - 检查文件内容，确保包含“电能表资产号”列，且文件名与内容中的表号完全一致。
    2.  **格式兼容与重塑 (Reshaping)**：
        - 自动识别 96 点或 1440 点等格式，仅提取分钟数为 15 倍数（00, 15, 30, 45）的时间列。
        - 将“日期 + 时刻列”的宽表结构转换为“表号 + 日期时间 + 示数”的长表结构。
    3.  **无条件增量入库**：
        - **准入规则**：只要格式正确，即刻落库。**不在此阶段**强制校验该电表是否已在 `customers` 档案中。
        - **去重逻辑**：基于 `(Meter_ID, DateTime)` 组合键，仅插入数据库中缺失的新记录，支持断点补录。
    4.  **落库终点**：数据存入 `raw_meter_data`（原始示数集合）。

#### 阶段二：聚合生成业务 (Computation Phase - 计算与审计)
*   **目标**：基于落库后的原始示数，通过档案绑定与业务计算，生成可用于结算和预测的统一负荷曲线。
*   **触发**：阶段一完成后自动触发（受影响的 Customer 列表），或通过定时任务/手动按钮触发。
*   **逻辑流（按用户+日期分组执行）**：
    1.  **档案匹配与参数绑定 (Binding)**：
        - 检索 `customers` 档案。如果该客户当日所有电表的数据都在`raw_meter_data`（原始示数集合）中，则可以继续执行后续步骤，否则记录告警。
        - 获取匹配客户的倍率 `M` 及分配系数 `Ratio`。
        - **校验判定**：若该用户下所有关联电表的 `Ratio` 均**不为空**，则视为“已通过校核”。
    2.  **高级数据清洗与插值 (Advanced Preprocessing)**：
        - **时机**：在计算差值之前，读取 `readings` 数组后立即执行。
        - **策略一：历史廓形填充（优先）**：若存在缺口，尝试读取该表 **前一日（T-1）** 的波动廓形，计算缩放因子 `k = Delta_Current / Delta_Ref` 进行特征还原补全。
        - **策略二：线性插值（兜底）**：若无 T-1 参考，采用标准线性插值补齐。
        - **回落处理**：若 `Reading[t] < Reading[t-1]`，判断是否为翻转（如 9999->0001），否则标记为脏数据并中断。
    3.  **差分计算与频度对齐**：
        - 计算 `Load_96[t] = (Reading[t] - Reading[t-1]) * M`。
        - 执行 **96转48对齐**：`Load_48[i] = Load_96[2i] + Load_96[2i+1]`。
    4.  **用户级聚合**：
        - **计算用系数**：若 `Ratio` 为空（null），聚合计算时**临时按 1.0 (100%) 处理**，后续在误差校验时进行更新。
        - `User_Load_Total[t] = Σ (Load_48_Meter_i[t] × Ratio_i) / 1000` (单位转为 **MWh**)。
    5.  **审计校验与分流 (Audit & Routing)**：
        - **比对 RPA**：若当日已有 RPA 真值，计算总电量偏差。
            - **偏差 ≤ 1%**：校验通过。若原 `Ratio` 为空，系统**自动更新档案，将系数设为 1.0**（或用户确认值），使其转为“已校核”状态。
            - **偏差 > 1%**：校验失败。系统**保持（或重置）系数为空**，并发出校核告警。
        - **分流落库**：
            - **情况 A (未签约客户)**：写入 `temporary_load_curve`（无论系数是否为空）。
            - **情况 B (已签约且系数不为空)**：写入 `unified_load_curve` 。
           

### 3.3 校核策略

#### A. “校核状态”判定逻辑
系统不再设立独立的 Boolean 状态位，而是基于档案中数据的完备性进行隐式判定：
*   **判定规则**：若该用户档案中所有关联电表的 `allocation_ratio` 均**不为空 (Not Null)**，则程序逻辑上视该用户为 **“已校核”** 状态。
*   **状态流转**：
    *   **初始/新建**：`allocation_ratio` 默认为 **`null`**（未校核）。
    *   **自动确认**：手工数据试算偏差 $\le 1\%$ 时，系统自动将试算比例（默认为 1.0）存入该字段，状态自动转为“已校核”。
    *   **自动熔断**：若偏差 $> 1\%$，系统保持（或强制重置）该字段为 **`null`**。
    *   **人工干预**：管理员在“数据校核”页面手动输入系数并保存后，字段由空变非空，状态转为“已校核”。
    *   **签约变更**：当用户从“未签约”转为“已签约”时，系统**清空 (Reset to Null)** 所有分配系数，强制触发 RPA 对账。

#### B. 校验规则
1.  **强制校验对象**：所有“已签约”且系数为空的用户。
2.  **豁免对象**：“未签约”用户（无论系数是否为空均允许生成临时曲线）。

#### C. 完整性分析
（保持不变）

## 4. 核心业务逻辑

### 4.1 数据处理与存储

#### A. 原始数据层 (Raw Layer)
*   **`raw_meter_data` (手工示数数据)**：
    - **内容**：存储经格式转换（宽转长）后的电表原始示数。
    - **处理逻辑**：仅执行身份校验、时间点对齐（15分钟粒度）和增量去重存储。**保持数据的高保真性**，不在此层应用倍率或插值。
    - **特点**：作为系统的“原始凭证库”，支持审计追溯。
*   **`raw_mp_data` (RPA 负荷数据)**：
    - **内容**：存储经官方 RPA 采集的计量点级别时段电量（差值）。
    - **特点**：权威真值，单位统一为 MWh。

#### B. 客户档案层 (Meta Layer)
*   **`customers`**：
    - **关键参数**：包含电表倍率 `multiplier` 和电表-计量点分配系数 `allocation_ratio`。
    - **隐式校验判定**：以 `allocation_ratio` 是否为 **`null`** 作为该电表是否通过校验的唯一判据。

#### C. 标准化数据层 (Unified Layer)
*   **`unified_load_curve` (统一标准化曲线)**：
    - **生成逻辑（RPA 触发）**：
        1. 自动聚合该用户下所有计量点数据。
        2. 以 `source='rpa'` 强制覆盖写入，拥有最高优先级。
    - **生成逻辑（手工计算触发）**：
        1. **档案匹配**：从原始层提取示数并绑定档案参数。
        2. **廓形还原**：应用历史廓形插值等高级算法清洗漏点。
        3. **转换计算**：执行差分、乘倍率、96转48对齐及加权聚合（系数为空时临时按 1.0 计算）。
        4. **审计分流**：偏差通过且为已签约客户时，以 `source='manual'` 写入。

#### D. 临时数据层 (Temporary Layer)
*   **`temporary_load_curve` (临时负荷曲线)**：
    - **用途**：存储未签约潜在客户的手工导入数据。
    - **结构**：与 `unified_load_curve` 完全一致。
    - **目的**：物理隔离非权威数据，仅供开发分析和用电特征研究使用，确保核心业务库的纯净。

### 4.2 数据校核逻辑

#### A. 流程
1.  **选择样本**：用户选择一个历史日期（该日期需同时拥有手工原始数据和RPA原始数据）。
2.  **控制变量**：
    - 取出 `raw_mp_data` (RPA) 作为**真值**。
    - 取出 `raw_meter_data` (手工) 结合**当前倍率和分配系数**实时计算出的拟合值。
3.  **对比分析**：
    - 计算两者的总电量偏差和曲线形态偏差。
    - 如果偏差过大，提示调整分配系数。

### 4.3 应用层调用逻辑与历史补全

由于底层已经做了统一融合，上层业务逻辑将变得非常简单且具备高度的数据完整性。

#### A. 历史补全机制 (Data Backfilling)
*   **核心逻辑**：手工数据的最大业务价值在于**追溯与补充历史空缺**。
*   **自动触发**：系统在进行聚合计算时，应对比 `unified_load_curve` 的现有记录。
*   **生成规则**：对于 `unified_load_curve` 中**未包括的所有历史时间点**，只要对应的 `raw_meter_data` 集合中存在有效的原始示数，系统均应自动按照当前的分配参数进行计算，并将结果插入 `unified_load_curve`（标记 `source='manual'`）。
*   *目的：确保用户能获得一条时间跨度尽可能长、且连续无断点的完整负荷曲线。*

#### B. 业务模块调用
- **负荷预测模块**：
    - **逻辑**：直接查询 `unified_load_curve`。
    - **说明**：
        - 如果查到的是 `manual` 数据（如 T-1 或历史补录点），说明是最新录入或后补的，直接用。
        - 如果查到的是 `rpa` 数据（如 T-2 及更早），说明是权威的，更可以用。
        - **无需**再做“数据来源借用”的判断逻辑。

- **预结算/电量分析模块**：
    - **逻辑**：直接查询 `unified_load_curve`。
    - **视觉区分**：前端展示时，根据 `source` 字段给予视觉提示（如实线代表权威 RPA，虚线代表手工补录），供业务人员参考。

## 5. 特殊场景处理

### 5.1 换表事件

**场景**：电表更换（发生概率极小）

**处理方案**：数据库手工修正

#### 步骤1：更新客户档案
```javascript
// MongoDB Shell
db.customers.updateOne(
  { "_id": ObjectId("customer_id") },
  {
    "$set": {
      "utility_accounts.0.metering_points.0.meter.meter_id": "新电表ID",
      "updated_at": new Date(),
      "updated_by": "admin",
      "update_reason": "2025-10-16换表"
    }
  }
)
```

#### 步骤2：标记历史数据（可选）
```javascript
// 在 raw_meter_data 中标记换表前的数据
db.raw_meter_data.updateMany(
  {
    "meter_id": "旧电表ID",
    "date": { "$lte": "2025-10-15" }
  },
  {
    "$set": {
      "meter_replaced": true,
      "replaced_date": "2025-10-16"
    }
  }
)
```

#### 步骤3：校核建议
- 换表前数据：使用旧的分配系数
- 换表后数据：使用新的分配系数（需要重新校核优化）
- 不建议跨换表日期进行校核

## 6. 数据质量标准

### 6.1 完整度等级

| 等级 | 完整度范围 | 状态 | 建议操作 |
|------|-----------|------|---------|
| 健康 | ≥ 90% | 🟢 绿色 | 保持现状 |
| 警告 | 70% ~ 89% | 🟡 黄色 | 建议补充数据 |
| 严重 | < 70% | 🔴 红色 | 必须补充数据 |

### 6.2 偏差容忍度

| 偏差率 | 评估 | 建议操作 |
|--------|------|---------|
| < 5% | 正常 | 无需调整系数 |
| 5% ~ 10% | 关注 | 建议校核分析 |
| > 10% | 异常 | 必须优化系数 |
