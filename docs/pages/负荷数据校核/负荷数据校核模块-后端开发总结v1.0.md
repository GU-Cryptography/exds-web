# 负荷数据校核模块 - 后端开发总结 v1.0

## 1. 模块概述
本模块旨在提供电力负荷数据的采集、清洗、聚合及校核功能。核心目标是将RPA采集的计量点数据（MP Data）与手工导入的电表示数数据（Meter Data）进行对比，验证数据的准确性，并生成统一的高质量负荷曲线。

## 2. 核心服务逻辑

### 2.1 数据导入服务
负责将 Excel 原始数据解析并存入 Raw 层数据库。

#### 2.1.1 电表数据导入 (`MeterDataImportService`)
*   **支持格式**: Excel (.xlsx/.xls)，包含96点（15分钟间隔）或1440点等密集数据。
*   **身份识别**: 优先从**文件名**中提取电表资产号（Regex: `^(\d{15,25})`），若文件名无匹配则尝试从文件内容第一列提取。
*   **数据清洗**:提取 00:00, 00:15 ... 23:45 等96个时间点的示数。
*   **存储**: 存入 `raw_meter_data` 集合，按 `(meter_id, date)` 唯一索引去重（Upsert）。

#### 2.1.2 计量点数据导入 (`MpDataImportService`)
*   **格式识别**: 兼容 `.xls` 和 `.xlsx`，模糊匹配"计量点ID"、"日期"等关键列。
*   **多时段自适应**: 自动识别 24、48 或 96 个时段列，**保持原始点数存储**（不进行自动转换），仅限制最大不超过 96 点。
*   **数据清洗**: 空值/非数字自动置零；缺失"日电量合计"时自动通过分时累加计算。
*   **元数据关联**: 导入时尝试提取`customer_name`和`account_id`，辅助后续档案匹配。
*   **存储策略**: `raw_mp_data` 集合，按 `(mp_id, date)` 唯一键覆盖更新（Upsert）。

### 2.2 负荷聚合服务 (`LoadAggregationService`)
核心业务逻辑所在，负责将 Raw 层数据转换为统一的 48 点负荷曲线 (`unified_load_curve`)。

#### 2.2.1 计量点数据聚合 (`aggregate_mp_load`)
*   **输入**: 客户ID、日期。
*   **逻辑**:
    1.  从档案获取客户名下所有计量点。
    2.  查询 `raw_mp_data`，计算**覆盖率**（实际采集点数 / 应采点数）。
    3.  直接累加各计量点的 48 点负荷值。
*   **输出**: 48点电量数组、覆盖率、缺失计量点列表。

#### 2.2.2 电表数据聚合 (`aggregate_meter_load`)
*   **输入**: 客户ID、日期。
*   **逻辑流程**:
    1.  **完整性强校验**: 检查该客户名下所有电表数据。**任意表缺失则当日聚合失败**。
    2.  **高级清洗与插值**:
        *   **异常回落**: 示数倒走（`curr < prev`）时标记脏数据并置零。
        *   **智能补缺**: 缺口≤3点（线性插值）；缺口>3点（优先尝试前一日历史廓形填充，失败则线性插值）。
    3.  **差分与倍率**: `Load[t] = (Reading[t] - Reading[t-1]) * Multiplier`。
    4.  **归一化 (96转48)**: 自动将 96 点数据按 `Load_48[i] = Load_96[2i] + Load_96[2i+1]` 累加对齐。
    5.  **加权聚合**: 应用分配系数，`Sum(Load_Meter * Ratio) / 1000` (kWh -> MWh)。
*   **输出**: 48点电量数组（MWh）、数据质量标记（插值点、脏数据点）。

#### 2.2.3 统一曲线生成与误差计算
*   **存储结构**: `unified_load_curve` 文档包含 `mp_load` (计量点聚合结果) 和 `meter_load` (电表聚合结果)。
*   **误差计算**:
    *   `daily_error_abs = mp_total - meter_total`
    *   `daily_error_pct = (mp_total - meter_total) / meter_total`
    *   **预警**: 若误差超过 5% (默认阈值)，标记 `is_warning=True`。

### 2.3 负荷查询服务 (`LoadQueryService`)
独立于 API 层的公共服务，供 API、预测、结算等模块复用查询逻辑。

*   **位置**: `webapp/services/load_query_service.py`
*   **依赖**: `ContractService` (用于查询签约客户)
*   **模型**: `webapp/models/load_enums.py` (包含 `FusionStrategy`)
*   **核心方法**:
    *   `get_customer_curve`: 获取单客户融合曲线（包含融合策略计算）。
    *   `get_batch_customer_curves`: 批量获取客户曲线/日电量。
    *   `get_total_load_curve`: 获取全场总负荷曲线。

#### 2.3.1 融合策略逻辑
`LoadQueryService._apply_fusion_strategy` 根据参数 `strategy` (默认 `mp_priority`) 和 `threshold` (默认 0.95) 动态决定每日使用的最终数据源。

| 策略 (`FusionStrategy`) | 逻辑描述 | 适用场景 |
| :--- | :--- | :--- |
| `mp_priority` (默认) | 1. 若计量点数据覆盖率 ≥ 阈值，优先使用 MP。<br>2. 否则，若有电表数据，使用 Meter。<br>3. 否则，使用 MP（即使覆盖率低）。 | 通用校核场景，RPA 数据质量好时优先采用。 |
| `meter_priority` | 若有电表数据则优先使用；否则使用 MP 数据。 | 信任手工数据，仅用 RPA 补缺。 |
| `mp_only` | 仅使用计量点数据，忽略电表数据。 | 仅查看 RPA 采集结果。 |
| `meter_only` | 仅使用电表数据，忽略计量点数据。 | 仅查看手工导入结果。 |

**返回值**: `{"values": [...], "total": 123.4, "source": "mp/meter/none"}`

## 3. 数据库设计

| 集合名称 | 用途 | 关键字段 | 索引 |
| :--- | :--- | :--- | :--- |
| `raw_mp_data` | 原始计量点电量 | `mp_id`, `date`, `load_values`(48点) | `(mp_id, date)` (Unique) |
| `raw_meter_data` | 原始电表示数 | `meter_id`, `date`, `readings`(96点) | `(meter_id, date)` (Unique) |
| `unified_load_curve` | 融合后的校核数据 | `customer_id`, `date`, `mp_load`, `meter_load`, `deviation` | `(customer_id, date)` (Unique) |
| `retail_contracts` | 零售合同信息 | `customer_id`, `purchase_start_month`, `purchase_end_month` | `customer_id` |

## 4. API 接口清单 (`v1_load_data.py`)

### 4.1 核心业务接口
*   `GET /load-data/summary`: 获取统计概览（异常数、待对比数等）。
*   `GET /load-data/customers`: 获取客户列表，支持分页、状态筛选、模糊搜索。
*   `GET /load-data/customers/{id}`: 获取单客户详细档案及概览。
*   `GET /load-data/customers/{id}/calendar`: 获取热力图日历数据。
*   `GET /load-data/customers/{id}/curves`: 获取曲线对比数据（支持日期范围或单日详情）。

### 4.2 数据操作接口
*   `POST /load-data/import/meter`: 上传并导入电表 Excel。
*   `POST /load-data/import/mp`: 上传并导入计量点 Excel。
*   `POST /load-data/reaggregate`: 触发重新聚合任务（支持按时间/客户范围重算）。

### 4.3 曲线查询接口
### 4.3 曲线查询接口
*   `GET /load-data/curve/{customer_id}`: 获取单客户最终融合曲线（调用 `LoadQueryService`）。
*   `GET /load-data/total-load`: 获取全场总负荷曲线（调用 `LoadQueryService`）。

## 5. 待优化项
1.  **分页性能**: `/customers` 接口目前采用全量聚合后内存分页，数据量大时会有性能瓶颈。建议后续改为基于 `unified_load_curve` 的查询分页或增加时间范围过滤（如按月查询）。
2.  **异步处理**: 大批量导入或重新聚合时，建议引入任务队列（如 Celery）进行异步处理。
