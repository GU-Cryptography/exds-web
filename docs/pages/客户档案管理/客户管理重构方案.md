# 客户管理重构技术方案

## 1. 概述
本方案旨在落实“客户管理重构需求分析”中的各项要求，主要包括：
1.  **功能定位调整**：客户档案回归基础信息维护，零售合同改为导入驱动。
2.  **菜单结构优化**：将档案与合同移至“基础数据”，原“客户管理”升级为“客户分析”。
3.  **核心流程重组**：移除复杂的“状态流转”逻辑，建立以“Excel导入”为核心的数据初始化流程。

---

## 2. 数据库与模型变更 (Backend Models)

### 2.1 客户模型 (`webapp/models/customer.py`)
-   **移除字段**：移除 `status` 字段（或保留作为兼容，但不再维护复杂状态流转）。移除任何与 `Contract` 的强耦合字段。
-   **目标结构**：纯粹的客户基础信息（名称、户号、地址、联系人等）。

### 2.2 零售合同模型 (`webapp/models/contract.py`)
-   **新增/确认字段**：确保支持存储从交易中心导入的订单核心字段。
-   **关联逻辑**：保持 `customer_id` 和 `package_id` 的外键引用。

---

## 3. 后端业务逻辑变更 (Backend Services & APIs)

### 3.1 客户服务 (`webapp/services/customer_service.py`)
-   **简化逻辑**：移除 `sign_contract` (签署), `activate` (激活), `terminate` (终止) 等状态变更方法。
-   **CRUD优化**：仅保留标准的增删改查逻辑。
-   **辅助方法**：新增 `get_or_create_by_name(name)` 方法，供合同导入时调用。

### 3.2 合同服务 (`webapp/services/contract_service.py`)
-   **新增导入逻辑**：核心重点。实现 `import_from_excel(file)` 方法。
    -   **解析**: 读取Excel文件。
    -   **处理行数据**:
        1.  根据“客户名称”查找客户。若不存在 -> 调用 `CustomerService.create` 创建一条仅含名称的待完善记录。
        2.  根据“套餐名称”查找零售套餐。若不存在 -> 调用 `PackageService.create` 创建一条仅含名称的待完善记录。
        3.  创建 `RetailContract` 记录，关联上述 ID。
-   **移除/禁用接口**：禁用单条合同的手动创建 (`create`) 接口，或将其限制为仅供管理员调试使用。

### 3.3 API 接口层 (`webapp/api/v1`)
-   **`v1/customers.py`**: 移除状态流转相关的 Endpoint (`/sign-contract`, `/activate` 等)。
-   **`v1/retail_contracts.py`**:
    -   新增 `POST /import`: 接收 Excel 文件，返回导入结果（成功数、失败数、新建客户数、新建套餐数）。
    -   移除/隐藏 `POST /` (新建), `PUT /{id}` (修改), `DELETE /{id}` (删除) 接口，除非特殊权限需要。

---

## 4. 前端变更 (Frontend)

### 4.1 菜单与路由 (`Sidebar.tsx` & `routes.tsx`)
-   **菜单重组**：
    -   **基础数据**:
        -   新增：客户档案管理 (原 `/customer/profiles` -> 建议路径 `/basic-data/customer-profiles`)
        -   新增：零售合同管理 (原 `/customer/retail-contracts` -> 建议路径 `/basic-data/retail-contracts`)
    -   **客户分析** (原“客户管理”):
        -   保留：客户负荷分析
        -   保留：客户聚类分析
        -   新增（占位）：客户面板、客户360详情、临时客户分析。

### 4.2 页面改造

#### 客户档案管理页 (`CustomerManagementPage.tsx`)
-   **表格列**：移除“状态”列。
-   **表单**：移除“关联套餐”、“状态”等字段。
-   **提示**：对于由导入自动创建的客户，增加视觉提示（如“待完善”标记），引导用户补全信息。

#### 零售合同管理页 (`RetailContractPage.tsx`)
-   **操作栏**：**移除**“新建”、“编辑”、“删除”按钮。
-   **新增功能**：添加“导入交易中心订单”按钮。
    -   点击弹出上传对话框。
    -   上传后显示导入报告（如：导入成功100条，自动创建新客户5个）。
-   **展示优化**：增加筛选功能，方便查看导入批次或数据来源。

---

## 5. 实施步骤

1.  **后端重构**：
    -   修改 Model 和 Service，优先实现导入逻辑。
    -   清理旧的状态流转代码。
2.  **API 联调**：
    -   使用 Swagger 测试导入接口。
3.  **前端调整**：
    -   调整菜单和路由结构。
    -   改造客户管理页。
    -   改造零售合同页，接入导入 API。
4.  **数据迁移/清洗** (如果涉及生产环境)：
    -   需要脚本处理现有的 dirty data。
