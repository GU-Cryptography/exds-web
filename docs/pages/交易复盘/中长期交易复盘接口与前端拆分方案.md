# 中长期交易复盘接口与前端拆分方案

## 1. 文档目的

本文档用于在《中长期交易复盘页面设计与实施计划》的基础上，进一步明确：

1. 后端接口如何定义。
2. 接口返回结构如何组织。
3. 前端页面如何拆分为可复用组件。
4. 开发顺序如何安排，便于前后端并行。

## 2. 后端接口设计

## 2.1. 接口清单

建议提供以下 4 个接口：

1. `GET /api/v1/trade-review/trade-dates`
2. `GET /api/v1/trade-review/overview`
3. `GET /api/v1/trade-review/detail`
4. `GET /api/v1/trade-review/batch-detail`

其中：

1. `trade-dates` 用于交易日期选择器。
2. `overview` 用于第一层面板和目标日期导航。
3. `detail` 用于页面主体内容。
4. `batch-detail` 用于按需加载单批次明细，避免 `detail` 过重。

## 2.2. 接口一：获取交易日期列表

### 2.2.1. 请求

`GET /api/v1/trade-review/trade-dates`

### 2.2.2. 用途

用于初始化交易日期选择器，并确定默认交易日。

### 2.2.3. 响应建议

```json
{
  "latest_trade_date": "2026-03-12",
  "trade_dates": [
    "2026-03-12",
    "2026-03-11",
    "2026-03-10"
  ]
}
```

### 2.2.4. 说明

1. 日期按倒序返回。
2. `latest_trade_date` 便于前端初始化默认值。

## 2.3. 接口二：获取交易日概览

### 2.3.1. 请求

`GET /api/v1/trade-review/overview?trade_date=2026-03-12`

### 2.3.2. 用途

用于在选中交易日后，返回该交易日包含的目标日期，以及每个目标日期的轻量级摘要。

### 2.3.3. 响应建议

```json
{
  "trade_date": "2026-03-12",
  "delivery_summaries": [
    {
      "delivery_date": "2026-03-14",
      "record_count": 96
    },
    {
      "delivery_date": "2026-03-15",
      "record_count": 64
    }
  ]
}
```

### 2.3.4. 说明

1. 第一层右侧直接展示 `delivery_summaries`，无需单独返回 `delivery_dates`。
2. 每个目标日期应附带申报记录数，供 UI 在目标日期后展示数字。

## 2.4. 接口三：获取目标日复盘详情

### 2.4.1. 请求

`GET /api/v1/trade-review/detail?trade_date=2026-03-12&delivery_date=2026-03-14`

### 2.4.2. 用途

用于页面主内容加载，包括：

1. 顶部 4 个卡片
2. 第三层成交结果复盘图形数据
3. 第三层成交结果复盘表格数据
4. 批次行为时间轴
5. 默认批次详情
6. 文本复盘结论

1. 年度分月电量
2. 月度电量
3. 机制电量
4. 历史月内净持仓修正量

6. 最后持仓
7. 实际电量或预测电量
8. 成交均价
9. 成交次数
10. 累计成交量
11. 月内交易目标日均价
12. 现货价格

#### `execution_table`

用于第三层 `数据表格` Tab，字段口径应与 `execution_chart` 完全一致。

#### `batch_timeline`

用于第四层左侧批次行为时间轴，数据量可控，适合直接在 `detail` 接口中返回。

#### `default_batch_detail`

用于页面首次渲染时直接展示第一个批次详情，减少一次额外请求。

## 2.5. 接口四：获取单批次详情

### 2.5.1. 请求

`GET /api/v1/trade-review/batch-detail?trade_date=2026-03-12&delivery_date=2026-03-14&batch_id=listing_001`

### 2.5.2. 用途

用于点击批次后按需加载详情，减少 `detail` 的返回体积。

### 2.5.3. 响应建议

```json
{
  "batch_id": "listing_001",
  "batch_action_type": "listing",
  "batch_start_time": "2026-03-12 09:15:08",
  "batch_end_time": "2026-03-12 09:15:25",
  "batch_chart_rows": [],
  "batch_records": []
}
```

### 2.5.4. 说明

若首版数据量不大，可以先不拆此接口，后续再做性能优化。

## 3. 后端模型建议

## 3.1. 路由文件建议

新增文件：

- `webapp/api/v1_trade_review.py`

并在主路由中注册。

## 3.2. Service 文件建议

新增文件：

- `webapp/services/trade_review_service.py`

建议职责拆分如下：

1. `get_trade_dates()`
2. `get_trade_overview(trade_date: str)`
3. `get_trade_detail(trade_date: str, delivery_date: str)`
4. `get_batch_detail(trade_date: str, delivery_date: str, batch_id: str)`

## 3.3. Pydantic 模型建议

新增文件：

- `webapp/models/trade_review.py`

建议定义以下模型：

1. `TradeDateListResponse`
2. `DeliveryDateSummary`
3. `TradeOverviewResponse`
4. `SummaryCardsResponse`
5. `ExecutionChartRow`
6. `ExecutionTableRow`
7. `BatchTimelineItem`
8. `BatchChartRow`
9. `BatchRecordItem`
10. `TradeDetailResponse`

## 3.4. 建议的 Service 内部处理步骤

### 3.4.1. 读取原始记录

从 `trade_declare` 中根据 `trade_date` 找到聚合文档，再从对应 `delivery_date` 的 `delivery_groups` 中取出记录。

### 3.4.2. 衍生字段计算

对每条记录计算：

1. `trade_direction`
2. `traded_mwh`
3. `is_traded`
4. `record_result`
5. `holding_seconds`

### 3.4.3. 批次归类

按 5 秒规则生成：

1. 上架批次
2. 下架批次

### 3.4.4. 外部数据对齐

按 `delivery_date + period` 对齐：

1. `contracts_aggregated_daily` 的全市场-市场化-月内价格
2. 现货价格
3. 实际用电量
4. 电量预测值
5. 年度分月、月度、机制电量

### 3.4.5. 聚合输出

最终生成：

1. 顶部卡片统计
2. 第三层图形复盘数据
3. 第三层表格数据
4. 批次时间轴数据
5. 默认批次详情
6. 文本摘要

## 4. 前端页面拆分方案

## 4.1. 页面文件建议

新增页面文件：

- `frontend/src/pages/TradeReviewPage.tsx`

页面职责：

1. 维护交易日期、目标日期、选中批次等状态。
2. 拉取 `trade-dates`、`overview`、`detail`、`batch-detail`。
3. 组织页面布局，不承载复杂图表渲染细节。

## 4.2. API 文件建议

- `frontend/src/api/tradeReview.ts`

1. `fetchTradeDates`

4. `fetchBatchDetail`

## 4.6. 建议的数据加载顺序

### 4.6.1. 页面首次进入

1. `fetchTradeDates`
2. 自动选中最近交易日
3. `fetchTradeOverview`
4. 自动选中最早目标日期
5. `fetchTradeDetail`
6. 若返回了 `default_batch_detail`，直接渲染默认批次

### 4.6.2. 切换交易日期

1. 先拉 `overview`
2. 再选中默认目标日期
3. 再拉 `detail`

### 4.6.3. 切换目标日期

1. 仅重拉 `detail`
2. 重置 `selectedBatchId`

### 4.6.4. 点击批次

1. 若 `detail` 中已带默认详情以外的所有批次，则直接本地切换
2. 若未带，则调用 `fetchBatchDetail`

## 5. 图表实现建议

## 5.1. 第三层成交结果复盘图

建议使用：

- `ComposedChart`
- X 轴使用 `1-48` 时段
- 左 Y 轴为价格
- 右 Y 轴为电量

注意事项：

1. 图层包含基础持仓浅色堆叠柱、历史月内净持仓修正柱、当日成交差值柱、最后持仓虚线、成交价格点。
2. 目标日电量背景只显示一种来源：
   - 有真实值显示真实电量
   - 无真实值显示预测电量
3. 交易日前持仓线默认隐藏，通过图例开关手动显示。
4. 成交次数大于 1 时，在成交点旁显示小数字。
5. 必须支持 `useChartFullscreen`。

## 5.2. 第三层数据表格

字段应与图形面板口径完全一致，并支持按 48 时段查看：

1. 基础持仓组成
2. 历史月内净持仓修正量
3. 当前交易日月内净成交量
4. 最后持仓
5. 实际用电量或预测用电量
6. 成交均价、成交次数、累计成交量
7. 参考价格

## 5.3. 批次行为时间轴

建议使用：

- `ScatterChart` 或自定义 `ComposedChart`
- X 轴为时间
- Y 轴为批次序号

首版重点是把“一天内几轮上架、下架行为”表达清楚，不追求复杂视觉效果。

## 5.4. 批次量价图

建议使用：

- `ComposedChart`
- 柱：挂牌电量
- 背景面积图：实际电量或预测电量
- 线：挂牌价格、市场均价、现货价
- 左 Y 轴：价格
- 右 Y 轴：电量

## 6. 开发顺序建议

## 6.1. 后端优先完成

1. `trade-dates`
2. `overview`
3. `detail`

原因：

前端页面依赖这三个接口即可完成首版主流程。

## 6.2. 前端首版开发顺序

1. 页面骨架与路由
2. 筛选条
3. 摘要卡片
4. 成交结果复盘
5. 批次行为分析
6. 批次详情图表 / 表格切换

## 6.3. 联调阶段重点验证

1. `listing_side` 的买入/卖出映射是否正确。
2. `off_shelf_type` 的业务口径是否与页面展示一致。
3. 5 秒批次划分是否符合交易员实际操作习惯。
4. 历史月内净持仓反推结果是否正确。
5. 图形面板与表格面板口径是否一致。

## 7. 最小可交付版本

首版建议只做以下能力：

1. 交易日期切换
2. 目标日期切换
3. 4 个摘要卡片
4. 第三层成交结果复盘图形面板
5. 第三层成交结果复盘数据表格
6. 批次行为时间轴
7. 单批次量价图
8. 单批次明细数据表

以下内容可以放到第二版：

1. 自动生成复盘结论文案优化
2. 更复杂的批次筛选
3. 更丰富的颜色和图例编码
4. 批次详情懒加载优化
