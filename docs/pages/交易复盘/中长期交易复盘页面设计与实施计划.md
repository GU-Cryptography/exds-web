# 中长期交易复盘页面设计与实施计划

## 1. 页面定位

### 1.1. 页面目标

“中长期交易复盘”页面用于复盘某个交易日内，交易员围绕某个目标日开展的月内交易过程，重点回答以下问题：

1. 这一天交易过哪些目标日期，当前查看的是哪一个目标日期。
2. 针对该目标日期，交易员在 48 个时段上做了哪些买入/卖出申报，哪些记录产生了成交。
3. 交易员当天发生了几轮集中挂牌申报、人工下架、自动下架行为，各次申报过程覆盖了哪些时段，操作后我方挂单价格和电量如何分布。
4. 成交价格相对全市场市场化月内均价、现货价格、实际用电量所处位置如何，是否体现出较好的交易执行效果。

### 1.2. 页面设计原则

1. 以“交易过程复盘”为核心，不以申报电量、成交率作为核心 KPI。
2. 先看交易对象和摘要，再看成交结果，最后看申报行为。
3. 保留同一时段多次成交、不同价格成交的细节，不做过度压缩。
4. 移动端优先，遵循现有前端规范，图表必须支持 `useChartFullscreen`。

### 1.3. 菜单位置

- 一级菜单：`交易复盘`
- 二级菜单：`月内交易复盘`
- 建议路由：`/trade-review/monthly-trading-review`

说明：

1. 本页面不属于“风险管理”范畴。
2. 页面定位应归类为“交易行为与结果分析”。

## 2. 数据基础与业务口径

### 2.1. 主要数据来源

1. `trade_declare`
   - 核心来源，提供交易日、目标日、申报记录、上架时间、下架时间、剩余电量、挂牌方向、下架类型等。
2. `contracts_aggregated_daily`
   - 用于读取“全市场-市场化-月内”口径的时段均价或日度参考价格。
3. 现货价格数据
   - 用于展示同目标日对应的现货价格曲线。
4. 实际用电量数据
   - 用于展示交易时段对应的实际电量或实际负荷水平，辅助判断买卖动作是否匹配实际需求。
5. 年度分月、月度、机制电量数据
   - 用于构建第三层图表中的基础持仓。
6. 电量预测值数据
   - 在目标日真实电量尚未产生时，用于替代实际电量作为背景参照。

### 2.2. 原始字段理解

#### 2.2.1. `listing_side`

- 含义：挂牌类型。
- 业务映射：本页面中统一映射为 `买入` / `卖出`。
- 展示规则：前端不直接解释原始文案，建议由后端统一输出 `trade_direction`。

#### 2.2.2. `off_shelf_type`

- `自动下架-成交`
  - 挂单因成交而被系统自动下架。
- `未下架`
  - 到交易结束仍未人工下架。
- `人工下架`
  - 交易员主动撤单。

该字段是复盘交易行为的重要依据，应作为页面主分析维度之一。

### 2.3. 页面使用的派生口径

建议由后端统一输出以下派生字段，避免前端重复计算：

| 字段 | 类型 | 说明 |
|------|------|------|
| `trade_direction` | String | `buy` / `sell` |
| `traded_mwh` | Number | `listing_mwh - remaining_mwh` |
| `is_traded` | Boolean | `traded_mwh > 0` |
| `record_result` | String | `成交自动下架` / `成交未下架` / `人工下架` / `未成交结束` |
| `holding_seconds` | Number / Null | 挂单存续时长 |
| `batch_id` | String | 同一目标日下 5 秒窗口内的批次 ID |
| `batch_action_type` | String | `listing` / `manual_off_shelf` / `auto_off_shelf` |

### 2.4. 申报过程划分规则

由于原始记录中没有存储交易申报 ID，只有上架时间、下架时间（秒级），因此需要根据上下架时间来进行申报过程划分。

以目标日为单位，对 `delivery_groups.records` 进行申报过程识别时，采用以下规则：

1. 挂牌申报基于 `listing_time` 识别，人工下架和自动下架均基于 `off_shelf_time` 识别。
2. 过程划分时间窗口暂按 5 秒处理，忽略 30 秒条件。
3. 一个申报过程内不能同时包含“挂牌申报”“人工下架”“自动下架”三种动作。
4. 一个交易对象（目标日期 + 时段）在同一个过程分组中只能出现一次。
5. 对于没有 `off_shelf_time` 的记录，不参与人工下架或自动下架过程划分。
6. `off_shelf_type=人工下架` 归入人工下架过程，`off_shelf_type=自动下架-成交` 归入自动下架过程。

说明：

1. 首版申报过程划分的核心依据是秒级时间聚类，而不是平台官方批次号。
2. 该层对外展示名称统一为“申报过程复盘”，按钮名称中不再出现“批次”字样。
3. 申报过程复盘的重点是“交易员某次集中动作完成后，我方在各时段上仍保留了哪些有效挂单、价格档位和电量如何分布”。

## 3. 页面总体布局

页面采用四层结构，其中顶部统计控制在一行 4 个卡片内，每个卡片承载多项信息。

### 3.1. 第一层：交易日与目标日导航面板

#### 3.1.1. 布局

- 左侧：交易日期切换区
- 右侧：目标日期选择区，目标日期后附带申报记录数

#### 3.1.2. 交互规则

1. 使用 `DatePicker + 左右箭头按钮` 选择交易日期。
2. 交易日期变化后自动加载数据，无需查询按钮。
3. 加载完成后，从 `delivery_dates` 中带出目标日期列表。
4. 目标日期展示在右侧，建议使用 `Tabs` 或 `Chip` 组。
5. 默认选中最早目标日期，通常为 `D+2`。
6. 加载中禁用日期切换与目标日切换控件。

#### 3.1.3. 展示内容

- 交易日期
- 该交易日包含的目标日期列表
- 当前选中的目标日期
- 各目标日期对应的申报记录数

### 3.2. 第二层：摘要卡片面板

#### 3.2.1. 设计原则

- 最多一行 4 个卡片。
- 每个卡片承载 2 到 3 组强相关指标。
- 不放“申报电量”“成交率”这类不适合本业务的指标。

#### 3.2.2. 推荐卡片方案

**卡片 1：记录概览**

- 总申报记录
- 成交笔数

**卡片 2：成交电量**

- 成交电量
- 买入成交电量 / 卖出成交电量

**卡片 3：成交时段**

- 成交时段数
- 买入成交时段数 / 卖出成交时段数

**卡片 4：申报过程**

- 挂牌申报次数
- 人工下架次数 / 自动下架次数

说明：

1. 顶部卡片控制为一行 4 个，避免信息过载。
2. 所有卡片文案应以“过程”和“行为”导向为主。

### 3.3. 第三层：成交结果复盘面板

该层聚焦“48 时段月内交易结果复盘”，将基础持仓、目标日电量、当日月内成交变化、最终持仓和成交价格放在同一张图中展示。

该面板提供两个 Tab 按键，用于切换不同展示方式：

1. `图形复盘`
2. `数据表格`

说明：

1. 默认进入 `图形复盘`。
2. `数据表格` 用于按 48 时段查看该层图表对应的明细数据。
3. 两个 Tab 共用同一套筛选条件和数据口径。

#### 3.3.1. 主图结构

**图表类型：**

- `ComposedChart`

**坐标轴：**

- X 轴：时段 `1-48`
- 左 Y 轴：价格
- 右 Y 轴：电量

#### 3.3.2. 图层设计

该图按从后到前的顺序叠加以下图层。

**图层 1：目标日电量曲线**

用于展示目标日的电量水平，作为交易复盘的参照背景。

规则：

1. 当真实电量存在时，显示真实电量。
2. 当真实电量不存在时，显示预测电量。
3. 两者不同时显示。

展示建议：

- 使用浅灰色面积图或浅灰色细折线。
- 图例文案动态显示为：
  - `实际用电量`
  - `预测用电量`

**图层 2：原持仓浅色堆叠柱**

用于展示该目标日在当前交易日开始前的基础持仓部分。

基础持仓仅由以下部分组成：

1. 年度分月电量
2. 月度电量
3. 机制电量

展示建议：

- 使用浅色堆叠柱状图。
- 各组成部分用不同浅色色彩分段标注。

**图层 3：历史月内净持仓修正柱**

用于展示在当前交易日前，针对该目标日已经发生的月内交易净影响。

定义：

1. 取所有 `trade_date < 当前交易日` 且 `delivery_date = 当前目标日` 的记录。
2. 仅统计有成交的记录。
3. 按时段聚合：
   - 买入成交记为正
   - 卖出成交记为负

展示建议：

1. 作为独立修正柱显示，不并入基础持仓堆叠柱。
2. 为正时向上显示。
3. 为负时向下显示。
4. 建议使用中性色，与“当日成交差值柱”区别开。

**图层 4：当日月内成交差值柱**

用于展示当前交易日对该目标日持仓造成的净变化。

规则：

1. 差值柱表示“当前交易日该时段月内交易净成交量”。
2. 买入成交向上显示。
3. 卖出成交向下显示。
4. 差值柱不覆盖原持仓柱，建议使用更窄的深色柱叠加显示。

说明：

1. 不采用“卖出直接遮挡原持仓柱”的方案，避免原持仓信息被覆盖。
2. 该层表达的是“当日交易影响量”，不是最后一笔成交量。

**图层 5：交易日前持仓线**

用于展示当前交易日开始前，该目标日在各时段上的实际持仓水平。

定义：

- `交易日前持仓 = 基础持仓 + 历史月内净持仓修正量`

显示规则：

1. 默认隐藏，不在主图中直接显示。
2. 通过图表右上角图例或图标开关手动显示。
3. 显示时建议使用灰色点划线，与“最后持仓虚线”明确区分。

说明：

1. 默认隐藏的原因是避免与“最后持仓虚线”同时出现时产生视觉混乱。
2. 该线用于深度对比“交易前”和“交易后”，不作为默认主阅读路径。

**图层 6：最后持仓虚线**

用于展示当前交易日成交完成后的最终持仓结果。

定义：

- `最后持仓 = 基础持仓 + 历史月内净持仓修正量 + 当前交易日月内交易净成交量`

展示建议：

- 使用一条虚线贯穿 48 个时段。
- 与基础持仓柱、历史修正柱和当日差值柱共同构成“交易前 - 当日变化 - 交易后”的闭环表达。

**图层 7：成交价格点**

用于展示有成交时段的价格信息。

规则：

1. 只在有成交的时段显示价格点。
2. 可不连线，仅显示少量原点。
3. 价格点表示该时段“当前交易日累计成交均价”。

多次成交处理规则：

1. 图上只显示一个价格点。
2. 点旁在成交次数大于 1 时显示一个小数字，例如 `3`。
3. 不在图中直接展开显示全部逐笔价格路径。

**图层 8：复盘参考价格**

用于在复盘场景下展示参考价格，不在交易当天跟踪场景下展示。

建议展示：

1. 月内交易目标日均价
2. 现货价格

说明：

1. 该层属于复盘信息，交易当天页面可不显示。
2. 若目标日均价为单值，建议使用参考线。
3. 若现货价格为 48 时段序列，建议使用折线。

#### 3.3.3. Tooltip 规则

为了避免信息过载，Tooltip 只展示最核心的成交信息：

1. 时段
2. 累计成交量
3. 成交均价
4. 成交次数

说明：

1. 不展示过多逐笔成交细节。
2. 若该时段成交次数大于 1，则通过点旁小数字提示存在多次成交。

#### 3.3.4. 该图的核心表达能力

该图应同时回答以下问题：

1. 该时段基础持仓是多少，持仓由哪些部分构成。
2. 当前交易日前是否已经存在月内交易净影响，方向和大小如何。
3. 当前交易日对该时段做了买入还是卖出，净变化量是多少。
4. 当前交易日结束后，该时段最终持仓落在什么位置。
5. 该时段是否发生过成交，成交均价是多少，是否出现多次成交。
6. 最终持仓相对目标日电量和参考价格处于什么位置。

#### 3.3.5. 数据表格 Tab

`数据表格` Tab 用于将第三层图形面板中的关键数据按 48 时段明细表方式展示，便于精确查看和导出。

建议列如下：

1. 时段
2. 年度分月电量
3. 月度电量
4. 机制电量
5. 历史月内净持仓修正量
6. 当前交易日月内净成交量
7. 最后持仓
8. 实际用电量或预测用电量
9. 成交均价
10. 成交次数
11. 累计成交量
12. 月内交易目标日均价
13. 现货价格

说明：

1. 该表与图形面板使用完全一致的数据口径。
2. 表格用于补充图形中的数值细节，不承担图形分析功能。
3. 移动端应支持横向滚动。

### 3.4. 第四层：申报过程复盘面板

该层聚焦“交易员一天内做了多少轮申报动作，每次动作完成后，我方在各时段仍保留了哪些有效挂单，价格档位和电量如何排布”。

整体布局建议如下：

1. 去掉左侧柱状图或时间轴，整体由左右结构改为上下结构。
2. 顶部使用横向单行滚动按钮带展示当天全部申报过程，按钮超出时横向滚动，不做多行铺开。
3. 按钮带下方先展示“本次申报动作说明区”，再展示 `3.4.2 申报后挂单详情图`。
4. `3.4.2 申报后挂单详情图` 与 `3.4.3 申报后挂单详情表` 通过 Tab 切换显示。
5. 默认显示 `图表` Tab，按需切换到 `数据` Tab。

#### 3.4.1. 顶部：申报过程按钮带

**布局建议：**

- 采用横向单行滚动按钮组，不使用左侧柱状图或时间轴。
- 每个按钮使用两行文案，保持紧凑宽度，避免一天 30 个过程时页面被撑高。
- 当前选中按钮高亮，切换后联动更新动作说明区、详情图和详情表。
- 默认选中当天第一笔申报过程。

**按钮文案：**

- 主文案：
  - `挂牌申报 09:15:08`
  - `人工下架 09:26:31`
  - `自动下架 09:32:05`
- 副文案：
  - 建议采用短格式，如 `8条 | 6时段`
  - 或 `买6 卖2`
- 按钮名称中不出现“批次”字样。

**筛选建议：**

当日过程较多时，可在按钮带上方增加轻量筛选：

- `全部`
- `挂牌申报`
- `人工下架`
- `自动下架`

#### 3.4.2. 中部：申报后挂单详情图

点击某个申报过程后，展示“该次动作完成后，我方在平台上的有效挂单状态”。

该区域与 `3.4.3` 互为 Tab 页面：

1. `图表` Tab：展示申报后挂单详情图
2. `数据` Tab：展示申报后挂单详情表

**图表建议：**

- 图表类型：`ComposedChart`
- X 轴：时段 `1-48`
- 上层：挂单价格散点层
- 下层：挂单电量柱状层
- 左 Y 轴：价格
- 右 Y 轴：电量
- 背景面积图：实际电量或预测电量
- 挂单价格仅显示圆点，不连线
- 参考价格仅保留实时价格，不显示月内均价

**图表用途：**

1. 看出该次申报过程完成后，我方在哪些时段仍保留有效挂单。
2. 看出同一时段内存在几档挂单价格，各档位电量如何分布。
3. 看出买入与卖出挂单是否同时存在，以及实时价格与挂单价格的大致关系。
4. 图表整体样式、tooltip 和图例风格应与“成交结果复盘”图表保持一致，避免页面风格割裂。

电量背景规则：

1. 当实际电量存在时，显示实际电量背景面积图。
2. 当实际电量不存在时，显示预测电量背景面积图。
3. 两者不同时显示。

**多次挂单处理规则：**

1. 图中展示的是“操作后仍有效的挂单状态”，不是该次动作的历史申报流水。
2. 同一时段、同一方向、同一价格的多笔挂单，可合并为一个价格点和一个堆叠柱段。
3. 同一时段、同一方向、不同价格的多笔挂单，采用堆叠柱显示电量，并用同色系深浅区分不同价格档位。
4. 同一时段同时存在买入和卖出挂单时：
   - 买入挂单电量显示在 0 轴上方，使用绿色系堆叠柱
   - 卖出挂单电量显示在 0 轴下方，使用红色系堆叠柱
5. 价格圆点与堆叠柱段使用一一对应的颜色映射，不额外画连线。
6. 同方向档位建议按价格排序后分配颜色深浅，价格越高颜色越深，保证映射稳定。

**Tooltip 规则：**

整体样式与成交分析图保持一致，采用白底悬浮卡片，先显示时段，再展示明细和参考值。

1. 标题区：
   - `时段 16`
   - 可选附带时间标签，如 `时段 16（07:30-08:00）`
2. 挂单明细区：
   - 先买入、后卖出
   - 同方向内按价格从高到低展示
   - 每行显示与图中一致的颜色标识、价格和电量
3. 汇总区：
   - 买入挂单总电量
   - 卖出挂单总电量
   - 净挂单电量
4. 参考信息区：
   - 实时价格
   - 实时价格未发布时显示 `未发布`

**图例规则：**

图例位置、排列方式和成交分析图保持一致，默认放在图表下方横向排列。

1. 主图例默认展示：
   - 买入挂单价格
   - 卖出挂单价格
   - 买入挂单电量
   - 卖出挂单电量
   - 实时价格
2. 档位深浅主要通过 tooltip 承接，图例仅补充说明：
   - `同色系深浅代表不同价格档`

**动作说明区：**

位于按钮带下方、图表上方，采用与成交分析面板提示区一致的静态信息条风格，不使用动画作为主表达方式。

建议固定展示三类内容：

1. 动作类型和时间，如 `挂牌申报 09:15:08`
2. 本次动作影响，如 `新增买入挂单 6 个时段，新增卖出挂单 2 个时段`
3. 操作后状态，如 `操作后保留 8 个时段有效挂单，买入总量 42.00 MWh，卖出总量 18.00 MWh`

#### 3.4.3. 申报后挂单详情表

该表与 `3.4.2` 互为 Tab 页面，用于查看该次申报过程完成后的挂单明细数据。

建议列如下：

- 时段
- 买入/卖出
- 挂单价格
- 挂单电量
- 价格档位序号
- 同方向档位数
- 实时价格
- 本次动作影响类型
- 本次动作影响电量

移动端可退化为卡片列表。

## 4. 页面关键交互流程

### 4.1. 首次进入

1. 默认交易日期为最近一个可用交易日。
2. 自动请求该交易日数据。
3. 自动选择最早目标日期。
4. 默认加载该目标日期对应的摘要、成交结果、申报过程复盘。

### 4.2. 切换交易日期

1. 更新目标日期列表。
2. 若新日期下存在目标日期，则默认选中最早目标日期。
3. 在加载过程中不卸载图表容器，使用覆盖层保留全屏状态。

### 4.3. 切换目标日期

1. 刷新摘要卡片。
2. 刷新成交结果图。
3. 刷新申报过程复盘图。
4. 重置申报过程选中状态，默认选中第一个过程或不选中。

### 4.4. 查看申报过程详情

1. 用户点击顶部按钮带中的某个申报过程。
2. 下方联动展示该次过程的动作说明、申报后挂单详情图和明细表。
3. 若当前图表已全屏，切换申报过程时不得退出全屏。

## 5. 接口设计建议

建议不要让前端直接消费 `trade_declare` 原始文档，而是提供面向页面的聚合接口。

### 5.1. 接口一：获取可选交易日

`GET /api/v1/trade-review/trade-dates`

返回：

- 可选交易日列表
- 最近交易日

### 5.2. 接口二：获取交易日概览

`GET /api/v1/trade-review/overview?trade_date=YYYY-MM-DD`

返回：

- `trade_date`
- `delivery_dates`
- 每个目标日期的摘要信息

目标：

1. 用于第一层面板展示目标日期。
2. 用于快速切换不同目标日时减少重复请求。

### 5.3. 接口三：获取目标日复盘详情

`GET /api/v1/trade-review/detail?trade_date=YYYY-MM-DD&delivery_date=YYYY-MM-DD`

### 5.4. 接口四：获取单个申报过程详情

若详情数据较大，可拆分为：

`GET /api/v1/trade-review/operation-detail?trade_date=...&delivery_date=...&operation_id=...`

建议返回两部分内容：

1. `operation_summary`

## 8. 风险与注意事项

### 8.1. 数据口径风险

1. 当前没有真实“成交时间”字段，成交结果复盘图中的价格点只能基于交易日内聚合口径表达，不能代表精确成交时点。
2. `off_shelf_type=未下架` 的业务含义要进一步确认是否必然对应“交易结束未撤单”。
3. 历史月内净持仓需要通过 `trade_declare` 反推，需提前确认买入/卖出对净持仓的映射规则。
4. 申报过程详情图的“操作后有效挂单状态”需要先在后端明确快照计算口径，避免前后端对状态定义不一致。

### 8.2. 交互复杂度风险

1. 页面图表较多，首次实现时应控制在 2 个核心图 + 1 个申报过程联动图。
2. 同一时段多档挂单会明显提高 tooltip 和堆叠图的复杂度，需要优先保证颜色映射稳定。
3. 申报过程明细若一次返回全部价格档位，可能存在数据量偏大问题，可后续拆接口。

### 8.3. 展示风险

1. 若顶部卡片指标过多，会削弱“交易过程复盘”的主线。
2. 若基础持仓、历史修正、当日成交变化与最终持仓的颜色区分不明显，用户容易误读持仓变化逻辑。
