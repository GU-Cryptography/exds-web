# 负荷特征分析服务重构与优化建议书

## 1. 现状评估 (Current State Assessment)

当前的 `CharacteristicService` 实现是一个的 **MVP (Minimum Viable Product)** 版本。虽然它跑通了从数据查询到标签生成的基础链路，但在面对真实复杂的电力业务场景时，存在以下局限性：

1.  **判定逻辑过于刚性 (Rigid Logic)**：
    *   依赖硬编码阈值（如 `day_ratio > 0.55`）。这种 "一刀切" 的逻辑无法适应不同行业差异。
    *   缺乏弹性和自适应性。

2.  **指标维度单一 (Shallow Metrics)**：
    *   缺乏时序分析（Time Series Analysis）深度，如未进行 STL 分解。
    *   缺乏对波形的“语义”理解。

3.  **扩展性受限 (Limited Extensibility)**：
    *   指标计算与标签规则高度耦合。

4.  **缺乏外部数据联动 (Isolated Data)**：
    *   未结合当天气象数据和分时电价策略。

---

## 2. 重构目标 (Refactoring Goals)

将 `CharacteristicService` 升级为 **Expert System (专家系统)** 水平：
1.  **规则引擎化**：将业务规则从代码中剥离，支持配置化。
2.  **算法智能化**：引入聚类、时序分解等算法。
3.  **画像立体化**：标签包含置信度、行业对标。

---

## 3. 详细优化建议 (Optimization Proposals)

### 3.1 核心算法升级

#### A. 班次判定：从“阈值”转向“形态聚类”
*   **建议**：
    *   **K-Means 聚类**：聚类出 5-8 种典型模式。
    *   **模板匹配**：计算 DTW 距离。

#### B. 季节性分析：引入 STL 分解
*   **建议**：
    *   使用 STL 分解剥离趋势和季节项。
    *   **气温相关性**：利用 `v1_weather.py` 提供的气温数据计算 CDD/HDD 相关性。

#### C. 光伏识别：基于物理模型
*   **建议**：
    *   **晴雨对比法**：对比晴天/阴雨天午间负荷差异。

### 3.2 架构重构建议

#### A. 引入规则引擎 (Rule Engine Pattern)
将标签逻辑抽象为独立的 `Rule` 类。

```python
class BaseRule(ABC):
    @abstractmethod
    def evaluate(self, context: LabelingContext) -> Optional[TagItem]:
        pass
```

#### B. 分层指标计算
1.  **基础统计层**：Min, Max, Mean, Std
2.  **形态特征层**：Ramp Rate, Peak Time
3.  **业务语义层**：Shift Pattern ID, Weather Sensitivity

### 3.3 业务深度增强

1.  **行业对标**：引入行业基准线。
2.  **异动检测**：引入 Isolation Forest 进行无监督检测。

## 5. 执行策略与工程化 (Execution Strategy)

### 5.1 扫描范围 (Scope)
原则：仅分析**有效签约客户**。通过 `ContractService` 获取列表。

### 5.2 运行模式 (Analysis Mode)

2.  **手工触发 (On-Demand)**：API `POST /analyze/all` 或 `/analyze/{id}`。

## 7. 代码目录结构规划 (Project Structure)

考虑到规则引擎的复杂性，建议将单文件 `characteristic_service.py` 拆分为 **Python Package (包)** 结构：

```
webapp/services/characteristics/  <-- [NEW PACKAGE]
├── __init__.py                   # 暴露 CharacteristicService
├── service.py                    # 服务入口类 (Service Facade)
├── engine/                       # 核心引擎
│   ├── __init__.py
│   ├── context.py                # LabelingContext (数据上下文)
│   └── base_rule.py              # Rule 抽象基类
├── algorithms/                   # 纯数学算法 (无业务)
│   ├── __init__.py
│   ├── stl_decomposition.py      # STL 分解
│   ├── clustering.py             # K-Means / DTW
│   └── statistics.py             # 基础统计 (Mean, CV)
├── rules/                        # 具体业务规则实现
│   ├── __init__.py
│   ├── long_term.py              # 经营趋势、季节性规则 (依赖 stl)
│   ├── short_term_shift.py       # 班次判定规则 (依赖 clustering)
│   ├── short_term_asset.py       # 光伏、储能规则 (依赖 weather)
│   └── anomaly.py                # 异动检测规则
└── constants.py                  # 阈值和配置常量
```

## 8. 待完善项与风险 (Analyzed Gaps & Risks)

经自我审查，本方案实施前还需补全以下 **技术缺口**：

### 8.1 依赖库缺失 (Missing Dependencies)
当前的 `requirements.txt` 仅包含 `scipy`，缺乏高级分析所需的库：
*   **statsmodels**：用于 STL/Loess 时序分解。
*   **scikit-learn**：用于 K-Means 聚类和 Isolation Forest。
*   **fastdtw**：用于计算 DTW 距离（模板匹配）。
*   **lunardate**：用于春节日期计算。
> **行动**：需更新 `webapp/requirements.txt` 并重新构建环境。

### 8.2 数据模型变更 (Schema Changes)
建议书中提到了“置信度”和“规则来源”，但目前的 database model (`Tag` in `customer.py`) 并不支持这些字段。
*   **Tag Model 需要新增字段**：
    *   `confidence` (float): 置信度评分 (0.0-1.0)。
    *   `rule_id` (str): 命中规则的唯一标识 (用于追溯)。
    *   `metadata` (dict): 辅助信息 (如 "cluster_id": 3, "similarity": 0.95)。
> **行动**：需修改 `webapp/models/customer.py` 并编写 MongoDB 迁移脚本（如果需要回填旧数据）。

### 8.3 前端接口契约 (Frontend Contract)
“立体画像”需要前端支持展示 Rich Tag（带颜色的置信度、悬停显示理由等）。
*   **API 响应变更**：前端需要适配新的 `Tag` 结构。
*   **UI 组件**：需要开发新的 `TagChip` 组件来展示复杂信息。
> **行动**：需同步更新《负荷特征分析接口文档》。

### 8.4 配置管理 (Configuration)
规则的阈值（如 `CV > 0.2`）存放在哪里？
*   **方案 A**：`constants.py` 硬编码 (开发修改)。
*   **方案 B**：`config.ini` 文件配置 (运维修改)。
*   **方案 C**：MongoDB配置表 (运营动态修改)。
> **决策**：建议第一阶段采用 **方案 A + B** (常量文件 + 关键阈值读配置)，避免过度设计。

### 8.5 新增数据集结构说明 (New Dataset Structures)

为了支撑"行业对标"和"形态聚类"，需新增以下 MongoDB 集合：

#### 1. `industry_benchmarks` (行业基准)
用于存储各行业在不同季节的典型负荷特征指标（如负荷率、峰谷差率），支持 P10/P50/P90 分位值。

```javascript
{
  "_id": ObjectId("..."),
  "industry_code": "C30",          // 行业代码
  "industry_name": "电子设备制造",
  "season": "summer",              // 季节 (spring/summer/autumn/winter)
  "metrics": {
    "avg_load_rate": {
      "p10": 0.45,
      "p50": 0.65,
      "p90": 0.85
    },
    "peak_valley_ratio": { ... }
  },
  "updated_at": ISODate("...")
}
```

#### 2. `load_patterns` (典型负荷模式)
存储算法生成的或人工定义的典型一日负荷曲线模板，用于 K-Means 聚类中心或 DTW 模板匹配。

```javascript
{
  "_id": ObjectId("..."),
  "name": "标准单班-早8晚5",
  "type": "shift_template",        // 类型：shift_template / cluster_center
  "algorithm": "manual",           // 来源：manual / kmeans_v1
  "points": [0.1, 0.1, ..., 0.8, ...], // 归一化后的96点曲线
  "features": {                    // 关联特征
    "shift_type": "single_shift",
    "work_hours": "08:00-17:00"
  }
}
```

#### 3. `analysis_history_log` (分析历史日志)
用于追溯客户标签的变化历史，支撑 A/B 测试和审计。

```javascript
{
  "_id": ObjectId("..."),
  "customer_id": "12345",
  "date": "2024-05-20",            // 业务日期
  "execution_time": ISODate("..."), // 执行时间
  "tags_snapshot": [               // 当次分析生成的完整标签快照
    { "name": "连续生产", "confidence": 0.95, "rule_id": "rule_shift_01" }
  ],
  "metrics_snapshot": {            // 关键指标快照
    "daily_avg_load_rate": 0.78,
    "cv": 0.12
  }
}
```
