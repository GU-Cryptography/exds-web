# 月度结算业务说明

## 1. 文档目的

本文档基于当前代码实现，完整说明系统内“批发侧月度结算”和“零售侧月度结算”的业务逻辑、数据来源、核心计算规则、字段口径以及关键差异点。

本文档以当前后端实现为准，主要对应以下代码：

- `webapp/services/wholesale_monthly_settlement_service.py`
- `webapp/services/retail_settlement_service.py`
- `webapp/services/retail_monthly_settlement_service.py`
- `webapp/services/settlement_service.py`

适用范围：

- 批发侧月度结算台账导入与对账
- 零售侧客户月度结算
- 月结后 `retail_settlement_daily(settlement_type="monthly")` 日清重算
- 月度利润、采购成本、采购均价等关键字段的真实口径

---

## 2. 总体业务结构

系统中的月度结算由两条主线组成：

1. 批发侧月度结算
2. 零售侧月度结算

它们之间的关系如下：

1. 批发侧先导入交易中心月结文件，形成 `wholesale_settlement_monthly`
2. 零售侧读取：
   - `customer_monthly_energy`
   - `wholesale_settlement_monthly`
   - 月内负荷、合同、价格等基础数据
3. 零售侧生成：
   - `retail_settlement_monthly_status`
   - `retail_settlement_monthly`
4. 零售月结完成后，再重算月结口径的日清数据，写入：
   - `retail_settlement_daily`，其中 `settlement_type="monthly"`

## 4. 批发侧月度结算

## 4.1 功能目标

批发侧月度结算模块的目标是：

1. 导入交易中心发布的月度结算文件
2. 抽取本公司主体行
3. 存储月结台账
4. 与 `settlement_daily(version=PLATFORM_DAILY)` 做月度对账

对应服务：

- [wholesale_monthly_settlement_service.py](/d:/Gitworks/exds-web/webapp/services/wholesale_monthly_settlement_service.py)

## 4.2 月结文件导入规则

### 4.2.1 文件来源

- Excel 文件（`.xls/.xlsx`）
- 使用双行表头读取

### 4.2.2 主体识别

导入时只保留本公司主体行：

- 默认主体名：
  - `国网江西综合能源服务有限公司`
- 可通过环境变量覆盖：
  - `WHOLESALE_SUBJECT_NAME`

会过滤掉：

- `合计`
- `全市场合计`

### 4.2.3 月份主键

- `_id = month`
- 格式：`YYYY-MM`

同月重复导入时：

- 若未传 `overwrite=true`，报冲突
- 若允许覆盖，则整月替换

## 4.3 批发月结落库内容

集合：

- `wholesale_settlement_monthly`

核心字段分为三类：

1. `settlement_items`
   - 直接来自月结文件主体行
2. `period_details`
   - 由 `settlement_daily(version=PLATFORM_DAILY)` 按月聚合得到
3. `reconciliation_results`
   - 月结文件值与日清聚合值之间的差异对比

## 4.4 批发月结分时明细 `period_details`

`wholesale_settlement_monthly.period_details` 不是文件原始明细，而是由日清聚合得到。

计算逻辑：

1. 读取当月所有 `settlement_daily(version=PLATFORM_DAILY)`
2. 展开 `period_details`
3. 以 `period(1-48)` 分组求和
4. 生成每时段：
   - `contract.volume/price/fee`
   - `day_ahead.volume/price/fee`
   - `real_time.volume/price/fee`
   - `total_energy_fee`

注意：

- 这里没有把月度 `balancing_fee`
- 也没有把 `fund_surplus_deficit_total`
- 也没有把 `deviation_recovery_return_fee`

拆到 48 时段

因此该 48 时段明细本质上是：

- 批发日清电能量口径的月聚合明细

不是完整月结总成本的 48 时段拆解。

## 4.5 批发月结对账逻辑

对账使用的日清来源固定为：

- `settlement_daily.version = PLATFORM_DAILY`

核心聚合规则：

- `contract_fee = sum(settlement_daily.contract_fee)`
- `day_ahead_fee = sum(settlement_daily.day_ahead_fee)`
- `real_time_fee = sum(settlement_daily.real_time_fee)`
- `deviation_recovery_fee = sum(settlement_daily.deviation_recovery_fee)`
- `total_energy_fee = sum(settlement_daily.total_energy_fee)`

并进一步计算：

- `energy_avg_price = total_energy_fee / real_time_volume`
- `settlement_fee_total = total_energy_fee + deviation_recovery_fee`
- `settlement_avg_price = settlement_fee_total / real_time_volume`

特殊处理：

- 对账时会把月结文件中的 `balancing_fee` 加回到日清聚合的 `energy_fee_total`
- 再得到日清侧的“近似月结口径”

即：

- `daily_energy_fee_with_balancing = sum(total_energy_fee) + balancing_fee`
- `daily_settlement_fee_with_balancing = daily_energy_fee_with_balancing + sum(deviation_recovery_fee)`

这意味着：

- 对账的“结算电费/结算均价”是近似口径
- 不等于完整月结所有字段逐项完全可逆

## 5. 批发侧日清成本基础

批发侧日清逻辑由 `SettlementService` 负责。

对应文件：

- [settlement_service.py](/d:/Gitworks/exds-web/webapp/services/settlement_service.py)

## 5.1 批发日清 48 时段计算

每个时段的核心输入：

- `q_rt`：实时用电量
- `p_rt`：实时价
- `q_da`：日前电量
- `p_da`：日前价
- `q_contract`：中长期合同量
- `p_contract`：合同价
- `q_mech`：机制电量
- `p_market_avg`：市场均价
- `p_company_contract`：公司合同均价

核心费用：

- `cost_contract = q_contract * (p_contract - p_rt)`
- `cost_da = q_da * (p_da - p_rt)`
- `cost_rt = q_rt * p_rt`
- `total_energy_fee = cost_contract + cost_da + cost_rt`

## 5.2 标准值考核逻辑

代码中采用“取大机制”：

1. 先计算实际费用 `cost_actual = total_energy_fee`
2. 再按签约比例 `k_ratio = (q_contract + q_mech) / q_rt` 计算标准值费用
3. 日汇总时：
   - `deviation_recovery_fee = max(0, total_standard_value_cost - total_energy_fee)`

最终日汇总得到：

- `predicted_wholesale_cost = total_energy_fee + deviation_recovery_fee`
- `predicted_wholesale_price = predicted_wholesale_cost / real_time_volume`

这套结果构成了零售侧日结采购成本的基础。

## 6. 零售侧日结

零售侧日结由 `RetailSettlementService` 负责。

对应文件：

- [retail_settlement_service.py](/d:/Gitworks/exds-web/webapp/services/retail_settlement_service.py)

## 6.1 日结输入

单客户单日计算依赖：

1. 客户当天有效合同
2. 套餐定价模型
3. 当天 48 时段 TOU
4. 客户当天 48 时段负荷（`MP_ONLY`）
5. 批发侧当天 48 时段采购价

## 6.2 批发采购价来源

零售日结采购价由 `_get_wholesale_period_prices(date_str)` 提供。

规则：

1. 优先取 `settlement_daily(version=PLATFORM_DAILY)`
2. 否则回退 `PRELIMINARY`
3. 判断：
   - 若 `total_standard_value_cost > total_energy_fee`
   - 则按 `standard_value_cost / real_time.volume`
   - 否则按 `total_energy_fee / real_time.volume`

因此零售侧日结采购价是“按批发日清取大机制”形成的 48 时段采购均价，单位：

- 元/MWh

## 6.3 零售售价模型

支持四类模型：

1. `price_spread*`
2. `fixed_linked*`
3. `reference_linked*`
4. `single_comprehensive*`

共同规则：

1. 先形成基础价格
2. 执行 463 比例校验
3. 如触发封顶，再执行封顶压缩

## 6.4 463 比例规则

当前代码固定比例为：

- 尖峰 `1.8`
- 高峰 `1.6`
- 平段 `1.0`
- 低谷 `0.4`
- 深谷 `0.3`

平段作为基准：

- 尖峰/高峰不足则上调
- 低谷/深谷超标则下调

从 `2026-02` 开始：

- 浮动价和浮动费不纳入 463 校验
- 仅对基准价部分做比例校验

## 6.5 封顶规则

日结和月结都存在封顶逻辑，但计算基准不同。

日结封顶：

1. 先算当天名义均价 `nominal_avg_price`
2. 读取月封顶价 `cap_price`
3. 若 `nominal_avg_price > cap_price`
4. 则按系数 `k = cap_price / nominal_avg_price` 对售价同比例压缩

当前日结封顶价来源：

- `price_sgcc.avg_on_grid_price`
- 尖峰月上浮 15%
- 非尖峰月上浮 10%

## 6.6 零售日结采购成本

对每个时段：

- `allocated_cost = load_mwh * wholesale_price`

其中：

- `wholesale_price` 为 `_get_wholesale_period_prices` 返回的当日时段采购价
- 单位元/MWh

日结结果存入：

- `retail_settlement_daily`
- `settlement_type = daily`

## 7. 零售侧月结总流程

零售月结由 `RetailMonthlySettlementService` 负责。

对应文件：

- [retail_monthly_settlement_service.py](/d:/Gitworks/exds-web/webapp/services/retail_monthly_settlement_service.py)

整体流程共 3 步：

1. 按客户聚合月度 48 时段电量并计算零售月结
2. 计算超额返还并回写客户月结结果
3. 重算 `settlement_type="monthly"` 的日清记录

## 7.1 月结前置条件

必须同时具备：

1. `customer_monthly_energy` 存在该月记录
2. `wholesale_settlement_monthly` 存在该月记录
3. 月封顶参考价可获取

若缺失则月结任务直接失败。

## 7.2 客户聚合规则

月结先对 `customer_monthly_energy.records` 按 `customer_name` 聚合：

- `monthly_energy_mwh = sum(energy_mwh)`
- `customer_nos = 去重后的 customer_no 列表`

然后通过以下顺序解析 `customer_id`：

1. 先按 `customer_archives.user_name == customer_name`
2. 若命中，直接返回单个 `customer_id`
3. 否则再按 `accounts.account_id in customer_nos` 反查多个客户

这意味着：

- 同名客户可能被合并
- 档案匹配不完整会影响后续成本和电量口径

## 7.3 月结负荷来源

月结计算使用客户月内所有日负荷的 `MP_ONLY` 结果：

1. 逐天读取每日 48 时段曲线
2. 聚合为月度 48 时段电量

因此月结价格计算的负荷口径是：

- 客户月内实际负荷聚合

不是 `customer_monthly_energy` 的导入电量。

## 7.4 月结价格模板计算

月结售价仍沿用零售日结的定价模型，但改为按月总电量、按月模板计算。

主要特点：

1. 选取模板日：
   - 一个常规日
   - 如存在与常规日 TOU 不同的节假日，则再选一个节假日模板
2. 生成：
   - `final_prices`
   - `final_prices_48`
3. 若使用双模板，则按当月各时段在常规/节假日中的真实构成来计算月度名义电费

## 7.5 月结封顶

月结封顶不是按日，而是按整月名义均价判断。

步骤：

1. 计算 `nominal_total_fee`
2. 计算：
   - `nominal_avg_price = nominal_total_fee / (total_energy * 1000)`
3. 读取月封顶价
4. 若超封顶，则对 `final_prices` 与 `final_prices_48` 同比例缩放

月结封顶基准优先级：

1. `retail_price_service.get_monthly_base_price(month, "market_longterm_flat_avg")`
2. 若缺失，回退 `price_sgcc.avg_on_grid_price`

## 7.6 月结 `period_details` 的零售价部分

月结 `retail_settlement_monthly.period_details` 中的：

- `load_mwh`
- `unit_price`
- `fee`

是按月度 48 时段电量与月结价格模板直接算出的。

对“混合时段”：

- `period_type = period_type_mix`
- 同时记录 `period_type_breakdown`

用于保留该月内同一个 30 分钟时段在不同日历日可能对应多种 TOU 的情况。

## 8. 零售月结中的采购成本逻辑

这部分是系统最容易产生口径误解的部分。

## 8.1 月结 `period_details.allocated_cost`

当前 `retail_settlement_monthly.period_details[].allocated_cost` 的来源不是批发月结台账，而是：

- 月内 `retail_settlement_daily(settlement_type="daily")`
- 同客户
- 同时段
- `allocated_cost` 的聚合和

公式：

- `monthly_period_allocated_cost[p] = sum(daily_period_allocated_cost[p])`

因此它的口径是：

- 日清采购成本的月聚合

不是：

- 批发月结总成本按客户月结电量分摊后的时段结果

## 8.2 `pre_wholesale_fee / pre_wholesale_unit_price`

调平前采购成本：

- `pre_wholesale_fee = total_allocated_cost`
- `pre_wholesale_unit_price = total_allocated_cost / daily_energy_mwh`

这里的 `total_allocated_cost` 就是上面的日清采购成本月聚合值。

## 8.3 `sttl_wholesale_fee / sttl_wholesale_unit_price`

调平后采购成本仍使用旧算法：

- `sttl_wholesale_fee = total_allocated_cost + balancing_wholesale_fee + surplus_fee`
- `sttl_wholesale_unit_price = sttl_wholesale_fee / total_energy_mwh`

其中：

- `balancing_wholesale_fee = balancing_energy_mwh * balancing_price`
- `surplus_unit_price = (fund_surplus_deficit_total - deviation_recovery_fee) / actual_monthly_volume`
- `surplus_fee = total_energy_mwh * surplus_unit_price`

因此 `sttl_*` 的口径是：

- 日清采购成本聚合
- 加月度调平批发成本
- 加资金余缺分摊

## 8.4 `final_wholesale_fee / final_wholesale_unit_price`

当前实现中，最终采购金额已切换为按批发月结总额反算：

- `wholesale_total_fee = wholesale_settlement_monthly.settlement_items.settlement_fee_total`
- 按客户最终结算电量 `total_energy_mwh` 占全部客户最终结算电量的比例分摊

公式：

- `final_wholesale_fee_i = wholesale_total_fee * total_energy_i / sum(total_energy_all)`
- `final_wholesale_unit_price_i = final_wholesale_fee_i / total_energy_i`

最后一个客户承担舍入尾差，保证全体客户合计后精确回到批发月结总金额。

注意：

- `final_wholesale_fee` 与 `sttl_wholesale_fee` 当前可能不相等
- 这是设计使然：最终利润字段按批发月结总额统一反算
- 但中间过程字段仍保留旧算法

## 8.5 `final_gross_profit`

最终毛利按最终零售收入减最终采购成本计算：

- `final_gross_profit = final_retail_fee - final_wholesale_fee`

因此 `final_gross_profit` 与 `final_wholesale_fee` 强绑定。

## 9. 零售月结中的调平逻辑

## 9.1 调平电量

- `balancing_energy_mwh = customer_monthly_energy - daily_energy_mwh`

其中：

- `customer_monthly_energy` 来自导入表
- `daily_energy_mwh` 来自月内实际负荷聚合

## 9.2 调平零售电费

规则：

1. 若 `balancing_energy_mwh < 0`
   - 调平电价不参与
   - 按调平前零售单价计费
2. 若 `balancing_energy_mwh >= 0`
   - 按批发侧月度调平电价 `balancing_price` 计费

## 9.3 封顶客户的调平二次校验

若客户原本已经触发封顶：

1. 先回退到封顶前名义电价口径
2. 加上调平电费
3. 再判断调平后是否仍超封顶
4. 若仍超封顶，则调平后单价维持封顶价

因此：

- 封顶客户的调平后零售价不一定等于简单的“原零售价 + 调平价”

---

## 10. 超额返还逻辑

## 10.1 月度状态中的返还池

系统先在 `retail_settlement_monthly_status` 中计算整月返还池。

步骤：

1. 计算全体客户调平后零售总收入 `retail_total_fee`
2. 计算全体客户调平后零售总电量 `retail_total_energy`
3. 得到：
   - `retail_avg_price = retail_total_fee / (retail_total_energy * 1000)`
4. 与批发月结均价比较：
   - `price_margin_per_mwh = retail_avg_price * 1000 - wholesale_avg_price`
5. 若超过阈值 10 元/MWh，则超额部分形成返还池

公式：

- `excess_profit_per_mwh = max(retail_avg_price * 1000 - wholesale_avg_price - 10, 0)`
- `excess_profit_total = excess_profit_per_mwh * retail_total_energy`
- `excess_refund_pool = excess_profit_total * 0.8`

即：

- 超额收益的 80% 返还客户
- 20% 保留给售电公司

## 10.2 客户返还分摊

返还金额按客户最终结算电量占比分配：

- `refund_ratio_i = total_energy_i / sum(total_energy_all)`
- `final_excess_refund_fee_i = excess_refund_pool * refund_ratio_i`

## 10.3 最终零售电费

- `final_retail_fee = total_fee_before_refund - final_excess_refund_fee`
- `final_retail_unit_price = final_retail_fee / final_energy_mwh`

## 11. 月结后日清重算

月结完成后，系统会生成 `retail_settlement_daily(settlement_type="monthly")`。

## 11.1 目的

目的不是重算调平，而是生成“月结口径下的每日零售收入与采购成本快照”，用于：

- 月度详情页逐日分析
- 月结后日清查询

## 11.2 重算零售价格

逐日重算时读取：

- `retail_settlement_monthly.price_model.final_prices`
- 若有 `period_details`，则优先反推更高精度的 `final_prices_48`

然后再减去客户度电返还：

- `refund_unit_price_kwh = final_excess_refund_fee / (final_energy_mwh * 1000)`
- `daily_unit_price_kwh = monthly_base_price_kwh - refund_unit_price_kwh`

允许出现负价。

## 11.3 重算采购价格

逐日采购单价不是来自 `retail_settlement_monthly.final_wholesale_unit_price`，而是：

- `procure_price_p = day_wholesale_price_p + surplus_unit_price`

其中：

- `day_wholesale_price_p` 来自当天 `settlement_daily(version=PLATFORM_DAILY)` 的时段采购价
- `surplus_unit_price` 来自月结状态表

注意：

- 这里没有把 `balancing_wholesale_fee` 分摊进每日 48 时段采购成本

因此 `retail_settlement_daily(settlement_type="monthly")` 的采购口径为：

1. 当天批发时段价
2. 加月度资金余缺度电分摊
3. 不含月度调平批发金额

## 11.4 `monthly` 类型日清记录的作用

该表适合做：

- 日维度收入分析
- 日维度采购趋势分析
- 月结口径下的日粒度展示

不适合直接用于解释：

- `retail_settlement_monthly.final_wholesale_fee`

因为两者采购口径并不完全一致。

### 12.1 批发侧

## 13. 当前实现中的重要差异点

以下差异是理解报表误差的关键。

### 13.1 `final_*` 与 `sttl_*` 不同口径

当前实现中：

- `sttl_wholesale_fee` 仍然使用旧算法
- `final_wholesale_fee` 改为按批发月结总额按客户电量比例反算

所以：

- `final_wholesale_fee != sttl_wholesale_fee` 是允许的

### 13.2 `period_details.allocated_cost` 不等于最终采购成本拆解

`period_details.allocated_cost` 仍是日清采购成本月聚合，没有同步替换为 `final_wholesale_fee` 的时段分摊结果。

因此：

- 明细时段采购金额之和
- 与最终采购金额

可能不一致。

### 13.3 月结后日清采购成本与客户最终采购金额不一致

`settlement_type="monthly"` 的日清记录采购成本：

- 用日时段价 + 资金余缺
- 不含月度调平批发金额

而客户最终采购金额：

- 直接按批发月结总额按客户电量分摊

因此两者不是同一套核算口径。

### 13.4 批发月结 `period_details` 也不是完整月结成本拆解

批发月结的 48 时段明细本身也只承载电能量日清聚合结果，并未拆解月度调平、资金余缺等月度项。

所以“从批发月结总额精确反推出客户 48 时段最终采购金额”在当前模型中并没有唯一答案，必须额外定义分摊规则。

## 14. 业务理解建议

在当前系统中，应按以下方式理解月结数据：

1. 批发侧月结总额
   - 以 `wholesale_settlement_monthly.settlement_items` 为准
2. 零售客户最终利润
   - 以 `retail_settlement_monthly.final_*` 为准
3. 零售客户中间过程分析
   - 以 `pre_*`、`sttl_*`、`period_details` 为准
4. 月结后逐日分析
   - 以 `retail_settlement_daily(settlement_type="monthly")` 为准

不要直接把以下两者当作同一口径：

- `retail_settlement_monthly.period_details.allocated_cost`
- `retail_settlement_monthly.final_wholesale_fee`

---

## 15. 后续若要进一步统一口径，可选方案

若未来希望所有页面、明细、合计完全统一，可考虑以下重构方向：

1. 为 `final_wholesale_fee` 设计统一的 48 时段分摊规则
2. 将该规则同步写回：
   - `retail_settlement_monthly.period_details.allocated_cost`
   - `retail_settlement_monthly.period_details.wholesale_price`
3. 再用同一套分摊规则重算：
   - `retail_settlement_daily(settlement_type="monthly")`

只有做到这一点，以下三者才能严格统一：

1. 客户月结最终采购金额
2. 客户月结时段采购明细
3. 月结后逐日采购成本

---

## 16. 结论

当前系统中的月度结算逻辑是“批发月结总额 + 零售月结价格模型 + 返还 + 月结后日清重算”的组合实现。

其核心特征是：

1. 批发月结总额是外部导入的权威口径
2. 零售月结价格模型按系统规则独立计算
3. 最终利润字段已经按批发月结总额统一反算
4. 明细采购金额和月结后日清采购金额仍保留历史算法

因此当前系统是“最终利润口径已统一，但中间采购过程字段仍保留旧算法”的状态。理解这一点，是正确使用各类月结报表的前提。
