# 月结后客户日清重算实施方案

## 1. 目标与范围

- 目标：在客户完成月度结算后，为该月生成“月结口径日清”数据，写入 `retail_settlement_daily`，并使用 `settlement_type="monthly"` 与日预结算隔离。
- 范围：
  - 后端结算流程（零售月结服务 + 零售日结读取链路）
  - 数据结构文档修订
  - 查询接口 `settlement_type` 过滤修复
- 不在本期范围：前端页面大改，仅补充必要查询参数与默认口径。

## 2. 现状与问题

### 2.1 已实现能力

- 已有月结结果表：`retail_settlement_monthly`。
- 已有日结表：`retail_settlement_daily`，主键支持 `customer_id + date + settlement_type`。
- 零售月结状态表已预留字段：`retail_daily_recomputed`，当前始终为 `False`。

### 2.2 关键缺口

1. 缺入口：月结流程未触发“月结口径日清重算”。
2. 缺过滤：多个查询未限制 `settlement_type`，会混淆 `daily/monthly`。
3. 文档冲突：`dataset_structures_v2.md` 第20节写明“月结流程不再写入 `retail_settlement_daily` 的 `monthly` 记录”，与新需求冲突。

## 3. 采购侧/零售侧规则复核与漏洞审计

### 3.1 采购侧（现状）

- 日结采购单价来自 `settlement_daily.period_details`，按“标准值费用/电能量费用二选一”计算每时段元/MWh单价。
- 月结状态中计算 `surplus_unit_price`：`(fund_surplus_deficit_total - deviation_recovery_fee) / actual_monthly_volume`。

### 3.2 零售侧（现状）

- 月结阶段已按月总电量执行封顶判断（正确）。
- 超额返还仅回写到客户月结总额字段（`final_excess_refund_fee`），尚未传导到“日清时段单价”。

### 3.3 发现的业务漏洞（需修复）

1. 月结后日清未重算：导致日清采购成本与零售价仍是预结算口径。
2. `settlement_type` 混查：月度总览、结算详情、客户详情存在混算风险。
3. 单位风险：
   - 日结 `unit_price` 用元/kWh，采购 `wholesale_price` 用元/MWh；
   - 返还按“度电价差”下沉时需明确换算（`/1000`）。
4. 返还下沉风险：若直接均摊返还，深谷/低谷可能出现负价，**已确认允许负价**。
5. 资金余缺口径待确认：当前代码将 `deviation_recovery_fee` 从资金余缺中扣除后再摊；需与财务口径最终确认。

## 4. 目标口径（本方案）

### 4.1 采购侧（月结口径日清）

- 使用当天真实48时段采购价（元/MWh），数据源为 `settlement_daily.version = PLATFORM_DAILY`。
- 每时段叠加月度资金余缺度电分摊价（元/MWh）：
  - `procure_price_p = day_wholesale_price_p + surplus_unit_price`
  - 注：`surplus_unit_price` 为公司级月度均摊值，统一叠加到所有客户所有时段，无法精确分时。
- 时段采购成本：
  - `allocated_cost_p = load_mwh_p * procure_price_p`

### 4.2 零售侧（月结口径日清）

- 使用月结计算得到的客户5时段价格（已包含月度封顶规则结果），读取自 `retail_settlement_monthly.{customer_id}.price_model.final_prices`（5时段字典，单位：元/kWh）。
- 计算每日各时段 `period_type_p` 时，使用该日实际峰谷时段定义，与月结计算逻辑保持一致。
- 按客户月度返还总额折算度电返还价差（元/kWh）：
  - `refund_unit_price_kwh = final_excess_refund_fee / (final_energy_mwh * 1000)`
- 日时段零售价：
  - `retail_unit_price_kwh_p = monthly_5tou_price_kwh(period_type_p) - refund_unit_price_kwh`
  - 允许负价（深谷/低谷时段返还后可出现负值，业务上接受）。

### 4.3 调平电量处理

- 月结口径日清不包含调平电量和调平费用（批零同价，价差收益为0）。
- 调平结果仅保留在 `retail_settlement_monthly`。

## 5. 技术实施方案

### 5.1 新增/完善入口（第6点）

在 `RetailMonthlySettlementService.run_monthly_settlement()` 增加步骤3：

1. 步骤1：按客户生成月结基础结果（已存在）。
2. 步骤2：计算并回写超额返还（已存在）。
3. 步骤3：重算该月客户日清月结版本（新增）。

新增方法建议：

- `_recompute_monthly_daily_settlements(month: str, force: bool = True) -> Dict`
- `_build_customer_monthly_daily_context(monthly_doc, status_doc) -> Dict`
- `_upsert_customer_monthly_daily_record(...)`

状态回写：

- `retail_settlement_monthly_status.retail_daily_recomputed = True`
- 新增建议字段：
  - `retail_daily_recomputed_at`
  - `retail_daily_recomputed_count`
  - `retail_daily_recomputed_failed`

### 5.2 查询过滤修复（第7点）

必须修复以下查询默认过滤：`settlement_type="daily"`。

- `webapp/api/v1_retail_settlement.py`
  - `GET /retail-settlement/daily` 增加 `settlement_type` 参数（默认 `daily`）。
- `webapp/services/settlement_service.py`
  - `get_settlement_detail()` 查询 `retail_settlement_daily` 增加 `settlement_type` 过滤。
  - `get_settlement_customer_detail()` 增加 `settlement_type` 参数并过滤。
- `webapp/api/v1_settlement.py`
  - `/overview` 聚合管道增加 `settlement_type` 过滤（默认 `daily`）。
  - `/detail`、`/customer-detail` 透传 `settlement_type`。
- `webapp/services/retail_settlement_service.py`
  - `get_settled_count()` 增加可选 `settlement_type` 参数，默认 `daily`。

### 5.3 数据落库策略

- 集合：`retail_settlement_daily`
- 主键：`customer_id + date + settlement_type`
- 写入方式：`upsert` 覆盖 `settlement_type="monthly"` 版本
- 不影响 `settlement_type="daily"` 预结算数据

## 6. 文档修订（第5点）

需要更新 `docs/dataset_structures_v2.md`：

1. 第20节删除/改写“月结流程不再写入 `retail_settlement_daily` 的 `monthly` 记录”。
2. 第14节补充“`monthly` 记录由月结完成后重算生成，口径不含调平电量”。
3. 增加字段口径备注：
   - `period_details.unit_price` 元/kWh
   - `period_details.wholesale_price` 元/MWh
   - `period_details.allocated_cost` 元

## 7. 边界与风控

1. 负价策略（**已确认选方案A**）：允许负价，严格反映返还和资金余缺分摊结果，不做截断。
   - 零售侧：深谷/低谷时段返还后单价可为负，业务上接受。
   - 采购侧：资金余缺为负（公司月度亏损）时，`surplus_unit_price < 0`，各时段采购单价可低于当天现货价，允许负采购成本。
2. 资金余缺公式确认：是否扣减 `deviation_recovery_fee` 后再分摊。确认是扣减’deviation_recovery_fee‘再分摊。
3. 客户映射：同名多ID客户需沿用月结结果中的 `customer_id`，避免重复写入。
4. 幂等：重复月结重算必须可覆盖，不新增重复记录。

## 8. 验收标准

1. 月结任务完成后，`retail_settlement_daily` 出现对应月份 `settlement_type="monthly"` 记录。
2. 任何日结查询默认不混入 `monthly` 记录（除非显式指定）。
3. 单客户月内校验：
   - `sum(daily_monthly.total_fee) ≈ final_retail_fee - sttl_balancing_retail_fee`（允许小额舍入差）。
     - 说明：`final_retail_fee` 已含超额返还扣减；`sttl_balancing_retail_fee` 为调平电费；两者相减即为"剔除调平的月结口径零售电费"，应与月结口径日清汇总对齐。
   - `sum(daily_monthly.total_allocated_cost)` 与“日真实采购 + 资金余缺分摊”一致。
4. 回归通过：现有 `daily` 页面与接口输出不发生口径漂移。

## 9. 实施顺序

1. 后端：先补重算入口与核心计算。
2. 后端：补 `settlement_type` 过滤并回归。
3. 文档：更新 `dataset_structures_v2.md` 与接口说明。
4. 前端：按需增加查询参数（默认 `daily`）。
5. 联调：选定 1 个已月结月份进行全链路核对。

## 10. 结论

本方案可在不破坏现有预结算流程的前提下，补齐“月结后日清重算”闭环。重点是先统一单位与口径，再落地入口和过滤修复，最后做月内汇总对账验收。
