# 零售月度结算功能方案

## 1. 目标与范围
- 建立“零售月度结算”完整闭环：按月触发、按客户顺序执行月度正式日清重算（`settlement_type = "monthly"`）、生成客户月结记录、计算并回写超额返还。
- 页面挂载于“结算管理 -> 零售月度结算”。
- 页面交互分三块：
1. 年份选择
2. 月度汇总（每月一行 + 合计行 + 结算按钮）
3. 客户月度结算明细（桌面表格、移动端卡片）

## 2. 数据依赖与口径

### 2.1 必要数据集
- `customer_monthly_energy`
- `wholesale_settlement_monthly`
- `retail_settlement_daily`（已有，月度重算写入 `settlement_type = "monthly"`）

### 2.2 关于 `retail_settlement_prices`
- 零售月结不受 `retail_settlement_prices` 文件导入约束。
- 文件未发布时，价格由 `retail_price_service.py` 的降级方案提供，支持月度场景：
1. 优先走可用参考价
2. 若缺失，使用现货口径聚合（含 `day_ahead_avg` / `real_time_avg` / `day_ahead_avg_econ`）
3. 实时现货联动部分可按“全月同一时段平均价”参与计算
- 文件后续发布后，可对同月执行“重算”。

### 2.3 前置条件（结算时校验）
- 必须存在当月 `customer_monthly_energy` 记录。
- 必须存在当月 `wholesale_settlement_monthly` 记录。
- 结算按钮可始终显示；点击后若不满足前置条件，后端返回错误并中止。

## 3. 新增数据集设计

### 3.1 月度状态表：`retail_settlement_monthly_status`
- 作用：记录当月零售月结汇总、返还触发判断结果。
- 主键：`_id = month (YYYY-MM)`
- 关键字段：
- `month`
- `wholesale_settled`
- `wholesale_avg_price`
- `retail_daily_recomputed`
- `retail_avg_price`
- `retail_total_energy`
- `retail_total_fee`
- `excess_profit_threshold`（默认 10 元/MWh）
- `excess_profit_total`
- `excess_refund_pool`
- `created_at` / `updated_at`

### 3.2 客户月结表：`customer_retail_monthly_settlement`
- 作用：保存客户月结最终记录（含调平、返还）。
- 不改动 `retail_settlement_daily` 结构；调平和返还仅保存在该月度表。
- 主键建议：`_id = {month}_{safe_customer_name}`（按客户名称聚合存储）
- 关键字段：
- `month`
- `customer_id`
- `customer_name`
- `customer_no`（聚合后的编号串）
- `daily_energy_mwh`
- `retail_fee`
- `retail_avg_price`
- `balancing_energy_mwh`
- `balancing_fee`
- `balancing_avg_price`
- `total_energy_mwh`
- `total_fee`
- `wholesale_avg_price`
- `excess_refund_fee`
- `excess_refund_unit_price`
- `excess_refund_ratio`
- `refund_allocated_at`
- `created_at` / `updated_at`


## 4. 核心业务规则

### 4.1 客户聚合口径
- 结算执行顺序按客户顺序逐个处理。
- `customer_monthly_energy` 按 `customer_name`或 `customer_id` 聚合为客户月度结算电量。
- `customer_monthly_energy` 仅用于计算调平电量，不作为日清重算电量来源。

### 4.2 月度正式日清重算
- 对客户逐日调用日清结算，写入 `retail_settlement_daily`，并明确 `settlement_type = "monthly"`。
- 月度日清电量与零售电费由 `settlement_type = "monthly"` 的结果聚合得到。

### 4.3 调平计算
- `balancing_energy_mwh = customer_monthly_energy(月度聚合) - daily_energy_mwh`
- `balancing_avg_price = retail_avg_price`
- `balancing_fee = balancing_energy_mwh * balancing_avg_price * 1000`

### 4.4 超额返还计算
- 触发阈值：月度平均批零价差上限 10 元/MWh（0.01 元/kWh）。
- 若 `(retail_avg_price*1000 - wholesale_avg_price) > 10`，触发返还。
- 超额部分按 2:8 分成：
- 售电公司保留 20%
- 客户返还池 80%
- 返还池按客户 `total_energy_mwh` 占比分配，不平均分配。

### 4.5 两阶段落库（已确认）
- 阶段1：完成所有客户日清重算后，先落地客户月结基础记录（返还字段先置 0）。
- 阶段2：计算全量返还参数（是否触发、返还池、返还单价）后，回写客户月结返还字段。

## 5. 月结执行流程
1. 点击某月“结算”。
2. 后端校验前置条件（批发月结、客户月度电量）。
3. 打开提示框弹窗，提示用户确认是否进行计算，并预估耗时，以下按客户顺序执行结果都需在弹窗中进度条体现
4. 按客户顺序执行月度正式日清重算（`monthly`）。
5. 生成并保存客户月结基础记录（无返还）。
6. 计算月度状态与返还池，回写客户返还字段。
7. 更新任务状态为 completed，刷新月度汇总与客户明细。
8. 重新计算将覆盖原有记录，不做版本记录。

## 6. UI 设计（最终版）

### 6.1 面板一：年份选择
- 仅保留年份下拉框。
- 不展示“最近导入时间”两行状态。

### 6.2 面板二：月度汇总
- 参考“批发月度结算”布局：每个月一行，按月份顺序排列，1月在第一行。
- 关键列：月份、客户数、总电量、批发均价、零售均价、批零价差、是否触发返还、采购金额、零售金额、返还金额、结算金额、结算均价、操作。
- 批发均价列显示规则：
- 未发布：显示“未发布”
- 已发布：显示具体批发均价
- 每个月均显示“结算”按钮。
- 若前置条件不满足，点击后弹错误提示，不继续执行。
- 合计信息放在该面板底部“合计行”。

### 6.3 面板三：客户月结明细
- 不分页、不折叠。
- 桌面端：表格展示。
- 移动端：卡片展示（样式参考月度记录详情）。
- 展示字段：
- 日清电量
- 日清电费
- 日清均价
- 调平电量
- 调平电费（使用日清均价）
- 合计电量
- 零售电费
- 超额返还
- 结算电费
- 结算均价
- 返还有值行使用背景色高亮。

### 6.4 进度弹窗
- 结算启动后立即弹出进度窗口。
- 显示：阶段消息、当前客户、已处理/总数、成功/失败、进度条。
- 阶段文案建议：
- 步骤1/2：客户月度正式日清重算
- 步骤2/2：计算并回写超额返还

## 7. 接口约定
- `POST /api/v1/retail-settlement/monthly-calc`
- 入参：`month`、`force`
- 出参：`job_id`
- 校验失败返回业务错误信息
- `GET /api/v1/retail-settlement/monthly-progress/{job_id}`
- 返回任务实时进度
- `GET /api/v1/retail-settlement/monthly-summaries`
- 返回按月汇总数据
- `GET /api/v1/retail-settlement/monthly-customers?month=YYYY-MM`
- 返回该月客户月结记录
- `GET /api/v1/retail-settlement/monthly-status?month=YYYY-MM`
- 返回该月状态表记录

## 8. 与现有模块边界
- 不扩展 `retail_settlement_daily` 结构字段。
- 日清表仅新增/更新 `settlement_type = "monthly"` 口径数据。
- 调平、返还及月度汇总都落在月结相关新表中。

## 9. 验收要点
- 2026-01、2026-02 在数据具备时可直接结算。
- 结算过程中可持续看到进度，不会“秒完成且无进度”。
- 客户按名称聚合后，月结电费不应普遍为 0。
- 返还采用“两阶段”结果：先基础、后回写。
- 桌面端为表格，移动端为卡片。
