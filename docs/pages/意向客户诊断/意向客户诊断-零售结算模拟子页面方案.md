# 意向客户诊断 - 零售结算模拟子页面方案

## 1. 文档目标

本文档用于单独说明“意向客户诊断”模块中 `零售结算模拟` 子页面的设计方案，重点覆盖以下内容：

- 页面定位与业务边界
- 页面 UI 布局与交互流程
- 数据集设计
- 后端接口设计
- 前端实现拆分方案
- 与现有签约客户月度结算详情页的复用方式

本方案是对主文档《意向客户诊断模块技术方案》的补充细化，聚焦 Tab3 子页面落地。

---

## 2. 页面定位

### 2.1 页面名称

原“零售套餐分析”子页面，调整为：

- `零售结算模拟`

### 2.2 业务目标

针对单个意向客户，在已完成批发侧月度模拟结算的前提下，允许用户选择不同零售套餐执行零售侧月度模拟结算，并展示：

- 零售收入
- 零售均价
- 批发成本
- 批发均价
- 月毛利
- 批零价差
- 调平影响
- 超额返还影响

### 2.3 业务约束

1. 零售结算模拟必须在批发结算模拟完成之后进行。
2. 批发侧结果按 `customer_id + settlement_month` 唯一。
3. 零售侧结果按 `customer_id + settlement_month + package_id` 唯一。
4. 批发侧重新计算时，覆盖相同唯一键记录。
5. 零售侧重新计算时，覆盖相同唯一键记录。
6. 页面第一层展示的“已计算套餐”直接来源于零售模拟结果集，不在 `intent_customer_profiles` 中冗余保存套餐信息。
7. “增加套餐结算”弹框中展示的套餐来源为 `retail_packages` 集合中所有 `status=active` 的套餐。

---

## 3. 页面整体结构

页面结构参考“批发结算模拟”Tab 的布局方式，采用三层面板：

1. 第一层面板：已计算套餐列表 + 增加套餐结算按钮
2. 第二层面板：当前选中套餐的月份结算结果表格
3. 第三层面板：当前选中 `套餐 + 月份` 的月度结算详情

## 4. UI 方案

### 4.1 第一层面板：已计算套餐列表

#### 4.1.1 面板内容

- 已计算套餐列表
- `增加套餐结算` 按钮

#### 4.1.2 布局示意

```text
+--------------------------------------------------------------------------------------------------+
| 已计算套餐： [套餐A] [套餐B] [套餐C]                                             [增加套餐结算] |
| 说明：点击套餐名称后，下方月份结果表和月度结算详情切换到对应套餐                                      |
+--------------------------------------------------------------------------------------------------+
```

#### 4.1.3 交互规则

1. 页面进入后，根据当前 `customer_id` 查询已计算套餐列表。
2. 已计算套餐列表按最近更新时间倒序展示。
3. 默认选中第一条套餐。
4. 点击某个套餐标签后：
   - 第二层月份结算结果表切换到该套餐
   - 第三层月度详情默认切换到该套餐当前默认月份
5. 点击 `增加套餐结算` 后，弹出套餐选择框。

#### 4.1.4 组件建议

- Material-UI `Paper`
- Material-UI `Chip` 或 `ButtonGroup`
- Material-UI `Button`

当前选中套餐建议使用高亮样式：

- 选中：`contained` / 主色背景
- 未选中：`outlined`

### 4.2 增加套餐结算弹框

#### 4.2.1 数据来源

弹框套餐列表来源：

- `retail_packages`
- 过滤条件：`status=active`

#### 4.2.2 弹框内容

建议内容如下：

- 当前客户：只读
- 当前月份：只读或下拉（当前方案建议按当前月份执行，优先只读）
- 套餐选择框：Autocomplete
- 套餐附加信息展示：
  - 套餐名称
  - 套餐类型
  - 定价模型
  - 是否绿电
- 操作按钮：
  - `取消`
  - `开始计算`

#### 4.2.3 布局示意

```text
+--------------------------------------------------------------+
| 增加套餐结算                                                  |
+--------------------------------------------------------------+
| 当前客户： XXX 工厂                                           |
| 当前月份： 2026-02                                            |
|                                                              |
| 套餐选择： [请选择活跃套餐 v]                                 |
|                                                              |
| 套餐信息：                                                    |
| - 套餐类型：分时/不分时                                       |
| - 定价模型：price_spread_time                                |
| - 绿电：否                                                    |
+--------------------------------------------------------------+
| [取消]                                          [开始计算]    |
+--------------------------------------------------------------+
```

#### 4.2.4 交互规则

1. 打开弹框时加载活跃套餐列表。
2. 用户选择套餐后点击 `开始计算`。
3. 后端执行零售模拟计算并写入结果集。
4. 若当前客户当前月份当前套餐已存在结果，则执行覆盖重算。
5. 计算成功后：
   - 刷新第一层已计算套餐列表
   - 自动切换到新计算套餐
   - 第二层、第三层同步刷新

#### 4.2.5 前端状态建议

```ts
{
  addDialogOpen: boolean;
  packageOptions: RetailPackageOption[];
  packageOptionsLoading: boolean;
  selectedPackage: RetailPackageOption | null;
  addSubmitting: boolean;
}
```

### 4.4 第三层面板：月度结算详情

#### 4.4.1 设计目标

第三层面板需要：

- 复刻 [`SingleCustomerMonthlyDetailPage.tsx`](/d:/Gitworks/exds-web/frontend/src/pages/SingleCustomerMonthlyDetailPage.tsx) 页面中 KPI 卡片区以下的全部内容
- 展示当前选中 `套餐 + 月份` 的零售模拟详情

#### 4.4.2 复刻范围

复刻内容包含以下全部区域：

1. 结算基础区
2. 阶段结算区
3. 图表区
4. 分时段结算明细表
5. 月度日度结算明细表

#### 4.4.3 当前页面中应保留的展示逻辑

第三层中需要完整展示并保留以下业务概念：

- 套餐信息
- 定价模型
- 参考价/联动配置
- 封顶信息
- 调平前结果
- 调平后结果
- 超额返还影响
- 最终结算结果
- 分时段零售收入、批发成本、毛利、价差
- 分日零售收入、批发成本、毛利、价差

#### 4.4.4 为什么不能只展示最终值

若只展示最终零售金额、毛利和价差，则用户无法追溯：

- 价格模板如何生成
- 是否触发封顶
- 调平如何影响零售收入
- 超额返还如何影响最终毛利

因此第三层必须展示完整计算链路。

## 6. 数据集设计

### 6.1 批发侧结果集

集合名：

- `intent_customer_monthly_wholesale`

#### 唯一键

- `customer_id + settlement_month`

#### 规则

1. 不保留 `settlement_version`
2. 同一客户同一月份只有一条批发模拟结果
3. 重新计算时覆盖原记录

- `intent_customer_monthly_retail_simulation`

#### 唯一键

## 7. 后端接口设计

```text
GET /api/v1/intent-customer-diagnosis/customers/{customer_id}/retail-simulations/packages
```

#### 返回内容

- 当前客户所有已计算套餐列表
- 每个套餐的最近更新时间

示例：

```json
{
  "customer_id": "xxx",
  "items": [
    {
      "package_id": "pkg1",
      "package_name": "套餐A",
      "updated_at": "2026-03-17T21:00:00"
    }
  ]

```

---

### 7.2 查询当前套餐的月份结果列表

```text
GET /api/v1/intent-customer-diagnosis/customers/{customer_id}/retail-simulations
```

#### 入参

- `package_id`

#### 返回内容

- 当前套餐的月份结果列表

---

### 7.3 增加套餐结算

```text
POST /api/v1/intent-customer-diagnosis/customers/{customer_id}/retail-simulations
```

#### 入参

```json
{
  "settlement_month": "2026-02",
  "package_id": "pkg001"
}
```

#### 处理规则

1. 校验该客户该月份批发模拟结果已存在
2. 校验 `package_id` 对应套餐存在且 `status=active`
3. 执行零售模拟计算
4. 按唯一键 `customer_id + settlement_month + package_id` upsert

---

### 7.4 查询单套餐单月份详情

```text
GET /api/v1/intent-customer-diagnosis/customers/{customer_id}/retail-simulations/{package_id}/detail
```

#### 入参

- `settlement_month`

#### 返回内容

直接返回第三层详情区所需完整数据。

## 9. 计算流程

### 9.1 增加套餐结算流程

```text
用户点击“增加套餐结算”
    ↓
前端加载 retail_packages 中 status=active 的套餐
    ↓
用户选择套餐并确认
    ↓
后端校验批发侧结果已存在
    ↓
后端读取套餐配置并执行零售模拟
    ↓
写入 intent_customer_monthly_retail_simulation
    ↓
前端刷新已计算套餐列表
    ↓
自动切换到新套餐
```

### 9.2 套餐切换流程

```text
点击第一层套餐标签
    ↓
更新 activePackageId
    ↓
加载第二层月份结算结果表
    ↓
默认选中当前套餐的默认月份
    ↓
加载第三层详情
```

### 9.3 月份切换流程

```text
点击第二层月份表中的某一行
    ↓
更新 activeSettlementMonth
    ↓
加载第三层详情数据
    ↓
刷新图表与明细表
```

## 10. 开发建议

### 10.1 前端复用优先

第三层详情必须优先通过抽取签约客户详情页已有代码的方式实现，而不是重新写一份类似页面。原因：

1. 可保持视觉与交互一致
2. 可减少重复维护
3. 可降低实现风险

### 10.2 后端返回结构优先兼容现有详情页

建议零售模拟详情接口尽量兼容 [`SingleCustomerMonthlyDetailPage.tsx`](/d:/Gitworks/exds-web/frontend/src/pages/SingleCustomerMonthlyDetailPage.tsx) 当前消费字段，这样第三层详情区改造量最小。

### 10.3 明细字段命名建议

若后端已有字段命名与签约客户详情页一致，则优先保持兼容；若需要增强表达，可在兼容字段基础上增加更语义化字段，如：

- `retail_revenue`
- `wholesale_cost`
- `gross_profit`
- `spread_yuan_per_mwh`

## 11. 结论

`零售结算模拟` 子页面最终采用如下结构：

1. 第一层：已计算套餐列表 + 增加套餐结算按钮
2. 第二层：当前套餐的月份结算结果表格
3. 第三层：当前套餐当前月份的月度结算详情

其中第三层应完整复刻签约客户月度结算详情页 KPI 卡片区以下的全部内容，并以零售模拟结果集提供完整可追溯数据支撑。

本方案落地后，意向客户场景可在不依赖正式签约合同链路的前提下，独立完成：

- 多套餐零售模拟
- 月度盈利分析
- 批零价差分析
- 调平与返还影响追溯
