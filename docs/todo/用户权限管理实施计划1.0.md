## 三、方案评审意见

### 3.1 ✅ 方案优点

1. **架构分层清晰**：User → Role → Permission 四层模型，编码规范 `{domain}:{resource}:{action}` 合理
2. **后端先行策略正确**：阶段1先做后端授权闭环，即使前端绕过也会被 403 拦截
3. **"外宽内紧"粒度策略务实**：读权限按页面打包、写权限独立打点，平衡了开发效率和安全性
4. **兼容与回滚策略**：保留旧 token 兼容、权限开关化，降低迁移风险
5. **安全增强（第十一章）覆盖面全**：单用户登录、邮件MFA、首登改密、审计日志

### 3.2 ⚠️ 问题与建议

#### 问题 1：`auth_users` 新集合 vs 复用现有 `users`

> [!WARNING]
> 方案建议新建 `auth_users` 保留 `users` 兼容，但现有 `users` 已有 `roles`/`email`/`is_superuser` 字段，且仅 3 条记录。新建集合会导致双写和数据同步复杂度，**性价比不高**。

**建议**：直接扩展现有 `users` 集合，添加缺失字段（`display_name`、`password_changed_at`、`must_change_password`等），避免维护两套用户表。改动量更小、风险更低。

#### 问题 2：6 个 `auth_*` 集合可能过度设计

方案设计了 6 个集合：`auth_users`、`auth_roles`、`auth_permissions`、`auth_user_role_bindings`、`auth_role_permission_bindings`、`auth_audit_logs`。

**建议精简为 4 个集合**（MongoDB 文档模型更适合内嵌）：

| 集合 | 说明 |
|---|---|
| `users`（现有，扩展） | 用户信息 + `roles: [role_code]` 字段（已有） |
| `auth_roles` | 角色定义 + `permissions: [permission_code]`（内嵌权限列表） |
| `auth_permissions` | 权限点定义（主要用于管理页面和初始化） |
| `auth_audit_logs` | 审计日志 |

省掉 `auth_user_role_bindings` 和 `auth_role_permission_bindings`，因为这是 MongoDB 而非关系型数据库，用内嵌数组更符合文档模型，查询也更简单。

#### 问题 3：阶段 1.5 "安全增强"范围偏大

方案第十一章的安全增强需求（单用户登录、邮件MFA、首登改密、设备指纹等）涉及 `auth_sessions`、`auth_trusted_devices`、`auth_login_challenges` 三个新集合和大量新逻辑。

**建议**：
- 阶段 1 聚焦 RBAC 核心（角色/权限/授权校验）+ SECRET_KEY 环境变量化
- 阶段 1.5 仅做：会话管理（`auth_sessions` + token 携带 `sid`）和登录审计
- 阶段 2 再做：邮件MFA、首登改密、设备信任等

这样每个阶段的范围更可控。

#### 问题 4：JWT token 中不应携带太多信息

方案提到 token 中携带 `sid`。建议 token payload 保持精简：仅 `sub`（username）+ `sid`（会话ID）+ `exp`。角色和权限**每次从服务端查库**（方案第四章 4.2 节的 `get_current_user_context` 已有此设计），不放入 token。

#### 问题 5：前端需要处理 403 响应

> [!IMPORTANT]
> 当前 [client.ts](file:///d:/Gitworks/exds-web/frontend/src/api/client.ts) 响应拦截器只处理 401，**没有处理 403**。阶段 1 实施时需要补充 403 响应的处理逻辑（弹出"无权限"提示，而不是静默失败）。

#### 问题 6：缺少无操作限时登出

当前 [AuthContext.tsx](file:///d:/Gitworks/exds-web/frontend/src/contexts/AuthContext.tsx) 仅基于 JWT `exp` 做固定过期（60分钟），**不区分用户是否在操作**。无论用户是否活跃，60 分钟一到就强制登出，体验不佳且不安全（用户离开后 token 仍可在 60 分钟内被滥用）。

**建议采用前后端结合方案**：
- **前端**：监听鼠标/键盘/触摸事件，空闲超过 N 分钟后前端主动登出（即时响应，用户体验好）
- **后端**：每次 API 请求刷新 `last_active_at`，鉴权时校验空闲时长，超时返回 401（安全兜底，防止前端被绕过）
- 空闲超时时长建议可配置，默认 30 分钟

## 六、风险评估

| 风险 | 可能性 | 影响 | 缓解措施 |
|---|---|---|---|
| 权限依赖导致现有功能 403 阻断 | 中 | 高 | 环境变量开关 `AUTH_ENABLED=false` 可临时禁用授权检查 |

| 前端权限缓存与后端不一致 | 中 | 中 | 权限变更后强制重新 fetch `/auth/me` |
| 改动范围过大导致交付周期拉长 | 中 | 中 | 严格按阶段划分，阶段 1 不含管理页面 |
