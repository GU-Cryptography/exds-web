# exds-web 用户权限管理完整方案（基于现状的可落地版本）

## 一、方案摘要
当前项目已经有“登录鉴权”，但没有“授权控制”：
- 后端：`get_current_active_user` 只校验用户存在和启用状态，未做角色/权限点判断。
- 前端：`ProtectedRoute` 仅判断 `localStorage` 是否有 token，菜单与按钮无权限过滤。
- 系统菜单里“用户与权限”仍是占位页，暂无角色、权限、用户管理闭环。

你已确认采用：
- 权限粒度：`RBAC + 动作级`
- 上线节奏：`两阶段`

目标是把系统从“只认证”升级为“认证 + 授权 + 审计可追踪”，且最小风险平滑迁移。

---

## 二、现状问题与约束（从仓库提炼）
1. 认证与授权耦合不足  
- `webapp/tools/security.py` 的 `User` 模型只有 `username/is_active`，无法承载角色/权限信息。  
- `webapp/main.py` 对 `/api/v1` 做全局登录依赖，但没有细粒度授权。

2. 前端权限体系缺失  
- `frontend/src/components/ProtectedRoute.tsx` 只做“有 token 就放行”。  
- 路由与侧边栏为静态配置，未基于权限动态裁剪。

3. 安全与规范缺口（建议同步修复）  
- `SECRET_KEY` 硬编码在 `security.py`。  
- 多处仍用 `datetime.utcnow()`，与项目“本地时间统一”规范冲突。  
- 缺少权限变更审计日志和拒绝访问审计。

---

## 三、目标架构（最终态）
采用“RBAC + 动作级权限”四层模型：

1. 用户（User）
- 账号身份、状态、角色绑定

2. 角色（Role）
- 角色为权限集合（如 `super_admin`、`ops_admin`、`analyst`、`viewer`）

3. 权限点（Permission）
- 统一编码：`{domain}:{resource}:{action}`
- 例：`customer:profile:read`、`customer:profile:update`、`system:logs:resolve`

4. 鉴权依赖（后端）
- 每个敏感 API 显式声明 `require_permission("xxx")`
- 未登录返回 401；已登录无权限返回 403

## 4.3 API 变更（新增）
统一前缀：`/api/v1/auth`

- 用户与角色权限查询

1. `POST /roles`  
2. `PUT /roles/{role_id}`  
3. `PUT /roles/{role_id}/permissions`（全量覆盖）

- 用户角色管理  
1. `POST /users`（创建后台账号）  
2. `PUT /users/{user_id}/roles`（全量覆盖）  
3. `PUT /users/{user_id}/status`（启用/禁用）  
4. `PUT /users/{user_id}/password/reset`

- 审计  
1. `GET /audit-logs`（按时间/操作者/对象筛选）

## 五、权限编码与初始矩阵（首版）
## 5.1 权限编码规范
`{domain}:{resource}:{action}`，动作统一枚举：`read/create/update/delete/import/export/approve/manage`

## 5.2 角色建议（首版）
- `super_admin`：全部权限
- `system_admin`：系统管理、日志、监控、用户角色管理
- `business_admin`：客户/合同/套餐全权限 + 导入导出
- `analyst`：分析与预测读权限 + 部分提交权限
- `viewer`：只读权限

## 5.3 页面最小映射示例
- 客户档案：`customer:profile:read/update/delete`
- 零售合同：`customer:contract:read/create/update/delete/export`
- 系统日志：`system:logs:read/resolve`
- 数据下载监控：`system:data_access:read/manage`
- 用户与权限：`system:auth:manage`

## 5.4 读写权限管控粒度策略（补充）
针对系统中各类复杂业务模块（客户、合同、套餐、结算、手工预测等），采用**“外宽内紧”的折中精细化授权策略**，兼顾开发效率与安全性：

1. **纯读操作（GET/LIST）按页面打包授权（外宽）**
对于大范围的数据展示页面，只要通过菜单和路由下发了页面访问权限（如 `customer:profile:read`），即可放开该页面内的所有的普通查询级数据读取接口。无需对每一个查询按钮/图表配置细分权限（即便有部分数据非其职权内，基于前期不做数据域隔离的假定也可接受），大幅降低后端依赖注册和前端组件层层包裹的工作量。

2. **写入与核心能力操作（POST/PUT/DELETE等）必须独立打点控制（内紧）**
此类接口直接影响业务一致性和系统数据安全：
- 数据修改与删除（如：客户/合同增删改、字典管理）
- 资金/核心逻辑（如：重新核算结算账单、改套餐价、批量改价）
- 操作型/敏感工作（如：人工调减预测结果、RPA任务启停、明细数据全量导出）
- 审计日志等核心安全日志的清洗

以上操作**决不允许只凭“页面准入授权”即可调用**。相关按钮/接口必须单独注册出明确的动作级权限（例如 `forecast:adjust:update`、`settlement:retail:recalc`），未单独授权给改角色的后端访问将直接返回 403 HTTP拦截，彻底切断由 UI 入口绕过的风险。

## 六、两阶段实施计划（与你选择一致）
## 阶段 1（先安全闭环，后 UI）
目标：后端先具备真实授权能力，前端最小接入，不可绕过

1. 后端  
1. 落地 `auth_*` 集合、索引、初始化脚本（内置默认角色与权限）  
2. 实现 `get_current_user_context + require_permission`  
3. 对高风险写接口优先加权限依赖：客户、合同、套餐、系统管理、导入导出  
4. 增加 `/api/v1/auth/me` 与审计写入  
5. 兼容旧 token 读取（仅 `sub`），服务端每次查库计算权限

2. 前端  
1. 登录后拉取 `/auth/me`  
2. `ProtectedRoute` 升级为“登录 + 基础权限上下文就绪”  
3. 先做菜单级与页面级权限过滤  
4. 高风险按钮（删除、导入、审批）接入 `PermissionGuard`

交付标准：即使前端绕过隐藏，后端仍会 403 拒绝。

## 阶段 2（管理能力与体验完善）
1. 完成“用户与权限”页面（替换 Placeholder）  
2. 支持角色授权、用户分配角色、用户启停、密码重置  
3. 审计日志检索页面  
4. 增加权限变更回放与导出  
5. 完善权限单测/集成测试矩阵

## 十一、安全防护增强需求（新增）

以下 4 项作为权限体系的安全增强子方案，建议插入“阶段 1.5（安全增强）”实施。

### 11.1 单用户登录（同账号互斥）

#### 目标
- 同一账号仅允许一个有效会话在线。
- 新环境登录时，若检测到已有在线会话，提示用户确认；确认后强制踢出旧会话。

#### 数据与状态设计
- 新增集合：`auth_sessions`
- 关键字段：
  - `session_id`（唯一）
  - `username`
  - `device_fingerprint`（浏览器/设备指纹摘要）
  - `ip`
  - `user_agent`
  - `login_at`
  - `last_active_at`
  - `logout_at`
  - `status`（`active`/`kicked`/`expired`/`logout`）
  - `kicked_by_session_id`（可选）

#### 后端流程
1. 登录请求校验用户名密码成功后，查询 `auth_sessions` 中该用户 `status=active` 的会话。  
2. 若存在活跃会话，返回 `409` + `conflict_session_info`（含登录时间/IP/设备摘要）。  
3. 前端确认“强制登录”后，带 `force=true` 再次发起登录。  
4. 后端将旧会话状态改为 `kicked`，创建新会话并签发 token（token 中携带 `sid`）。  
5. 所有受保护接口在认证时校验 `sid` 对应会话仍为 `active`，否则返回 `401/440`。  

#### 前端交互
- 登录页增加“检测到其他环境在线”的确认弹窗。
- 被踢出的旧会话在下次请求收到失效响应后自动登出并提示“账号已在其他环境登录”。

---

### 11.2 新机器登录邮件验证码（二次确认，可按用户开关）

#### 目标
- 用户在新设备登录时，需邮件验证码二次确认。
- 支持按用户启用/关闭（字段级控制）。

#### 用户字段扩展
- `email`：有效邮箱（必填后方可启用邮件验证）
- `email_verified`：邮箱是否已验证
- `email_mfa_enabled`：是否启用新设备邮件验证（默认 `false`）

#### 设备识别策略
- 使用 `device_fingerprint`（浏览器指纹 + UA + 平台摘要）做“已信任设备”判定。
- 新增集合：`auth_trusted_devices`
  - `username`
  - `device_fingerprint`
  - `first_seen_at`
  - `last_seen_at`
  - `is_active`

#### 后端流程
1. 登录第一步（密码通过）后判断设备是否已信任。  
2. 若 `email_mfa_enabled=true` 且为新设备，生成 6 位验证码，写入 `auth_login_challenges`（有效期 5~10 分钟，错误次数上限）。  
3. 通过邮件发送验证码，返回 `202` + `challenge_id`。  
4. 前端提交 `challenge_id + code` 到验证接口，验证通过后才创建会话并签发 token。  
5. 验证成功后可选择“信任本设备 N 天”（可配置），写入 `auth_trusted_devices`。  

#### 接口建议
- `POST /api/v1/auth/login`（可能返回 `202` challenge）
- `POST /api/v1/auth/login/verify-email-code`
- `POST /api/v1/auth/users/{user_id}/email-mfa-toggle`

---

### 11.3 自动记录登录时间、IP、使用时长

#### 目标
- 形成可审计登录日志与会话时长统计。

#### 数据记录
- 登录成功：写 `auth_sessions` + `auth_audit_logs`（事件 `LOGIN_SUCCESS`）
- 登录失败：写 `auth_audit_logs`（事件 `LOGIN_FAILED`，含失败原因）
- 登出/被踢/超时：更新 `logout_at` 与 `status`
- 使用时长计算：`duration_seconds = logout_at - login_at`；若在线则按 `now - login_at` 实时计算

#### 接口建议
- `GET /api/v1/auth/session-history`（按用户、时间范围、IP筛选）
- `GET /api/v1/auth/online-users`（管理员）

#### 补充
- 建议记录 `last_active_at`（通过请求心跳或受保护 API 访问刷新）用于在线状态准确性。

### 11.4 首次登录强制改密 + 登记有效邮箱

#### 目标
- 新建账号首次登录必须完成：修改初始密码 + 填写并验证邮箱。

#### 用户字段扩展
- `must_change_password`（默认新用户为 `true`）
- `is_first_login`（默认 `true`）
- `password_changed_at`
- `email`、`email_verified`

#### 登录与放行规则
1. 密码校验成功后，若 `must_change_password=true` 或 `email` 为空/未验证：  
   - 返回 `403` + `required_actions`（如 `CHANGE_PASSWORD`、`BIND_EMAIL`、`VERIFY_EMAIL`）。  
2. 前端跳转“首次安全设置”流程页。  
3. 完成改密与邮箱验证后，后端更新字段并允许正式签发业务会话 token。  

#### 接口建议
- `POST /api/v1/auth/first-login/change-password`
- `POST /api/v1/auth/first-login/bind-email`
- `POST /api/v1/auth/first-login/verify-email`

### 11.5 实施顺序与验收补充

#### 推荐顺序
1. 先做 `auth_sessions` 与 token `sid` 校验（实现单点在线）。  
2. 再做首次登录强制改密 + 邮箱绑定。  
3. 最后接入新设备邮件挑战（MFA）与信任设备。  

#### 验收用例（新增）
1. 同账号 A 端在线，B 端登录触发冲突提示，B 端确认后 A 端被踢下线。  
2. 开启邮件验证的用户在新设备登录，未输入正确验证码不能拿到 token。  
3. 登录日志可查询到时间/IP/设备/会话时长，状态准确区分主动登出与被踢。  
4. 新建用户首次登录未改密或未验证邮箱，无法进入业务页面。  
5. 关闭 `email_mfa_enabled` 的用户在已知设备登录不触发验证码。

### 11.6 范围确认与必要性说明（已确认）

#### 已确认范围调整
1. 不纳入本期：`密码过期策略（如 90/180 天）+ 到期提醒`。  
2. 暂不纳入本期：此前“应做项目”中的增强项。  
3. 本期聚焦：会话可控（可踢下线）+ 新设备邮件确认 + 登录审计 + 首登安全设置。  

#### 第一条“令牌与会话安全”的含义
第一条包含三个不可拆分的子项：
1. `SECRET_KEY` 环境变量化：JWT 签名密钥不写入代码仓库。  
2. token 携带 `sid`：每次登录分配唯一会话 ID。  
3. 服务端会话校验：每次鉴权除校验签名外，还要校验 `sid` 对应会话状态为 `active`。  

#### 第一条的必要性
1. 若 `SECRET_KEY` 硬编码泄露，攻击者可伪造 token，风险为系统级。  
2. 若 token 不带 `sid`，系统无法区分“哪一次登录”，无法精准强制下线。  
3. 若不做服务端会话状态校验，被踢掉的旧 token 在过期前仍可继续访问，无法满足“单用户登录”和“确认后强制退出其他环境”的需求。
